# 进阶与扩展专题

进阶不等于引入更新、更复杂的框架，而是能在新的主导约束下重新使用基础模型。存储关注掉电与寿命，GUI 关注渲染吞吐和交互状态，TinyML 关注数据分布和资源，安全可靠性关注威胁、失效与系统不变量。

这些专题按项目需求选择。若启动、内存、并发、测试和发布基础尚未形成，过早追逐专题通常只会扩大无法解释的系统表面积。

## 模块契约

### 进入条件

- 已完成至少一条系统主线和一个阶段项目。
- 能做基本资源预算和故障注入。
- 选择专题有明确产品目标，而不是只为增加技术栈。

### 专题入口

| 专题 | 进入前置 | 主导约束 | 最低项目证据 |
|---|---|---|---|
| [存储与文件系统](middleware-storage-filesystem.md) | Flash、序列化、测试 | 掉电、磨损、容量、schema | 断电/损坏注入、寿命预算、恢复 |
| [LVGL 与 Qt](gui-lvgl-qt.md) | 并发、DMA/Linux 图形基础 | 显存、带宽、帧时延、交互 | 帧率/时延、资源、并发和失败恢复 |
| [Edge AI 与 TinyML](edge-ai-tinyml.md) | 传感器链、数据处理、资源预算 | 数据分布、模型资源、误报漏报 | 现场数据、延迟/RAM/功耗和回归 |
| [安全、可靠性与功能安全](safety-reliability.md) | 系统架构、测试、生命周期 | 威胁、失效、不变量、残余风险 | 威胁/故障分析、独立保护和证据 |

## 选择专题的决策表

开始前回答：

- 产品目标是什么，专题解决的瓶颈是否已被证据确认？
- 当前系统最主要的资源和风险约束是什么？
- 新组件会引入哪些任务、内存、存储、依赖、攻击面和更新责任？
- 哪些现有不变量必须保持？
- 如何在最小垂直路径中验证价值，再决定是否全面集成？
- 如果效果不达标，如何移除或降级而不破坏主系统？

如果无法回答，先回到项目架构和问题证据，不要直接选框架。

## 专题学习统一闭环

```text
产品问题
  -> 主导约束和基线
  -> 候选方案与资源预算
  -> 最小垂直实验
  -> 失败与极限条件
  -> 集成到主项目
  -> 发布、更新和现场诊断
  -> 与基线比较并决定保留/调整/删除
```

### 存储专题

不能只证明读写成功。至少覆盖：断电原子性、坏块/磨损假设、空间耗尽、schema 迁移、回滚兼容和数据恢复。主机 [配置日志实验](../00-learning-path/host-config-journal.md) 是起点，不是目标 Flash 的最终证明。

### GUI 专题

不能只展示界面截图。至少测量最坏帧时间、输入到显示延迟、buffer/显存、总线吞吐和高负载下的降级。UI 线程、驱动回调和业务状态所有权必须清楚。

### TinyML 专题

不能只报告离线准确率。至少记录输入契约、现场数据分布、误报漏报代价、RAM/Flash、推理延迟、功耗、模型/预处理版本和更新回滚。模型输出触发物理动作时应有独立安全边界。

### 安全与可靠性专题

不能只列加密算法和看门狗。需要从资产、攻击者、信任边界、失效模式和安全目标出发，明确检测、隔离、恢复、独立保护和残余风险。功能安全标准的流程和证据要求不能由一篇入门文章替代。

## 专题完成证据

无论选择哪个专题，都提交：

- 引入前的基线指标与问题证据。
- 至少两个候选方案的预算和决策记录。
- 最小实验及其证据边界。
- 资源极限或故障注入结果。
- 与主系统的数据、控制、错误和更新接口。
- 引入后的量化结果、回归测试和现场诊断指标。
- 已知限制、回退方案和重新评审条件。

## 退出标准

- [ ] 专题选择由产品约束驱动，并有引入前基线。
- [ ] 新组件资源、时间、数据和生命周期边界明确。
- [ ] 关键失败路径可注入、可观察、可恢复或被接受。
- [ ] 专题没有破坏原系统的实时、安全、更新和诊断不变量。
- [ ] 指标改善能被同条件对比，而不是只报告新方案结果。
- [ ] 换平台、模型、显示、存储或安全假设后知道哪些测试必须重跑。

进阶能力的标志不是使用更多技术，而是知道什么时候值得引入、如何证明收益，以及如何控制它带来的新风险。
