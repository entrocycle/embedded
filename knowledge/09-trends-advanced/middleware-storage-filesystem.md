# 中间件、存储与文件系统

存储是很多嵌入式产品的薄弱点。配置、日志、校准参数、离线缓存、OTA 镜像和用户数据都要落盘；一旦遇到断电、写放大、磨损、文件系统损坏，问题会直接变成现场故障。

## 存储介质

| 介质 | 特点 | 常见用途 |
|---|---|---|
| MCU 内部 Flash | 直接可用，擦写次数有限 | 少量配置、Bootloader |
| 外部 SPI NOR | 读快、容量中等、XIP 可选 | 文件系统、字库、OTA |
| EEPROM | 小容量、字节擦写 | SN、校准参数 |
| FRAM | 高擦写寿命、低功耗 | 高频小数据记录 |
| SD 卡/eMMC | 容量大、块设备 | Linux RootFS、日志、媒体 |
| NAND | 容量大、坏块管理复杂 | 大容量嵌入式系统 |

## Flash 基础

Flash 写入有三个约束：

- 擦除单位通常大于写入单位。
- 擦除后位为 1，写入只能把 1 改为 0。
- 擦写次数有限。

因此不能把 Flash 当作普通 RAM 使用。配置更新要有磨损均衡和掉电保护。

## 配置存储方案

一个可靠配置块包含：

```text
magic
version
sequence
length
payload
crc
```

双副本策略：

1. 读取时选择 CRC 正确且 sequence 最新的副本。
2. 写入时先写旧副本之外的另一份。
3. 写完校验成功后再切换。
4. 任意阶段断电，至少有一份旧配置可用。

## 文件系统选择

| 文件系统 | 适用场景 | 注意事项 |
|---|---|---|
| LittleFS | MCU 外部 NOR、掉电保护配置/日志 | 适合小型嵌入式 |
| FATFS | SD 卡、U 盘、与 PC 兼容 | 掉电一致性弱 |
| SPIFFS | 旧项目常见 | 新项目优先考虑 LittleFS |
| ext4 | Linux 块设备 | 需要正确挂载和断电策略 |
| squashfs | Linux 只读 RootFS | 适合 A/B 或只读系统 |
| ubifs | raw NAND | 处理坏块和磨损 |

## 日志存储

日志设计要限制：

- 单文件大小。
- 文件数量。
- 总空间。
- 写入频率。
- 错误刷屏。

MCU 上推荐环形日志：

```text
fixed-size record
  timestamp
  level
  module
  code
  arg0
  arg1
```

Linux 上可以使用 journald、logrotate 或自定义 ring file。

## OTA 镜像存储

OTA 存储要考虑：

- 下载分区大小。
- 分片校验。
- 整包 hash。
- 签名。
- 断点续传。
- 写入失败。
- 新固件健康确认。
- 回滚标志。

不要下载完才第一次校验所有数据。分片校验可以更早发现传输或 Flash 问题。

## 掉电保护

关键写入要遵循：

1. 写新数据到空闲区域。
2. 校验新数据。
3. 原子更新有效标记。
4. 延迟删除旧数据。

危险做法：

- 原地覆盖配置。
- 先擦旧数据再写新数据。
- 写日志没有空间上限。
- 文件写完不 flush/sync 就立即断电。

## 存储健康监控

建议统计：

- 擦写次数。
- 写失败次数。
- CRC 失败次数。
- 剩余空间。
- 坏块数量。
- 最近一次配置恢复原因。

这些数据可以帮助判断现场问题是软件逻辑、掉电、介质质量还是寿命问题。

## 练习

1. 设计一个双副本配置区格式，支持版本和 CRC。
2. 在 MCU 项目中加入固定大小环形日志。
3. 比较 LittleFS 和 FATFS 在断电场景下的风险差异。
