# 中间件、存储与文件系统

存储是很多嵌入式产品的薄弱点。配置、日志、校准参数、离线缓存、OTA 镜像和用户数据都要落盘；一旦遇到断电、写放大、磨损、文件系统损坏，问题会直接变成现场故障。

## 本章任务与边界

完成本章后，你应能按数据语义选择裸 Flash、KV、日志或文件系统，计算容量、写放大与寿命，设计提交、恢复、空间耗尽、坏块/损坏和 schema 迁移，并建立存储健康指标。

主机文件和内存 Flash 模型不能证明真实介质的原子写、擦除粒度、掉电行为、FTL 和磨损。文件系统“支持掉电恢复”也不等于应用事务已同步提交。

完成证据：运行 [双槽配置日志实验](../00-learning-path/host-config-journal.md)，再基于目标器件手册建立原子模型；提交容量/寿命预算、逐阶段复位、写满和版本回滚结果。

## 存储介质

| 介质 | 特点 | 常见用途 |
|---|---|---|
| MCU 内部 Flash | 直接可用，擦写次数有限 | 少量配置、Bootloader |
| 外部 SPI NOR | 读快、容量中等、XIP 可选 | 文件系统、字库、OTA |
| EEPROM | 小容量、字节擦写 | SN、校准参数 |
| FRAM | 高擦写寿命、低功耗 | 高频小数据记录 |
| SD 卡/eMMC | 容量大、块设备 | Linux RootFS、日志、媒体 |
| NAND | 容量大、坏块管理复杂 | 大容量嵌入式系统 |

## Flash 基础

Flash 写入有三个约束：

- 擦除单位通常大于写入单位。
- 擦除后位为 1，写入只能把 1 改为 0。
- 擦写次数有限。

因此不能把 Flash 当作普通 RAM 使用。配置更新要有磨损均衡和掉电保护。

## 配置存储方案

一个可靠配置块包含：

```text
magic
version
sequence
length
payload
crc
```

双副本策略：

1. 读取时选择 CRC 正确且 sequence 最新的副本。
2. 擦除并写入旧副本之外的另一份，但暂不标记提交。
3. 回读校验 header、payload 和 CRC。
4. 最后用一个独立 program unit 写入 `COMMITTED` 标记。
5. 新记录提交后，旧记录才允许在后续维护中擦除。
6. 任意阶段断电，至少有一份已提交配置可用。

最后一步的“切换”不是抽象的 C 变量赋值，而必须映射到介质保证的原子写入粒度。示例记录状态可利用 Flash 只能 `1 -> 0` 的性质：

```text
ERASED (0xFFFF_FFFF)
  -> WRITING (program header/payload/crc)
  -> COMMITTED (单独 program 一个对齐 commit word)
```

启动时只接受完整、长度合法、CRC 正确且 commit word 有效的记录。不能先写 `COMMITTED` 再写 payload，也不能假设结构体一次写调用等于物理原子事务。

sequence 比较还要处理回绕。对 32 位 generation，可在保证任意两个有效记录差距小于 `2^31` 的前提下用模运算判断新旧：

```c
static bool generation_after(uint32_t a, uint32_t b)
{
    uint32_t distance = a - b;
    return distance != 0u && distance < 0x80000000u;
}
```

无符号减法按模 `2^32` 定义，但“半个序号空间以内才可比较”仍是系统假设；必须保证两个有效记录差距小于 `2^31`，并为 `0xFFFFFFFF -> 0`、相等和恰好相差 `2^31` 写测试。若两个记录 generation 相同但内容不同，或落在无法排序的半圈位置，必须定义保守选择或进入诊断状态，不能随意选第一个。

每个 `erase`、每个 program unit、回读验证和 commit 前后都应能注入复位。仓库的主机配置日志实验用 byte array 模拟这种事务；实板还要验证真实 Flash 的 program 粒度、对齐、部分写行为、brown-out 和擦写寿命。

## 文件系统选择

| 文件系统 | 适用场景 | 注意事项 |
|---|---|---|
| LittleFS | MCU 外部 NOR、掉电保护配置/日志 | 适合小型嵌入式 |
| FATFS | SD 卡、U 盘、与 PC 兼容 | 掉电一致性弱 |
| SPIFFS | 旧项目常见 | 新项目优先考虑 LittleFS |
| ext4 | Linux 块设备 | 需要正确挂载和断电策略 |
| squashfs | Linux 只读 RootFS | 适合 A/B 或只读系统 |
| UBI + UBIFS | raw NAND | UBI 管理坏块、磨损均衡和 volume，UBIFS 在 UBI volume 上提供文件系统 |

## 日志存储

日志设计要限制：

- 单文件大小。
- 文件数量。
- 总空间。
- 写入频率。
- 错误刷屏。

MCU 上推荐环形日志：

```text
fixed-size record
  timestamp
  level
  module
  code
  arg0
  arg1
```

Linux 上可以使用 journald、logrotate 或自定义 ring file。

## OTA 镜像存储

OTA 存储要考虑：

- 下载分区大小。
- 分片校验。
- 整包 hash。
- 签名。
- 断点续传。
- 写入失败。
- 新固件健康确认。
- 回滚标志。

不要下载完才第一次校验所有数据。分片校验可以更早发现传输或 Flash 问题。

## 掉电保护

关键写入要遵循：

1. 写新数据到空闲区域。
2. 校验新数据。
3. 原子更新有效标记。
4. 延迟删除旧数据。

危险做法：

- 原地覆盖配置。
- 先擦旧数据再写新数据。
- 写日志没有空间上限。
- 文件写完不 flush/sync 就立即断电。

Linux 上的原子配置替换通常是：在同一文件系统创建临时文件，写入并 `fsync(temp_fd)`，关闭后 `rename(temp, target)`，再 `fsync(parent_dir_fd)` 使目录项持久化。即使如此，块设备写缓存、控制器、电源保持能力和具体文件系统挂载选项仍会影响断电语义，必须用目标存储验证。`flush`/`sync` 返回成功也不自动保证廉价介质诚实地完成持久化。

## 存储健康监控

建议统计：

- 擦写次数。
- 写失败次数。
- CRC 失败次数。
- 剩余空间。
- 坏块数量。
- 最近一次配置恢复原因。

这些数据可以帮助判断现场问题是软件逻辑、掉电、介质质量还是寿命问题。

## 练习

1. 设计一个双副本配置区格式，支持版本和 CRC。
2. 在 MCU 项目中加入固定大小环形日志。
3. 比较 LittleFS 和 FATFS 在断电场景下的风险差异。
4. 枚举双副本配置每个 program unit 后断电的结果，证明启动选择规则不会加载半写记录。

## 深入思考

1. CRC、commit word 和数字签名分别证明什么？为什么不能互相替代？
2. 如果新 schema 写入后旧固件回滚，它能否安全读取新配置？兼容性应该由谁声明？
3. 双副本把一次配置更新放大成多少次擦写？给定更新频率和额定擦写次数，寿命预算是否成立？
4. 两个副本都无效时，恢复默认值、停机和远程恢复各有什么产品风险？

## 工程洞察

存储系统的核心不是“能保存数据”，而是在掉电、磨损、版本变化、空间耗尽和介质异常下维护数据语义。嵌入式存储经常承载配置、日志、校准、证书、OTA 镜像和用户数据；这些数据的可靠性要求并不相同，不能用同一个策略处理。

设计存储前先分类：

- 配置：小容量、低频更新、要求原子提交和 schema 兼容。
- 日志：高频追加、允许丢弃部分旧数据、要求限额和可导出。
- 校准/SN：低频写入、强追溯、通常不应被恢复出厂清除。
- 证书/密钥：要求访问控制、防提取、防回滚和生命周期管理。
- OTA 镜像：大容量、分片写入、完整性/真实性验证和回滚状态。

文件系统能解决目录、文件和磨损管理的一部分问题，但不能自动解决业务原子性、配置兼容、签名验证和安全删除。越关键的数据，越需要显式不变量和断电枚举。
