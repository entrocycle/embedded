# 安全、可靠性与功能安全入门

嵌入式设备通常长期运行在不可控环境中。产品化必须考虑安全、防篡改、故障恢复、可靠性测试和在危险场景下的安全状态。

## 安全目标

常见安全目标：

- 只有可信固件能运行。
- OTA 包不能被篡改。
- 设备密钥不能被轻易读取。
- 调试口不能在量产后无保护开放。
- 通信链路不能被伪造或重放。
- 故障时设备进入安全状态。

## 安全启动

安全启动链路：

```text
ROM trust root
  -> verify Bootloader
  -> verify App
  -> boot
```

关键点：

- 公钥或 hash 根要受硬件保护。
- Bootloader 验证应用签名。
- 防止版本回滚到有漏洞固件。
- 失败时进入恢复模式，而不是直接运行未知镜像。

## OTA 安全

OTA 至少要有：

- 版本号。
- 硬件兼容信息。
- 包 hash。
- 数字签名。
- 分片校验。
- A/B 或回滚机制。
- 升级结果上报。

不要只依赖 HTTPS。传输加密不能替代固件签名。

## 密钥管理

密钥不要硬编码在源码中。

可选方案：

- MCU 安全区/TrustZone。
- OTP/Option Bytes。
- 安全芯片。
- Linux TPM/TEE。
- 每设备唯一证书。

要考虑：

- 生产写入。
- 读取权限。
- 更新和吊销。
- 备份和恢复。
- 日志脱敏。

## 调试口策略

量产设备要明确：

- SWD/JTAG 是否关闭。
- 是否允许授权解锁。
- 解锁是否擦除密钥。
- 串口 shell 是否需要认证。
- 工厂模式如何进入和退出。

开发便利和安全要分阶段处理，不能把开发后门带到量产。

## 可靠性设计

可靠性不是最后测试阶段才补。设计阶段就要考虑：

- 看门狗。
- Brown-out 检测。
- 配置双副本。
- 断电保护。
- 任务心跳。
- 错误码和诊断。
- 资源上限。
- 故障降级。
- 安全默认态。

## 功能安全入门

功能安全关注“系统发生故障时是否会造成危险”。

基本概念：

- hazard：危险。
- risk：风险。
- fail-safe：故障后进入安全状态。
- diagnostic coverage：诊断覆盖率。
- redundancy：冗余。
- FMEA：失效模式与影响分析。

简单 FMEA 表：

| 模块 | 失效模式 | 影响 | 检测 | 缓解 |
|---|---|---|---|---|
| 温度传感器 | 读数固定 | 控制误判 | 合理范围和变化率检查 | 使用默认策略并告警 |
| 电机驱动 | 输出失控 | 机械危险 | 电流/速度反馈 | 关闭驱动 |
| 网络 | 长时间断连 | 无法远程控制 | 心跳超时 | 本地安全策略 |

## 可靠性测试

建议测试：

- 长稳运行。
- 断电重启。
- OTA 中断电。
- 低电压。
- 高低温。
- 振动。
- 弱网。
- Flash 写满。
- 传感器断连。
- 执行器堵转。

## 安全检查清单

- [ ] 固件签名验证。
- [ ] OTA 防回滚。
- [ ] 每设备唯一身份。
- [ ] 密钥不进日志。
- [ ] 量产调试口策略明确。
- [ ] 远程命令有认证和审计。
- [ ] 配置有 CRC 和默认恢复。
- [ ] 看门狗基于健康状态喂狗。
- [ ] 故障时输出进入安全态。

## 练习

1. 为一个智能插座做 FMEA，至少列 5 个失效模式。
2. 设计一个 OTA manifest，包含版本、hash、签名和硬件兼容字段。
3. 制定一份量产调试口关闭和授权解锁策略。
