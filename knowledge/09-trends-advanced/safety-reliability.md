# 安全、可靠性与功能安全入门

嵌入式设备通常长期运行在不可控环境中。产品化必须考虑安全、防篡改、故障恢复、可靠性测试和在危险场景下的安全状态。

## 本章任务与边界

完成本章后，你应能区分信息安全、可靠性与功能安全目标，识别资产/威胁和随机失效，使用不变量、FMEA 与独立保护设计检测、隔离、降级和恢复，并明确诊断覆盖与残余风险。

本章是工程入门，不等价于 IEC 61508、ISO 26262、IEC 62304 等行业标准培训或认证。风险可接受性、独立性、系统能力等级和证据需由适用法规、组织流程和合格评审决定。

完成证据：项目威胁模型、FMEA、三条安全不变量、一次保护通道故障注入，以及“当前系统不能保证什么”的残余风险记录。

## 安全目标

常见安全目标：

- 只有可信固件能运行。
- OTA 包不能被篡改。
- 设备密钥不能被轻易读取。
- 调试口不能在量产后无保护开放。
- 通信链路不能被伪造或重放。
- 故障时设备进入安全状态。

## 安全启动

安全启动链路：

```text
ROM trust root
  -> verify Bootloader
  -> verify App
  -> boot
```

关键点：

- 公钥或 hash 根要受硬件保护。
- Bootloader 验证应用签名。
- 防止版本回滚到有漏洞固件。
- 失败时进入恢复模式，而不是直接运行未知镜像。

## OTA 安全

OTA 至少要有：

- 版本号。
- 硬件兼容信息。
- 包 hash。
- 数字签名。
- 分片校验。
- A/B 或回滚机制。
- 升级结果上报。

不要只依赖 HTTPS。传输加密不能替代固件签名。

## 密钥管理

密钥不要硬编码在源码中。

可选方案：

- MCU 安全区/TrustZone。
- OTP/Option Bytes。
- 安全芯片。
- Linux TPM/TEE。
- 每设备唯一证书。

要考虑：

- 生产写入。
- 读取权限。
- 更新和吊销。
- 备份和恢复。
- 日志脱敏。

## 调试口策略

量产设备要明确：

- SWD/JTAG 是否关闭。
- 是否允许授权解锁。
- 解锁是否擦除密钥。
- 串口 shell 是否需要认证。
- 工厂模式如何进入和退出。

开发便利和安全要分阶段处理，不能把开发后门带到量产。

## 可靠性设计

可靠性不是最后测试阶段才补。设计阶段就要考虑：

- 看门狗。
- Brown-out 检测。
- 配置双副本。
- 断电保护。
- 任务心跳。
- 错误码和诊断。
- 资源上限。
- 故障降级。
- 安全默认态。

## 故障包含与权限隔离

可靠系统不仅检测故障，还要限制故障传播范围。可用机制取决于平台：

| 机制 | 可缓解 | 不能自动解决 |
|---|---|---|
| MCU MPU | 栈越界、非法区域访问、模块权限 | 逻辑错误、DMA 越权配置、多核共享 |
| TrustZone/安全世界 | 密钥、启动根、受控安全服务 | 安全世界自身漏洞、侧信道、错误接口 |
| Linux 进程/用户/capability | 服务崩溃和权限扩散 | 内核漏洞、错误授权、共享数据语义 |
| IOMMU/DMA 隔离 | 设备 DMA 地址范围 | 设备固件恶意、错误映射授权 |
| 独立硬件保护 | 在软件失效时切断危险输出 | 保护器件共因失效、错误阈值 |

MPU 配置要和链接布局、任务切换、DMA 区域及 fault handler 一起设计。只启用 MPU 但所有区域仍可读写，或 fault 后无诊断地复位，保护价值有限。

安全服务接口应最小化：普通世界提交长度受限的请求，安全世界验证权限和参数，不把密钥指针或任意内存读写能力暴露出去。隔离边界本身也是攻击面，需要 fuzz、权限和生命周期测试。

## 冗余与独立性

两个相同传感器、两个相同软件任务或重复计算不一定构成有效冗余。要分析共同原因：

- 同一电源、时钟、总线、连接器或物理环境。
- 同一驱动、算法、编译器和需求错误。
- 同一校准、配置和更新包。
- 同一通信链路和云端判定。

冗余设计要说明：故障假设、独立程度、仲裁规则、分歧时安全动作和诊断覆盖。异构冗余可能降低共同软件故障，却增加接口、校准和验证复杂度。

看门狗也需要独立性评审：内部 watchdog 可能与 CPU 时钟或电源共因失效，外部 watchdog 增加硬件与喂狗协议。无论哪种，都应由有意义进展驱动，并在复位前后保留足够上下文。

## 多核与异构系统边界

多核 MCU、Linux + 实时核、CPU + DSP/NPU 会引入新的所有权和恢复问题：

- 哪个核拥有外设、共享内存和电源状态。
- cache 一致性由硬件保证还是软件维护。
- 消息队列、mailbox 和共享 ring 的内存顺序。
- 一个核复位、升级或失联时另一个核如何进入安全状态。
- 两侧固件和协议版本如何兼容、如何原子升级。
- 时间基准如何同步，trace 如何关联。

“另一个核会处理”不是故障隔离。跨核接口需要和网络协议一样定义 framing、版本、deadline、错误、重启握手和容量上限。对安全关键动作，还要考虑一个失控核是否能绕过另一核的保护。

## 功能安全入门

功能安全关注“系统发生故障时是否会造成危险”。

基本概念：

- hazard：危险。
- risk：风险。
- fail-safe：故障后进入安全状态。
- diagnostic coverage：诊断覆盖率。
- redundancy：冗余。
- FMEA：失效模式与影响分析。

简单 FMEA 表：

| 模块 | 失效模式 | 影响 | 检测 | 缓解 |
|---|---|---|---|---|
| 温度传感器 | 读数固定 | 控制误判 | 合理范围和变化率检查 | 使用默认策略并告警 |
| 电机驱动 | 输出失控 | 机械危险 | 电流/速度反馈 | 关闭驱动 |
| 网络 | 长时间断连 | 无法远程控制 | 心跳超时 | 本地安全策略 |

## 可靠性测试

建议测试：

- 长稳运行。
- 断电重启。
- OTA 中断电。
- 低电压。
- 高低温。
- 振动。
- 弱网。
- Flash 写满。
- 传感器断连。
- 执行器堵转。

## 安全检查清单

- [ ] 固件签名验证。
- [ ] OTA 防回滚。
- [ ] 每设备唯一身份。
- [ ] 密钥不进日志。
- [ ] 量产调试口策略明确。
- [ ] 远程命令有认证和审计。
- [ ] 配置有 CRC 和默认恢复。
- [ ] 看门狗基于健康状态喂狗。
- [ ] 故障时输出进入安全态。
- [ ] MPU/进程/TrustZone 等隔离与链接、DMA、权限和 fault 处理一致。
- [ ] 冗余通道的共同原因和分歧策略已评审。
- [ ] 多核/异构接口有版本、所有权、重启和超时契约。

## 练习

1. 为一个智能插座做 FMEA，至少列 5 个失效模式。
2. 设计一个 OTA manifest，包含版本、hash、签名和硬件兼容字段。
3. 制定一份量产调试口关闭和授权解锁策略。

## 工程洞察

安全和可靠性不是测试阶段补丁，而是设计阶段对风险的系统管理。安全关注恶意或非授权行为，可靠性关注随机故障、磨损、环境和长期运行；功能安全则进一步关注故障发生后系统是否进入可接受风险状态。三者目标不同，但都要求清楚资产、失效模式、检测、隔离、恢复和残余风险。

做可靠性设计时，建议从 FMEA 思维开始：

```text
功能 -> 失效模式 -> 影响 -> 原因 -> 检测 -> 缓解 -> 残余风险 -> 验证
```

例如继电器控制不是只看 GPIO 输出，还要考虑继电器粘连、驱动管短路、电流传感器失效、固件卡死、云命令重放、用户误操作和电源异常。每个风险不一定都能用软件完全消除，但必须被识别、分级和验证。

安全可靠性能力的成熟标志，是能坦诚说明系统不能保证什么。工程不是制造“绝对安全”的幻觉，而是在约束下把风险降低到可接受，并让残余风险可见、可审计、可响应。
