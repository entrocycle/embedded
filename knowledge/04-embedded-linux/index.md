# Embedded Linux 与 BSP

Embedded Linux 不是在开发板上会用 shell，而是能控制一条完整系统链：BootROM、SPL/U-Boot、Kernel、DTB、RootFS、init、系统服务和应用。问题可能跨越镜像、启动参数、设备模型、权限、文件系统和版本边界，因此本模块训练的是分层定位与系统集成。

Linux/BSP 是从共同基础出发的一条独立主线，不要求先完成 RTOS。已有 MCU 经验有助于理解寄存器、中断和电气行为，但 Linux 的设备模型、并发和生命周期需要重新学习，不能把裸机驱动直接搬进内核。

## 模块契约

### 进入条件

- 通过共同基础阶段门 G1。
- 熟悉 Linux 文件、进程、权限、管道、Git 和基本 shell。
- 能区分主机编译器与交叉编译器，并检查 ELF 目标架构。
- 有 Linux 开发板或可启动的 QEMU 环境，能看到串口/控制台日志。

### 完成后应能

- 从启动日志判断系统位于 Bootloader、Kernel、RootFS、init 还是应用阶段。
- 追溯正在运行的 Kernel、DTB、RootFS、模块、应用和配置版本。
- 修改设备树并用运行时设备树和 driver binding 证明实际生效。
- 为设备选择标准内核子系统，并设计 probe/remove/suspend 生命周期。
- 区分内核机制与用户态策略，构建可管理、可更新的系统服务。
- 使用 Buildroot/Yocto 或等价流程产生可复现系统镜像。
- 对驱动、服务、更新和文件系统失败建立诊断和回滚路径。

### 本模块不解决

- 不覆盖 Linux 内核全部子系统和通用发行版运维。
- 不把自定义字符设备作为所有硬件的默认答案。
- 示例 API 需结合目标内核版本核对；内核接口和设备树 binding 会演进。

## 贯穿项目：可追溯传感器网关

### B0：交叉编译与目标运行

- 用交叉工具链构建一个程序。
- 用 `file`、`readelf` 和目标运行结果证明架构、ABI 和依赖正确。
- 记录 sysroot、动态链接器和运行库版本。

### B1：可解释启动

- 保存完整串口启动日志。
- 标注 Bootloader、Kernel 解压/启动、DTB、root mount、init 和服务阶段。
- 故意设置错误 root 参数或禁用服务，比较不同失败层的证据。

### B2：设备树与驱动

- 为一个 GPIO、I2C/SPI 传感器或虚拟 platform 设备补节点。
- 用 `dtc`/schema 检查、启动日志、`/proc/device-tree` 和 sysfs 形成证据链。
- 设计 probe 中途失败、remove 与异步 work/IRQ 并发、suspend 时在途事务的收敛路径。

### B3：用户态服务

- 通过标准子系统或稳定设备接口读取数据。
- 用 systemd 管理启动顺序、权限、重启限制和 watchdog。
- 配置采用原子更新，日志可限速，健康状态可查询。

### B4：系统镜像与更新

- 从干净环境重建同样镜像并保存 manifest/hash。
- 区分只读系统、持久数据、设备身份和临时运行数据。
- 模拟应用更新失败与系统更新回滚，验证持久数据兼容。

## 跨层诊断表

| 现象 | 优先证据 | 不应先做 |
|---|---|---|
| U-Boot 找不到内核 | 存储布局、环境变量、文件/分区和加载地址 | 修改内核驱动 |
| 内核早期无日志 | 串口参数、earlycon、镜像/DTB 地址和入口 | 等到 systemd 日志 |
| 根文件系统挂载失败 | cmdline、驱动、分区、文件系统和 UUID | 重写应用 |
| `probe()` 未执行 | 运行时 DT、compatible、bus、deferred probe | 在 probe 内加更多日志 |
| 有设备无 `/dev` | 子系统、udev/devtmpfs、权限和接口类型 | 直接自定义 ioctl |
| 服务反复重启 | exit status、依赖、权限、限速和 watchdog | 设置无限 `Restart=always` |

## 必做证据

- 完整启动链标注和一次分层故障对比。
- 当前运行 Kernel、DTB、RootFS、模块和应用的版本清单。
- 一份 DTS 源码、DTB 构建和运行时节点三者一致性的证明。
- 一个驱动的资源获取、异步活动和逆序清理图。
- 一个 systemd 服务的启动、失败、重启限速和日志证据。
- 一份可复现镜像的配置、工具链、依赖和 hash 记录。

## 阶段门 G2-G4

- [ ] 能从启动日志将失败定位到具体层，并说出下一条高区分度证据。
- [ ] 能证明实际加载的是哪个 Kernel、DTB 和 RootFS，而不只看源码目录。
- [ ] 驱动选择了合适标准子系统，私有 ABI 有明确必要性。
- [ ] probe defer、错误传播、remove、suspend 和异步取消有设计和测试。
- [ ] 用户态服务有权限最小化、有限重启、配置原子性和健康接口。
- [ ] 系统更新不会覆盖设备身份和持久数据，并有版本兼容与回滚策略。
- [ ] 发布镜像可追溯到源码、配置、工具链和依赖。

## 迁移与开放问题

1. 从 Buildroot 迁移到 Yocto 时，哪些系统组成与发布不变量保持不变，哪些构建概念需要重建？
2. 驱动从固定板载设备迁移到热插拔设备后，引用、remove 和用户 fd 生命周期会怎样变化？
3. systemd 自动重启让服务恢复后，如何避免重启风暴掩盖损坏配置或驱动故障？
4. A/B RootFS 回滚时，数据库、配置和证书 schema 如何保持前后兼容？

本模块的出口不是“成功启动 Linux”，而是能构建、解释、修改、验证、发布和恢复一个完整设备系统。
