# RTOS 与实时系统

RTOS 提供任务、调度、同步和时间服务，但不会自动让系统实时或可靠。本模块训练的是：如何把多个执行上下文的周期、截止期、阻塞、资源和失败关系显式化，并用分析与 trace 判断设计是否成立。

创建更多任务不是进步。只有任务边界减少了无界等待、明确了资源所有权并提高了可观察性，RTOS 才真正降低系统复杂度。

## 模块契约

### 进入条件

- 通过 MCU 阶段的启动与驱动门，至少有一个非阻塞传感器或通信模块。
- 理解中断、临界区、DMA buffer 所有权和栈的基本模型。
- 能区分平均响应时间与最坏响应时间。

### 完成后应能

- 根据周期、事件、阻塞边界和资源所有权划分任务。
- 用调度时间线和响应时间分析检查关键截止期。
- 正确选择队列、任务通知、信号量、互斥锁、事件组和定时器。
- 识别优先级反转、死锁、无界阻塞、队列积压和栈不足。
- 定义 ISR 到任务的交接、背压和过载策略。
- 使用 trace、栈水位、运行时间、队列高水位和任务心跳建立证据。
- 在队列满、任务卡死、外设超时和内存不足时让系统可诊断地收敛。

### 本模块不解决

- 不把某个 FreeRTOS/RT-Thread API 作为唯一实现；机制可迁移，配置细节需查对应版本文档。
- 不承诺优先级越高响应就一定越快；阻塞、临界区和中断会共同决定结果。
- 不用看门狗替代任务健康、错误恢复和根因记录。

## 贯穿项目：从裸机采集器到 RTOS 节点

### 第一步：先测裸机基线

- 记录采样周期、最大采样耗时、通信阻塞和错误率。
- 列出当前主循环里所有等待和共享变量。
- 明确引入 RTOS 要解决的具体问题。

### 第二步：按时间与所有权拆分

为每个候选任务填写：

| 字段 | 必须回答 |
|---|---|
| 触发 | 周期、事件还是请求？ |
| 截止期 | 多久完成仍有业务意义？ |
| WCET | 最坏执行时间怎样估计或测量？ |
| 阻塞 | 等什么、最长多久、超时后怎样处理？ |
| 资源 | 拥有哪些外设、buffer、锁和状态？ |
| 通信 | 数据是值、事件、令牌还是共享对象？ |
| 健康 | 哪些进展才能证明任务正常？ |

### 第三步：制造压力

- 暂停消费者制造队列满。
- 延长低优先级任务持锁时间制造优先级反转。
- 缩小任务栈触发栈保护。
- 让外设永不完成，检查超时和取消。
- 让日志输出变慢，观察它是否污染实时路径。

### 第四步：比较 RTOS 前后

RTOS 化应有可测收益，例如关键路径最大响应更小、阻塞边界更清楚、错误隔离更好。若只是文件和任务数量增加，就需要重新审视拆分。

## 原语选择不是 API 记忆

| 需要表达的语义 | 首选候选 | 评审问题 |
|---|---|---|
| 传递有值消息 | 队列/stream buffer | 复制还是传指针？满时如何处理？ |
| 一次事件唤醒 | 任务通知/二值信号量 | 事件能否合并？丢一次是否可接受？ |
| 共享资源互斥 | mutex/服务任务 | 是否会睡眠？是否需要优先级继承？ |
| 多条件状态 | 事件组/状态机 | bit 是状态还是事件？谁清除？ |
| 周期触发 | 绝对延时/硬件 timer | 漂移、抖动和回调上下文是什么？ |
| ISR 交接 | FromISR API/无锁 SPSC | 是否请求切换？过载如何可见？ |

原语正确不等于系统语义正确。队列能可靠复制消息，但无法决定缓存满时应丢状态、保事件还是阻塞控制任务。

## 必做证据

- 任务表：周期、截止期、优先级、WCET、栈、阻塞点和健康指标。
- 阻塞链：任务等待对象、对象拥有者、最坏持有时间和超时策略。
- 通信图：ISR、任务、DMA、队列、锁和 buffer 所有权。
- trace：至少一次正常负载和一次压力负载下的调度记录。
- 资源记录：任务栈水位、heap 最低值、队列高水位和丢弃计数。
- 故障报告：队列满、任务卡死、外设超时和栈不足中至少三项。

## 阶段门 G3

- [ ] 每个任务都有存在理由，不按“一个功能一个任务”机械拆分。
- [ ] 关键任务的优先级和截止期有分析或测量依据。
- [ ] 所有锁和等待都有顺序、上限或恢复策略。
- [ ] 队列满行为由数据语义决定，并能通过指标观察。
- [ ] ISR 使用正确上下文 API，且 ISR 到任务的过载路径被验证。
- [ ] 栈和 heap 预算来自水位/压力证据，而不是一次正常运行。
- [ ] 看门狗只在必要任务都取得有意义进展时喂狗。
- [ ] 复位后可以知道哪个任务、哪个资源或哪个截止期先失效。

## 迁移与开放问题

1. 从单核 FreeRTOS 迁移到双核 MCU 后，哪些“关中断即可保护”的结论会失效？
2. tickless 降低空闲功耗后，时间精度、唤醒延迟和驱动假设如何变化？
3. 网络栈内部有自己的线程和锁时，应用任务优先级还能否单独决定端到端时延？
4. 若队列需要覆盖 24 小时离线数据，为什么问题已经从 RTOS 参数变成存储与数据语义架构？

RTOS 阶段的出口不是“会用原语”，而是能解释系统在最坏负载和失败条件下如何调度、阻塞、恢复和留下证据。
