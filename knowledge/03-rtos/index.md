# RTOS

RTOS 解决的是嵌入式系统中的并发、实时性和资源管理问题。学习 RTOS 的重点不是记 API，而是知道什么逻辑该放在哪个执行上下文中，以及如何避免优先级、栈、锁和中断带来的隐性问题。

## 从裸机到 RTOS

裸机通常是：

```text
while (1) {
  poll_button();
  read_sensor();
  update_display();
  upload_data();
}
```

RTOS 通常拆成：

```text
sensor_task
display_task
network_task
control_task
logger_task
```

拆任务不是为了“看起来高级”，而是为了让不同周期、不同优先级、不同阻塞行为的逻辑互不拖累。

## 使用 RTOS 前先问

- 系统是否有多个周期差异很大的功能？
- 是否需要等待多个外部事件？
- 是否有明确实时响应要求？
- 是否存在网络、文件系统、显示等会阻塞的组件？
- 团队是否能管理好多任务复杂性？

简单控制器不一定需要 RTOS；复杂联网设备通常需要。

## 推荐推进顺序

1. 先用裸机实现功能闭环。
2. 把不同周期和阻塞行为的逻辑拆成任务。
3. 用队列、事件组和互斥锁替代全局变量互相访问。
4. 加入日志、栈监控、任务心跳和看门狗。
5. 做故障注入：队列满、任务卡死、外设超时、内存不足。

在工程项目中，还要理解 [RTOS 移植与配置](rtos-porting-config.md)，尤其是 tick、中断优先级、任务栈、heap 和调试钩子。

## 深挖问题

- 一个任务的存在理由是什么：周期、事件、阻塞边界还是资源所有权？
- 高优先级任务为什么仍然可能被低优先级路径拖住？
- 队列满、锁等待、栈溢出和内存不足分别应该由谁发现？
- 看门狗是在证明系统健康，还是只是在证明某段代码还会运行？

## 阶段产出

- 一张任务表：优先级、周期、栈大小、阻塞点、共享资源。
- 一张任务通信图：队列、事件组、信号量、互斥锁和 ISR 通知。
- 一份故障注入报告：队列满、任务卡死、栈不足、外设超时。
