# RTOS 工程实践

RTOS 项目真正的难点在工程边界：任务怎么拆、优先级怎么定、共享资源怎么管、故障怎么恢复、长稳怎么验证。

## 任务拆分原则

不要按“代码文件”拆任务，要按执行特征拆。

| 执行特征 | 推荐形式 |
|---|---|
| 高频、强实时、短逻辑 | 高优先级周期任务或中断下半部 |
| 低频采样 | 普通周期任务 |
| 阻塞网络 | 独立网络任务 |
| 慢速存储 | 独立存储任务 |
| 日志输出 | 低优先级日志任务 |
| 多模块状态协调 | 状态机任务 |

反例：

- 每个驱动都创建一个任务。
- 所有业务都堆在一个 mega task。
- 中断里直接做协议解析和网络发送。

## 优先级设计

优先级不是越高越好。推荐从实时性和阻塞风险判断：

| 优先级 | 任务类型 |
|---:|---|
| 最高 | 安全保护、看门狗健康监控、硬实时控制 |
| 高 | 中断下半部、关键采样、控制闭环 |
| 中 | 普通传感器、状态机、命令处理 |
| 低 | 网络上传、文件存储、日志 |
| 最低 | 统计、清理、后台维护 |

判断标准：

- 如果延迟 10 ms 会出事故，它应较高。
- 如果会阻塞等待网络，它不应高。
- 如果持有共享资源时间长，它不应高。
- 如果只是打印日志，它应低。

## 通信模型选择

| 需求 | 机制 |
|---|---|
| 传递数据 | 队列 |
| 通知事件 | 二值信号量、任务通知 |
| 等待多个条件 | 事件组 |
| 保护共享资源 | 互斥锁 |
| 管理固定资源数量 | 计数信号量 |
| 低开销 ISR 到任务通知 | direct task notification |

任务通知通常比队列/信号量更轻，但只能通知指定任务，不适合复杂数据传递。

## 共享资源管理

I2C、SPI、Flash、日志输出都是共享资源。

两种模式：

### 互斥锁模式

多个任务直接调用驱动，访问前拿锁。

优点：简单。

缺点：任务可能因外设阻塞，锁持有时间难控。

### 服务任务模式

所有请求发给专用任务，由它串行访问资源。

```text
sensor_task ─┐
display_task ├── request_queue -> i2c_service_task -> I2C
config_task ─┘
```

优点：资源访问集中，易统计和限流。

缺点：代码结构稍复杂。

关键总线和慢速 Flash 推荐服务任务模式。

## 任务心跳

看门狗不要无脑定时喂。应由健康监控确认关键任务都在推进。

```c
typedef enum {
    HB_SENSOR = 0,
    HB_NET,
    HB_CONTROL,
    HB_STORAGE,
    HB_COUNT
} heartbeat_id_t;

static volatile uint32_t heartbeat[HB_COUNT];

void heartbeat_kick(heartbeat_id_t id)
{
    heartbeat[id]++;
}
```

watchdog_task 周期检查：

- 关键任务心跳是否变化。
- 队列是否长期满。
- heap 是否低于阈值。
- 错误计数是否异常。

只有系统健康才喂狗。

## 队列满怎么办

队列满是设计问题，不应简单忽略。

处理策略：

| 数据类型 | 队列满策略 |
|---|---|
| 高频遥测 | 丢弃旧数据或合并最新值 |
| 告警事件 | 持久化或阻塞短时间 |
| 控制命令 | 返回 busy 或错误 |
| 日志 | 按等级丢弃低优先级日志 |

要统计队列满次数，否则问题会被隐藏。

## 内存策略

推荐：

- 任务、队列、定时器尽量静态创建。
- 网络和协议大 buffer 集中管理。
- 使用固定块内存池。
- 运行期避免频繁 `malloc/free`。
- 为每个任务记录最小剩余栈。

内存池示例：

```text
packet_pool:
  block_size = 256 bytes
  block_count = 16
```

比随意动态分配更可预测。

## 日志设计

RTOS 中日志建议异步化：

```text
task -> log_queue -> logger_task -> UART/RTT/File
```

日志等级：

- ERROR：故障和恢复。
- WARN：异常但可恢复。
- INFO：状态变化。
- DEBUG：调试细节。

日志中不要打印超大数据；需要时使用 hexdump 并限制长度。

## 长稳测试

至少覆盖：

- 24/72/168 小时运行。
- 高频采样。
- 网络反复断开恢复。
- MQTT broker 重启。
- Flash 写满或写失败。
- 低电压/电源波动。
- 外设断开。
- OTA 重复升级。

观测指标：

- 重启次数。
- heap 最小值。
- 任务栈最小值。
- 队列最大占用。
- 错误码计数。
- 平均/最大延迟。
