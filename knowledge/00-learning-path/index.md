# 学习路线总览

嵌入式学习最大的风险不是知识太少，而是学习顺序与能力目标脱节：会使用某块开发板，却解释不了启动和内存；会创建 RTOS 任务，却算不清阻塞和截止期；会连接 MQTT，却处理不了重复命令、缓存满和 OTA 回滚。

本模块先建立学习方法、路线和证据标准。它的退出条件不是读完所有方法论，而是能选择一条路线、完成一次可复现实验，并开始用目标、约束、模型和证据组织学习。

## 模块契约

### 进入条件

- 能在 macOS、Linux 或 WSL 中管理文件并运行终端命令。
- 可以使用 Git 获取和保存项目；完全不会 C 语言也可以从环境页开始。
- 愿意保留失败记录，而不是只保存最终成功截图。

### 完成后应能

- 区分共同基础、方向主线、横切能力和扩展专题。
- 选择 MCU/RTOS 或 Embedded Linux/BSP 作为第一条主线，并说明原因。
- 把“理解、稳定、实时、低功耗”等目标改写为可判定结果。
- 为一个问题写出竞争假设、预测、观测和结论边界。
- 完成至少一个仓库实验，保留环境、命令、失败和回归证据。
- 使用阶段门决定继续、回补或跳读，而不是按页数衡量进度。

### 模块产出

- 一份个人路线选择与 8-12 周阶段计划。
- 一份主机环境和工具版本记录。
- 一份可复现实验报告，包含一次主动制造的失败。
- 一份初始能力矩阵，标注“有证据”“只了解”“尚未接触”。

## 路线选择不是技术高低排序

### MCU/RTOS 路线

适合直接控制外设、资源受限、低功耗、确定性响应和成本敏感设备。重点不是“代码更底层”，而是必须对每个字节、每个中断和每段阻塞负责。

首个系统里程碑：从复位到采集一个真实传感器，能够用 map、寄存器、日志和总线波形解释完整链路。

### Embedded Linux/BSP 路线

适合复杂 SoC、网关、HMI、多媒体和需要完整用户态生态的设备。重点不是“会 Linux 命令”，而是理解启动链、设备模型、内核/用户态边界和系统镜像的可追溯性。

首个系统里程碑：启动一个可复现系统，标注 Bootloader、Kernel、RootFS 和服务阶段，并完成一个设备树或驱动修改的运行时验证。

### IoT 产品路线

IoT 是建立在 MCU/RTOS 或 Linux 系统能力上的产品路线。若没有本地系统基础，联网后很容易把设备故障、网络故障、云端故障和业务语义混在一起。

首个系统里程碑：在断网、重复命令和服务不可达时，设备仍维护本地安全状态，并能用状态机和指标解释发生了什么。

## 四阶段能力循环

无论选择哪条路线，都反复经历四次能力升级。

### 阶段一：能解释

核心问题：代码如何变成机器行为？

- C 对象、地址、生命周期、对齐和未定义行为。
- CPU、MMIO、中断、DMA、cache 和并发。
- 编译、链接、ELF、镜像、启动和内存布局。
- 电源、时钟、复位、引脚、手册和原理图。

退出标准：能用产物和原始资料解释系统，而不是只引用框架 API。

### 阶段二：能实现并验证

核心问题：如何在真实平台上实现主路径，并证明机制符合预期？

- MCU 路线实现启动、外设、传感器和中断/DMA。
- Linux 路线实现启动链、DTS、驱动或用户态服务。
- 对关键行为使用目标板日志、寄存器、波形、trace 或系统状态验证。

退出标准：至少一个真实模块同时具有接口契约、主路径证据和错误路径。

### 阶段三：能处理失败和取舍

核心问题：并发、弱网、存储、功耗和资源压力下系统如何收敛？

- 定义超时、背压、丢弃、重试、降级和恢复。
- 计算队列、栈、Flash、功耗、带宽或截止期预算。
- 主动注入队列满、设备无响应、断网、断电和资源不足。
- 记录方案选择、代价、证据和残余风险。

退出标准：不是“故障后能重启”，而是能说明故障如何检测、隔离、恢复、记录和回归。

### 阶段四：能交付和迁移

核心问题：如何让其他人、其他设备和未来版本可靠复用当前成果？

- 可复现构建、自动测试、发布和版本追溯。
- 量产、校准、身份注入、升级、回滚和现场诊断。
- 换平台、换负载、换器件或换协议后识别不变量和适配点。
- 面向开发、硬件、测试、运维和面试场景表达证据。

退出标准：交付物在作者不在场时仍能被构建、验证、运行、诊断和回滚。

## 制定个人学习计划

不要按章节数量排期，按项目里程碑排期。一个 10 周的 MCU/RTOS 示例：

| 周次 | 能力焦点 | 里程碑和证据 |
|---|---|---|
| 1 | 环境、C、测试 | 主机严格构建、ring buffer、失败回归 |
| 2 | ELF、内存、手册 | map/反汇编分析、板卡电源与最小系统标注 |
| 3 | 启动、GPIO、UART | 断点到 `main`、启动日志、GPIO/UART 波形 |
| 4 | I2C/SPI 与传感器 | 读 ID、NACK/超时、波形和驱动契约 |
| 5 | 中断、Timer、DMA | 时序测量、buffer 所有权和过载实验 |
| 6 | RTOS 核心 | 任务表、队列、互斥、trace 和栈水位 |
| 7 | RTOS 故障与低功耗 | 队列满、任务卡死、能量模型和实测 |
| 8 | 网络和状态机 | 本地 broker、分片解析、断网退避和指标 |
| 9 | 配置和 OTA | 断电不变量、分区、校验、首次启动确认 |
| 10 | CI、发布和复盘 | 自动测试、发布包、故障复盘和岗位表达 |

Linux/BSP 路线可以用相同结构替换第 3-7 周：交叉构建、启动链、RootFS、DTS、驱动生命周期和 systemd 服务。时间可以伸缩，但“模型、实践、失败、证据、项目”五部分不能被压缩成只读正文。

## 每周不是输入输出分离

推荐每个主题在 2-3 天内完成一个小闭环：

1. 做诊断性前测，暴露当前模型。
2. 阅读边界和机制，写出运行前预测。
3. 完成最小路径，保存原始证据。
4. 改变一个变量或主动制造失败。
5. 解释证据，记录仍未排除的假设。
6. 把结论加入当前项目、测试或接口契约。

阅读和实践间隔太久，会让读者产生“看懂等于会做”的错觉；只做不读，则容易把平台偶然行为当成通用规律。

## 模块退出检查

进入基础能力模块前，确认：

- [ ] 已选择第一条主线，并知道另一条路线不是必需前置。
- [ ] 已完成环境 90 分钟闭环，保存了工具版本和退出码。
- [ ] 已运行至少一个仓库实验，并主动制造一次失败。
- [ ] 能写出一条结论的适用条件、证据和失效条件。
- [ ] 已建立个人能力矩阵，证据不足的项目没有标记为“掌握”。
- [ ] 已为第一阶段项目建立仓库，而不是只建立笔记文件。

未通过时不要重复阅读整页。根据失败项回到环境、实验地图、系统思维或深度学习方法，补齐具体动作和证据。

## 本模块最重要的认识

路线图不是知识清单，而是复杂度管理工具。每进入下一阶段，都应该因为已有证据足以支撑新的复杂度，而不是因为上一章已经翻到末尾。

真正有效的学习进度可以这样衡量：

```text
可独立解释的机制数量
+ 可重复运行的实验数量
+ 被主动验证的失败路径数量
+ 已沉淀为工程资产的结论数量
```

当这些数量增长时，阅读速度可能变慢，但解决陌生问题的速度会开始变快。
