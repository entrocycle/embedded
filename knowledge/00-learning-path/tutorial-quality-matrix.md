# 教程质量与覆盖矩阵

这张矩阵用于评审教程，而不是要求每一页套用相同标题。章节可以按主题组织，但一组完整的学习材料应让读者经历“明确边界、理解机制、动手验证、处理失败、做出取舍、迁移结论”的闭环。

如果需要先明确“深度教程”的共同标准，阅读 [深度教程契约：从知识到洞察](depth-contract.md)。本矩阵用于把那套标准落到章节评审。

## 核心覆盖

| 维度 | 读者应能回答 | 最低证据 |
|---|---|---|
| 目标与边界 | 学完能做什么，哪些问题不在本章范围内？ | 可观察的学习目标、前置条件与适用边界 |
| 机制模型 | 数据、控制、时间和资源如何流动？ | 调用链、状态机、时序、内存布局或资源表 |
| 反例与洞察 | 哪些常见直觉会误导读者？真正的主导约束是什么？ | 反例、失效条件、约束变化后的结论修正 |
| 最小实践 | 如何用最小成本验证主路径？ | 可执行步骤、环境版本、预期现象和实际证据 |
| 边界与失败 | 在什么条件下会失效，系统如何恢复？ | 边界值、故障注入、错误码、日志或波形 |
| 设计取舍 | 为什么选择当前方案，代价是什么？ | 至少两个方案及资源、实时性、复杂度、可靠性比较 |
| 证据边界 | 当前证据能证明什么，不能证明什么？ | 竞争假设、观测点、排除记录、仍需验证的问题 |
| 验收与复盘 | 如何证明达到目标并沉淀经验？ | 指标、通过阈值、复现记录、回归用例和复盘产物 |
| 迁移与拓展 | 换平台或约束后哪些结论仍成立？ | 平台不变量、适配点、进一步追问和权威资料入口 |

## 分层标准

| 等级 | 教程表现 | 读者产出 |
|---|---|---|
| L0 说明 | 给出术语、API 和成功路径 | 能复述概念 |
| L1 解释 | 揭示资源、时序和因果关系 | 能画图并解释机制 |
| L2 验证 | 提供可复现步骤、预言和证据 | 能证明主路径并排除错误假设 |
| L3 工程 | 覆盖异常、恢复、预算和方案取舍 | 能在约束下做设计并评审风险 |
| L4 迁移 | 改变平台、规模或产品约束重新推理 | 能识别不变量和原结论的失效条件 |

主线技术章至少达到 L2，项目、调试、安全、OTA、并发与可靠性主题应达到 L3。L4 不要求给出唯一答案，但必须提供开放场景和评价标准。

## 深度评分细则

评审时不要只看“内容多不多”，而要看它是否改变读者的判断方式。

| 维度 | 不合格表现 | 合格表现 | 优秀表现 |
|---|---|---|---|
| 问题引入 | 直接讲概念 | 给出真实场景和约束 | 展示同一概念在不同约束下结论如何变化 |
| 机制解释 | 堆 API 和术语 | 说明资源、时序、状态或调用关系 | 能用模型预测异常现象 |
| 反例 | 只给成功路径 | 至少指出一个误区 | 通过反例修正读者的原始直觉 |
| 实验 | 只有操作步骤 | 有预期、观测和判定标准 | 能区分两个以上竞争假设 |
| 故障 | 简单列问题 | 有故障注入和恢复策略 | 说明哪些故障仍无法被当前证据覆盖 |
| 取舍 | 只说推荐方案 | 比较备选方案 | 明确代价、残余风险和重新评审条件 |
| 迁移 | 链接更多资料 | 指出平台适配点 | 抽象出不变量，并构造使结论失效的条件 |

## 洞察密度检查

一章不需要处处拔高，但至少应有 2-3 个能让读者停下来重新思考的点。可用下面问题检查：

- 是否指出了一个“表面问题”和“真实约束”之间的差异？
- 是否解释了一个常见做法为什么有用，以及什么时候会有害？
- 是否把一次成功运行和可靠交付之间的距离说清楚？
- 是否明确哪些证据只是相关，哪些证据能排除假设？
- 是否让读者看到成本、功耗、实时性、安全、可靠性之间的交换关系？
- 是否给出一个没有唯一答案、但有评价标准的开放问题？

## 站点一致性

- 新页面必须加入对应 `.pages`，并从章节索引或学习路线可达。
- 内部链接必须指向存在的文档；涉及版本敏感内容时优先链接官方资料并注明适用版本。
- 每章至少有一个可验证产出，不能只以“阅读并理解”作为练习。
- 每章至少有一个反例、失效条件或约束变化问题，避免把推荐做法写成绝对规则。
- 涉及实验结论时要说明证据边界：当前观测能证明什么，不能排除什么。
- 代码块要说明运行环境；伪代码要明确标识，避免被误当成可直接编译的实现。
- 同一术语、单位、错误语义和安全边界在跨章节引用时保持一致。
- 站点变更以 `mkdocs build --strict` 为最低门禁，并额外检查导航遗漏和生成后的本地链接。

## 评审记录

评审一章时只记录能被证据支持的结论：

```text
章节：
当前等级：L0 / L1 / L2 / L3 / L4
已具备的证据：
缺失的学习闭环：
最值得补的一项：
验证命令或人工检查：
```

优先补影响理解和工程判断的缺口，而不是为了让所有章节拥有相同目录。短章节可以与实验、项目或故障案例共同形成闭环，但应通过链接把这条路径明确呈现给读者。

## 如何判断一章是否激发了思考

一章内容真正激发思考，通常不是因为问题很多，而是因为问题迫使读者重建自己的模型。可以用下面标准评审：

- 读者读完后是否会改变一个原有直觉，例如“队列越大越安全”或“看门狗就是可靠性”？
- 章节是否让读者意识到某个成功现象背后还有未证明条件？
- 练习是否需要读者做预测、收集证据、解释反例，而不是只查答案？
- 内容是否展示了方案代价，让读者理解工程不是寻找唯一正确答案，而是在约束下选择可维护的失败方式？
- 章节是否把具体技术迁移到更通用的问题模式，例如所有权、时间尺度、不变量、证据链或生命周期？

如果一章只是增加信息量，读者可能“知道更多”；如果一章能改变读者提出问题的方式，读者才会“判断更准”。质量矩阵最终要服务于后者。
