# 课程依赖地图与阶段门

这套教程不是从第 00 章顺序读到第 09 章的百科全书。它采用“共同基础、方向分流、横切能力、项目汇合”的课程结构：先建立所有嵌入式方向共享的底层模型，再根据目标选择 MCU/RTOS 或 Embedded Linux/BSP 主线，最后在网络、安全、架构、发布和维护中重新汇合。

本页回答四个问题：

1. 为什么按这个顺序学习？
2. 哪些内容必须学，哪些内容应按方向选择？
3. 每一阶段做到什么才可以继续？
4. 章节、实验和项目如何共同构成能力证据？

## 课程范围

本教程面向嵌入式软件、固件、BSP/驱动和 IoT 设备工程。主线覆盖：

- C 语言、CPU、内存、编译链接和硬件阅读。
- Cortex-M 裸机、外设、中断、DMA、低功耗和 Bootloader。
- RTOS 调度、同步、移植和工程化。
- Embedded Linux 启动、设备树、驱动、RootFS 和用户态服务。
- 本地总线、IP 网络、IoT 协议、弱网、安全和 OTA。
- 调试、测试、可观测性、架构、CI、量产和现场维护。

本教程不会替代以下专业领域的完整训练：模拟电路和 PCB 设计、射频设计、控制理论、Linux 内核全部子系统、密码学证明、功能安全认证流程和机器学习算法研究。相关章节的目标是让嵌入式软件工程师理解接口、约束和协作边界，而不是假装一套教程可以覆盖所有专业。

## 总体依赖图

```text
学习方法 / 环境 / 证据意识
             |
             v
共同基础：C 与内存 -> CPU/MMIO/并发 -> 编译链接 -> 硬件与手册
             |                                      |
             +------------------+-------------------+
                                |
                 +--------------+---------------+
                 |                              |
                 v                              v
         MCU 裸机与外设                   Embedded Linux 全链路
                 |                              |
                 v                              v
         RTOS 与实时系统                  驱动模型与用户态服务
                 |                              |
                 +--------------+---------------+
                                |
                                v
              网络 / 存储 / 安全 / OTA / 可观测性
                                |
                                v
                  架构 / CI / 量产 / 现场维护
                                |
                                v
                  综合项目、岗位迁移与进阶专题
```

调试、测试、可靠性和工程记录不是图中最后一层。它们从第一个主机实验开始贯穿全程，只是证据要求逐步提高：

```text
主机输出 -> 符号和反汇编 -> 寄存器和波形 -> trace 和抓包
        -> 故障注入 -> 长稳与功耗 -> 发布记录和现场诊断
```

## 四类课程内容

| 类型 | 是否必修 | 作用 | 完成标志 |
|---|---|---|---|
| 共同基础 | 所有路线必修 | 建立跨平台的内存、CPU、构建和证据模型 | 能从源码、产物、运行时和硬件互相解释 |
| 方向主线 | 至少选择一条 | 形成 MCU/RTOS 或 Linux/BSP 的独立工作能力 | 完成该路线项目并通过阶段门 |
| 横切能力 | 全程必修 | 调试、测试、安全、可靠性、工具链和表达 | 每个项目都有相应证据，不只读过章节 |
| 扩展专题 | 按岗位和项目选修 | 存储、GUI、TinyML、功能安全等 | 能把专题接入主项目并说明代价 |

不要把“必修”理解为必须把每一页背完。必修意味着其能力必须被证明；已有经验的读者可以直接挑战阶段门，用证据决定是否跳过。

## 进入课程前的诊断

先完成下面五个任务。无法完成的部分就是最优先的回补入口。

| 诊断任务 | 合格证据 | 回补位置 |
|---|---|---|
| 严格编译一段 C 程序并让测试主动失败 | 编译命令、退出码、修复后的回归结果 | [环境准备](environment.md) |
| 解释全局变量、局部变量、常量和静态变量的位置与生命周期 | 对象表和符号/段证据 | [C 语言与内存模型](../01-foundations/c-language-memory.md) |
| 说明 `volatile`、原子性、临界区和 cache 维护分别解决什么 | 一个共享数据反例 | [CPU、内存与并发](../01-foundations/cpu-memory-concurrency.md) |
| 从原理图找到电源、复位、时钟、调试口和一个外设 | 一页 bring-up 标注 | [数据手册与原理图方法](../01-foundations/datasheet-schematic-method.md) |
| 对一个“偶发失败”提出至少三个竞争假设和区分实验 | 假设、预测和观测表 | [系统思维](system-thinking.md) |

如果只能说出答案，却拿不出证据，仍建议完成对应实验。课程评估的是可执行能力，不是自我熟悉感。

## 路线 A：MCU 与 RTOS

适合固件、工业控制、消费电子、低功耗设备、可穿戴和 MCU IoT 方向。

```text
共同基础
  -> MCU 启动与最小系统
  -> 时钟、中断、DMA
  -> GPIO/UART/SPI/I2C/ADC/Timer
  -> 传感器与执行器
  -> RTOS 核心与移植
  -> RTOS 工程、低功耗、存储
  -> 网络、OTA、安全
  -> 产品化项目
```

推荐顺序与退出证据：

| 阶段 | 主体章节 | 项目里程碑 | 退出证据 |
|---|---|---|---|
| A0 | 共同基础 | 主机 C 实验 | 严格构建、对象/段分析、故障测试 |
| A1 | MCU 启动、时钟、GPIO、UART | P0 最小固件 | 断点到 `main`、启动日志、map、GPIO 波形 |
| A2 | SPI/I2C/ADC/Timer/DMA、传感器 | P1 采集控制器 | 总线波形、超时错误、资源和时序表 |
| A3 | RTOS 核心、移植、工程 | P2 RTOS 节点 | 任务表、阻塞链、栈水位、队列满注入 |
| A4 | 网络、存储、低功耗、OTA、安全 | P3-P4 联网与升级 | 状态机、功耗预算、断电和弱网证据 |
| A5 | 架构、CI、量产、诊断 | P6 产品化闭环 | 可追溯发布包、产测、回滚、现场诊断 |

### MCU/RTOS 路线的核心不变量

- 无论是否加入 RTOS，启动、时钟、内存和外设状态都必须可解释。
- 无论吞吐多高，ISR 和 DMA buffer 的所有权都必须明确。
- 无论网络是否在线，本地实时路径和安全状态不能失控。
- 无论何时复位或断电，关键配置和可启动镜像必须满足设计不变量。
- 无论自动恢复多少次，系统都必须保留足以诊断根因的现场信息。

## 路线 B：Embedded Linux 与 BSP

适合 Linux BSP、内核驱动、系统集成、网关、HMI、摄像头和边缘计算设备。

```text
共同基础 + Linux 主机基础
  -> BootROM/SPL/U-Boot/Kernel/DTB/RootFS 启动链
  -> 内核配置、交叉构建和系统镜像
  -> 设备树、platform 和总线驱动
  -> 用户态服务、IPC、配置和日志
  -> 网络、安全、存储和应用更新
  -> 可复现 BSP 与产品化网关
```

推荐顺序与退出证据：

| 阶段 | 主体章节 | 项目里程碑 | 退出证据 |
|---|---|---|---|
| B0 | 共同基础、Linux 命令和交叉编译 | 主机与目标程序 | ELF 架构、动态依赖、交叉编译记录 |
| B1 | Linux 全链路 | 可启动最小系统 | 分阶段启动日志、内核命令行、挂载根文件系统 |
| B2 | BSP 与 RootFS | 可复现镜像 | 配置、工具链、镜像 hash 和构建记录 |
| B3 | 驱动模型 | 标准子系统或 platform 驱动 | binding、probe 证据、并发与清理测试 |
| B4 | 用户态服务与网络 | 网关服务 | systemd 状态、日志、IPC、弱网和更新证据 |
| B5 | CI、发布、诊断和安全 | 产品化网关 | SBOM/版本、发布包、回滚和远程诊断 |

### Linux/BSP 路线的核心不变量

- 启动链每一层使用的镜像、参数和版本都必须可追溯。
- 用户接口可见时，内核资源和硬件必须处于可用状态。
- IRQ、DMA、work、timer 仍可能运行时，驱动数据不得被释放。
- 策略尽量留在用户态，内核只承担必须由内核实现的机制。
- 更新系统不能破坏持久数据、设备身份和回滚路径。

## 路线 C：IoT 产品与边缘设备

IoT 不是完全独立的底层路线。读者应先选择 MCU/RTOS 或 Linux 作为设备基础，再学习连接、安全和产品生命周期。

```text
路线 A 或 B 的系统基础
  -> 本地总线与 IP 传输语义
  -> MQTT/HTTP/BLE/蜂窝等接入
  -> 连接状态机、退避与离线缓存
  -> 身份、TLS、命令授权与密钥生命周期
  -> OTA、配置迁移和回滚
  -> 云-边-设备证据链与现场运维
```

IoT 路线不能只以“云端收到一次数据”为完成标准。最低退出证据包括：

- 区分链路成功、协议成功、业务成功和物理动作成功。
- 对状态、事件、命令、文件和日志采用不同的数据语义。
- 断网、重复、乱序、过期和缓存满行为都有明确策略。
- 命令具有业务 request ID、幂等或去重语义和物理反馈边界。
- 身份、证书、时间、密钥、调试口和设备退役有生命周期设计。
- OTA 在下载、校验、切换、首次启动和业务确认阶段均可恢复。

## 横切能力的螺旋式要求

同一种能力会在不同阶段重复出现，但验收标准逐步提高。

| 能力 | 基础阶段 | 系统阶段 | 产品阶段 |
|---|---|---|---|
| 调试 | 能读取编译错误、符号和变量 | 能联合寄存器、波形、trace、抓包 | 能在现场证据不完整时远程收敛 |
| 测试 | 有可判定断言 | 覆盖边界、状态和故障注入 | 有分层回归、长稳、HIL 和发布门禁 |
| 安全 | 输入和长度校验 | 身份、权限、信任边界和安全更新 | 密钥生命周期、响应和退役 |
| 可靠性 | 处理返回值和超时 | 维护不变量、降级和恢复 | 量产差异、寿命、现场故障和残余风险 |
| 功耗 | 理解状态电流 | 建立能量模型并实测 | 联合业务、网络和电池寿命优化 |
| 表达 | 记录复现步骤 | 写接口、状态机和决策 | 面向开发、测试、硬件和运维交付证据 |

如果一个后期项目丢失了前期能力，例如联网后不再记录总线错误、加入 RTOS 后不再理解启动链，就不是能力升级，而是用新抽象掩盖旧问题。

## 七个阶段门

阶段门不是考试题，而是决定是否应继续增加系统复杂度的证据检查。

### G0：环境可复现

- 从干净目录用一条命令构建。
- 记录 OS、编译器、依赖和目标架构。
- 能让测试失败并确认非零退出码。
- 能区分构建失败、运行失败和测试失败。

### G1：基础模型成立

- 能从 C 对象推导段、地址、生命周期和访问者。
- 能解释源码如何形成 ELF、BIN、map 和启动布局。
- 能区分编译器优化、CPU 原子性、屏障、cache 和锁。
- 能从数据手册与原理图找出一个外设的物理和软件约束。

### G2：目标系统可观察

- MCU 路线：从复位到 `main()` 可解释，日志、调试器和至少一种波形证据可用。
- Linux 路线：启动日志能按 Bootloader、Kernel、RootFS 和服务分段定位。
- 发生异常时能保留版本、复位原因或内核错误上下文。

### G3：驱动与并发可证明

- 接口说明输入输出、超时、错误码、所有权和执行上下文。
- 至少一个驱动在成功和失败条件下都有目标板证据。
- RTOS 路线有任务和阻塞分析；Linux 路线有生命周期和并发清理分析。
- 队列满、超时、资源不足或异步取消至少被注入一次。

### G4：系统面对外部不确定性仍可控

- 网络、存储、传感器或用户输入失败时不会无限阻塞。
- 数据语义、缓存策略、重试和幂等边界明确。
- 功耗、容量、实时性和寿命至少完成一种定量预算。
- 关键状态机可从日志或诊断接口观察。

### G5：升级和恢复维护不变量

- 配置写入和 OTA 有断电模型、有效性判定和回滚路径。
- 镜像真实性、完整性、版本策略和首次启动确认被区分。
- 版本升级与配置 schema、证书和持久数据兼容性被评审。
- 自动恢复不会抹掉根因证据。

### G6：具备交付闭环

- 构建、测试、静态分析和核心实验进入 CI。
- 发布包可追溯到源码、工具链、配置、依赖和测试结果。
- 有产测、校准、身份注入、升级、回滚和退役策略。
- 有现场诊断命令、健康指标、故障码和复盘资产。

## 章节、实验、项目的关系

三者承担不同职责：

```text
章节：建立模型和方法
实验：检验一条具体主张
项目：让多条主张在同一组约束中相互作用
```

例如“队列”这一主题：

1. 在 RTOS 章节中理解队列的复制语义、阻塞和调度影响。
2. 在主机或目标板实验中验证 wrap、满队列和生产消费速率。
3. 在联网项目中决定遥测、告警和命令分别如何处理队列满。
4. 在产品阶段把高水位、丢弃计数和恢复策略加入现场诊断。

如果只完成第 1 步，获得的是知识；完成前 3 步，才开始形成工程能力；第 4 步使它成为可交付能力。

## 证据分级

不同证据能支持的结论不同。

| 等级 | 证据 | 可以支持 | 不能自动支持 |
|---|---|---|---|
| E0 | 阅读和口头解释 | 术语和初步模型 | 实现正确或系统可靠 |
| E1 | 主机测试、静态分析 | 纯逻辑、边界和部分不变量 | 目标架构、时序和电气行为 |
| E2 | 目标板日志、寄存器、调试器 | 固件路径和硬件状态 | 信号完整性和外部设备真实时序 |
| E3 | 波形、trace、抓包、功耗曲线 | 物理时序、调度、协议和能量行为 | 长期和规模化可靠性 |
| E4 | 故障注入、压力和长稳 | 已定义故障空间与负载下的恢复能力 | 未定义环境和所有现场风险 |
| E5 | 量产与现场数据 | 批次、寿命和真实环境表现 | 未来变化或未覆盖攻击 |

教程中的“验证通过”必须同时写明证据等级和边界。一次 E3 波形不能证明 168 小时长稳，一次 E4 长稳也不能证明任意供电、温度和器件批次都可靠。

## 建议学习节奏

每个学习单元采用同一节奏，而不是把一周完全分成“先读后做”：

```text
诊断前测
  -> 阅读边界和机制
  -> 写出预测
  -> 完成最小实验
  -> 注入一个失败
  -> 整理证据
  -> 放入阶段项目
  -> 用阶段门复盘
```

建议把时间按大致比例投入：

| 活动 | 时间占比 | 目的 |
|---|---:|---|
| 阅读和建模 | 25% | 建立可预测的机制模型 |
| 编码和实验 | 35% | 把模型转为行为 |
| 调试和故障注入 | 20% | 发现模型与现实的差异 |
| 证据、文档和复盘 | 20% | 把个人经验沉淀为可复用资产 |

比例不是硬性规定。如果实验失败，应该把时间转向假设和证据，而不是为了赶进度直接复制答案。

## 如何跳读而不破坏体系

有经验的读者可以跳过内容，但不要跳过能力验证：

1. 先挑战该模块阶段门。
2. 对每个失败项定位到具体章节或实验。
3. 只补模型、证据或实践缺口。
4. 将补齐后的产物放进当前项目。
5. 再次挑战阶段门。

这种方式比从头顺读更适合有工作经验的人，也能避免“因为做过某个 API 就认为已经理解整个主题”。
