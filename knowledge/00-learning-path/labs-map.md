# 实验地图：从主机模型到产品证据

实验的目标不是让示例“跑起来”，而是把教程中的设计主张变成可复核证据。本页组织仓库内实验、目标板任务和项目故障注入，帮助读者逐步提高证据等级。

## 三层实验体系

| 层次 | 主要用途 | 优点 | 证据边界 |
|---|---|---|---|
| 主机模型 | C、算法、状态机、序列化、不变量、边界枚举 | 快、可自动化、容易注入故障 | 不证明目标 CPU、外设、电气和实时行为 |
| 目标板实验 | 启动、寄存器、中断、DMA、总线、功耗 | 验证真实平台约束 | 单板单次结果不代表批次和长期可靠性 |
| 系统项目 | 跨模块、弱网、升级、长稳、发布、现场诊断 | 验证约束相互作用 | 只覆盖已定义场景，仍有残余风险 |

三层不是互相替代。推荐先在主机上压缩状态空间、穷举边界，再把无法由主机证明的平台假设带到目标板，最后在项目负载中验证交互。

## 仓库内可运行实验

仓库根目录的 `labs/` 保存可执行实验。每个实验都应支持严格编译、自动判定和故障练习。

| 实验 | 对应主题 | 核心主张 | 主要证据 |
|---|---|---|---|
| `host-config-journal` | Flash、配置、断电一致性 | 任意写入步骤后断电，只能恢复旧值或完整新值 | 穷举断电点、CRC 损坏和序列回绕测试 |
| `ring-buffer` | C、边界、ISR/任务通信 | 明确保留槽规则后，空/满/wrap 行为可由不变量验证 | 模型对照、边界序列和随机操作测试 |
| `byte-stream-parser` | TCP/UART 字节流、协议解析 | 分片和粘连不改变消息结果；畸形长度必须有有界失败策略 | 全切分、合并帧、短流和超长帧测试 |

运行全部实验：

```bash
make -C labs/host-config-journal test
make -C labs/ring-buffer test
make -C labs/byte-stream-parser test
```

实验源码在 `knowledge/` 目录之外，因此站点页面只负责教学说明，仓库中的 README、源码和测试才是可执行证据。发布教程时，CI 必须运行这些命令，避免文档主张和实现长期漂移。

## 推荐实验顺序

### L0：先学会让机器判定

完成 [环境准备](environment.md) 的主机实验：

- 使用严格警告编译。
- 让程序以非零退出码失败。
- 修复后运行同一条命令回归。
- 保存工具版本、命令、预期和实际结果。

这里训练的不是 C 语法，而是最小证据闭环。

### L1：环形缓冲与边界不变量

运行 `labs/ring-buffer`，先回答：

- `head == tail` 表示空时，如何区分满？
- 保留一个槽会损失多少容量，换来了什么简化？
- `size_t` 回绕和数组索引 wrap 是同一件事吗？
- 单生产者/单消费者的算法边界，为什么不等于任意多线程安全？

故意删除满检查或改变 `count` 计算，记录第一个反例。再把该模型迁移到 UART RX：由目标架构保证的原子访问宽度、ISR/任务内存可见性和错误计数仍需单独验证。

### L2：字节流解析与协议边界

运行 `labs/byte-stream-parser`，验证同一消息在每个字节切分点下都能解析。然后把两帧合并为一次输入，确认回调得到两条消息。

重点不是实现某个私有协议，而是建立三条认识：

1. TCP、UART DMA buffer 和文件读取都可能在任意边界返回数据。
2. 长度前缀解决消息边界，但畸形长度仍需要上限、错误策略和整数溢出检查。
3. 没有同步字或转义机制的长度协议在头部损坏后无法可靠流内重同步，最稳妥策略可能是关闭连接或由外层重建会话。

### L3：断电一致性

完成 [双槽配置日志](host-config-journal.md)：

- 先写出旧值保留、提交在最后和半写不可见等不变量。
- 预测每个断电点的恢复集合。
- 运行穷举测试，而不是抽样拔电。
- 把单字节原子模型替换为目标 Flash 的编程粒度和擦除粒度。

主机通过后，再设计目标板验证：查数据手册、放置配置分区、注入复位、读取真实 Flash 错误标志并记录磨损假设。

## 目标板必做实验矩阵

不同开发板寄存器和工具不同，因此教程定义实验契约，不伪装成唯一平台命令。

| 阶段 | 实验 | 必须改变的变量 | 必须保留的证据 |
|---|---|---|---|
| 启动 | 从复位停到 `Reset_Handler` 和 `main` | 错误链接地址或向量表 | PC/SP/VTOR、map、复位日志 |
| GPIO/UART | 原子置位、波特率与时钟 | 系统时钟或分频 | 寄存器、逻辑分析仪位宽、实际波特率 |
| I2C/SPI | 读器件 ID | 地址、模式、上拉或时钟 | 手册时序、波形、错误标志 |
| 中断 | 测中断响应与执行时间 | 临界区长度和负载 | GPIO 探针波形、计数、最大延迟 |
| DMA | 双缓冲或循环接收 | 消费速度和 cache 处理 | 所有权时间线、丢失计数、数据模式 |
| RTOS | 队列和锁压力 | 优先级、队列深度、持锁时间 | trace、阻塞链、栈水位、deadline |
| 低功耗 | 状态能量模型 | 唤醒周期和外设状态 | 电流曲线、状态时间、平均电流计算 |
| OTA | 下载、校验、切换和确认 | 每一阶段复位/断电 | 分区、启动决策、版本、回滚记录 |

### 目标板实验的共同格式

每次实验都记录：

```text
设计主张：
目标板 / 芯片修订 / 板卡版本：
工具链 / SDK / RTOS 或内核版本：
接线与仪器设置：
改变变量与控制变量：
运行前预测：
原始证据：
判定：
竞争解释：
本次没有证明什么：
回归测试或设计约束：
```

如果无法填写“本次没有证明什么”，通常说明结论写得过度泛化。

## 从实验进入项目

实验产物不能做完即丢弃。把它们升级为项目资产：

| 实验产物 | 项目中的去向 |
|---|---|
| map 和段分析 | 固件资源预算与发布包 |
| 总线波形 | 驱动验收基线和硬件问题对照 |
| ring buffer 边界测试 | UART、日志或网络 buffer 单元测试 |
| 队列满策略 | 数据语义和现场 overflow 指标 |
| 配置断电测试 | 配置服务、OTA schema 迁移和回滚 |
| 弱网状态机记录 | 设备健康指标和云端运维看板 |
| HardFault/dump | 现场诊断格式和符号化流程 |
| 功耗曲线 | 电池寿命预算和发布回归门槛 |

当实验结论进入接口契约、自动测试、监控指标和发布检查后，学习才真正转化为工程能力。

## 实验完成量规

| 等级 | 表现 |
|---|---|
| 0 | 运行了命令，但无法解释结果 |
| 1 | 能复现成功路径并说明主要机制 |
| 2 | 能预测结果、主动制造失败并用证据判定 |
| 3 | 能说明证据边界，补目标板验证并沉淀回归 |
| 4 | 改变平台或约束后能重建模型和实验 |

核心主线实验至少做到等级 2。进入产品项目的配置、并发、OTA、安全和低功耗实验应做到等级 3。

## 失败时如何处理

实验失败时，不要先寻找“正确输出”截图。按下面顺序收敛：

1. 记录完整命令、退出码和第一条错误。
2. 判断失败属于环境、构建、连接、运行还是断言。
3. 检查实验前置和版本基线。
4. 写出至少两个能解释现象的假设。
5. 选择成本最低且区分度最高的观测。
6. 修复后运行原失败条件，而不是只运行正常路径。
7. 把新发现加入排错说明或自动测试。

实验失败并不妨碍学习；无法复现、无法区分和没有记录的失败才会浪费学习机会。
