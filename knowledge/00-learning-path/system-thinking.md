# 用系统思维学习嵌入式

嵌入式问题很少只属于某一章。一次“传感器偶发超时”可能同时涉及供电、总线时序、中断延迟、锁竞争、缓存、重试策略和现场可观测性。系统思维的目标不是把问题说复杂，而是在信息不完整时建立一个可证伪的模型，避免凭经验不断试参数。

## 从知识点到工程判断

知识题通常有预设条件和唯一答案，工程问题却要先找出条件。

| 层次 | 典型问题 | 合格产出 |
|---|---|---|
| 事实 | I2C 的 ACK 在哪里？ | 能结合手册和波形指出位置 |
| 机制 | 为什么会 NACK？ | 能列出供电、地址、时序和器件状态等原因 |
| 诊断 | 这次 NACK 的根因是什么？ | 假设排序、观测证据和排除记录 |
| 设计 | NACK 后系统应该怎么办？ | 结合数据重要性和实时性定义恢复策略 |
| 演化 | 换器件或进入量产后是否仍成立？ | 接口边界、参数来源和回归测试 |

“知道所有可能原因”不是终点。你还要能用最低成本的观测，把多个可能世界区分开。

## 推理闭环

处理任何主题或故障时，按下面八步推进：

```text
目标 -> 约束 -> 模型 -> 假设 -> 预测 -> 观测 -> 决策 -> 沉淀
                                ^                |
                                +----------------+
```

### 1. 把目标写成可判定指标

“低功耗”“实时”“稳定”都不可直接验收，应改写成带条件的指标：

- 在 3.0 V、25 C、每 60 s 上报一次时，24 h 平均电流小于 80 uA。
- 控制任务在所有已定义负载下，端到端响应时间的最大值小于 5 ms。
- 连续运行 168 h，无非预期复位，且堆、栈和队列高水位不持续恶化。

指标至少包含对象、条件、量纲、统计口径和通过阈值。平均延迟不能证明最坏延迟，单次成功不能证明可靠性。

### 2. 区分约束、假设和偏好

| 类型 | 示例 | 处理方式 |
|---|---|---|
| 硬约束 | RAM 只有 128 KB；电机必须在 10 ms 内关闭 | 违反即方案无效 |
| 外部约束 | 云端 API、法规、器件供货、成本 | 与相关方确认并留出处 |
| 假设 | 网络每天最多断开 10 min | 必须验证，错误时重新设计 |
| 偏好 | 团队更熟悉 FreeRTOS | 可以影响选择，但不能伪装成硬约束 |

很多返工来自把假设当事实。例如离线缓存按“最多断网 10 min”设计，现场断网两天后，缓存满不是偶发 bug，而是需求模型不完整。

### 3. 建立最小模型

模型只保留能影响决策的变量。常用模型包括：

- 资源模型：Flash、RAM、栈、DMA 通道、定时器、文件描述符。
- 时间模型：周期、截止时间、最坏执行时间、阻塞时间、抖动。
- 状态模型：状态、事件、守卫条件、超时、重试和终止状态。
- 可靠性模型：失效模式、检测、隔离、恢复和残余风险。
- 信任模型：资产、攻击者能力、入口、信任边界和密钥生命周期。

模型不需要一开始就精确，但必须能产生预测。不能产生预测的图通常只是展示，不是分析工具。

### 4. 提出竞争假设

不要只写“怀疑 I2C 时序问题”，至少保留两个能解释现象的假设：

| 假设 | 如果为真，应观察到什么 | 能否被一次观测排除 |
|---|---|---|
| H1：设备未完成上电 | 首次访问失败，延时后稳定成功 | 改变启动延时并抓波形 |
| H2：上拉过弱 | 上升沿慢，升温或加长线后更严重 | 测上升时间并更换上拉 |
| H3：软件消费不及时 | 硬件 ACK 正常，但驱动报超时或 overrun | 同时看波形、IRQ 和队列指标 |
| H4：电源瞬降 | 无线发射时集中失败，电源轨有压降 | 示波器触发故障时刻 |

好的假设应该可证伪。把“系统不稳定”解释成“可能受干扰”没有预测力，也无法指导下一步。

### 5. 先写预测，再做实验

先写预测能防止看到结果后改故事。实验只改变一个关键变量，并记录：

- 控制变量和改变变量。
- 正常组与异常组。
- 预期结果及其区分能力。
- 仪器带宽、采样率、触发条件和时间基准。
- 软件、硬件、配置和工具版本。

若多个模块使用不同时间源，要先对齐时间。否则“日志先出现”不一定代表事件先发生。

### 6. 区分相关、触发和根因

一个问题可能有四个不同层次：

```text
薄弱设计：共享总线没有有界等待
触发条件：日志任务长时间占锁
故障机制：控制任务错过截止时间
可见症状：执行器保护动作延迟
```

只删除触发条件，例如“降低日志量”，可能让问题暂时消失，却没有修复无界阻塞。根因修复后还要保留能覆盖该触发条件的回归测试。

### 7. 用不变量检查设计

不变量是在所有合法路径中都必须成立的条件，比逐条枚举正常流程更能暴露漏洞。

典型不变量：

- 任意时刻至少存在一个已验证且可启动的固件镜像。
- 未验证长度和权限的输入永远不能驱动内存访问或执行器动作。
- 任何任务都不能无限持有共享资源。
- 命令重放不能让同一非幂等动作执行两次。
- 设备失去云连接时仍能进入本地安全状态。

为每条不变量反问：谁维护它？哪些操作可能破坏它？断电或复位发生在任意指令后，它还成立吗？

### 8. 把结论变成资产

一次解决问题至少应留下其中两类资产：

- 自动化测试或故障注入用例。
- 设计约束或接口契约。
- 监控指标、错误码或诊断命令。
- 数据手册勘误、板级限制或配置检查。
- 故障复盘和决策记录。

没有进入代码、测试、监控或文档约束的经验，很容易在人员或平台变化后丢失。

## 定量推理示例：队列应该多深

假设传感器每 100 ms 产生一条消息，即生产速率为 10 条/s。网络任务平时每秒消费 20 条，但每次重连会完全停止消费，实测重连时间的 99.9 分位为 2.4 s。

只覆盖重连暂停，最低积压为：

```text
backlog = producer_rate * pause_time
        = 10 msg/s * 2.4 s
        = 24 msg
```

若每条消息 40 B，队列数据区至少约 960 B，另有 RTOS 元数据和对齐开销。把长度设为 24 仍不够，因为还要回答：

- 2.4 s 之外的尾部风险如何处理？
- 重连后消费能力是否真的持续高于生产能力？
- RAM 预算是否允许覆盖最坏断网时间？
- 数据是事件还是状态，能否合并、覆盖或持久化？

如果需求是“断网 10 min 不丢告警”，单纯加 RAM 队列需要 6000 条，说明问题已经从“队列参数”变成“数据分级和持久化架构”。计算的价值不是给出一个漂亮数字，而是尽早暴露方案边界。

## 设计决策记录

关键选择使用一页短记录，避免后来只记得结论：

```text
决策：
日期与状态：proposed / accepted / superseded
问题与可判定目标：
硬约束与待验证假设：
候选方案：
选择及证据：
放弃方案及原因：
代价和残余风险：
触发重新评审的条件：
验证与回滚方式：
```

“使用 MQTT”不是完整决策。完整决策要说明消息语义、网络条件、资源开销、失败行为以及何时应改用其他方案。

## 开放式推演：远程水泵控制器

场景约束：设备使用蜂窝网络，控制水泵，云端可下发启停命令；现场可能断网 48 h，传感器可能漂移，继电器可能粘连，设备由电池和太阳能供电。

先不要写代码，完成下面推演：

1. 定义“安全状态”。断网时保持原状态、立即停机还是本地自治，分别会造成什么风险？
2. 列出至少 8 个假设，并指出其中哪 3 个最值得先验证。
3. 画出数据流、控制流、错误流和信任边界；四张图中哪些节点不同？
4. 为“云端显示已关闭但继电器粘连”设计独立证据链。命令 ACK 能证明物理动作完成吗？
5. 如果成本要求删掉电流传感器，软件还能检测哪些故障，哪些风险只能接受或用其他硬件缓解？
6. 设计三条系统不变量，并给出会破坏它们的故障注入。

这道题没有唯一架构。评价重点是：目标是否可判定，假设是否暴露，风险是否与证据对应，取舍是否承认代价。

## 自我评审量规

| 等级 | 表现 |
|---|---|
| 0 复述 | 能重复术语和推荐做法 |
| 1 解释 | 能画机制图并解释因果 |
| 2 验证 | 能设计实验区分竞争假设 |
| 3 设计 | 能在约束下比较方案并定义不变量 |
| 4 迁移 | 条件变化后能重建模型，并说明原结论为何失效 |

完成一章时不要笼统地问“我会了吗”，而要选择一个结论，至少做到等级 2；核心项目中的关键决策应达到等级 3 或 4。

## 练习

1. 从最近遇到的一个问题中分离症状、触发条件、故障机制和薄弱设计。
2. 选择教程中的一个 API 示例，为它写出资源、时间、并发和错误契约。
3. 把一个“系统要稳定运行”的需求改写为至少五个可判定指标，并说明它们仍遗漏了什么。
4. 找一个自己认同的技术结论，构造一个能让它失效的约束变化。

## 系统思维的常见误区

系统思维不是把所有因素都列出来，也不是把问题画成复杂大图。真正有用的系统模型应该帮助你做两件事：减少无效尝试，发现局部优化带来的全局代价。

常见误区包括：

- 把相关性当根因：修改日志后故障消失，不代表日志是根因，可能只是改变了调度时序。
- 把局部成功当系统成功：传感器读数正确，不代表采样、缓存、上报、告警和恢复链路可靠。
- 把平均值当最坏值：平均响应时间好看，不能证明控制路径满足截止期。
- 把“能恢复”当“可维护”：设备自动重启后能工作，不代表现场能知道为什么重启。
- 把复杂方案当高级方案：如果没有明确约束驱动，更多状态、更多线程和更多抽象只会增加失效面。

一个成熟的系统模型通常是可删减的：它只保留能改变决策的变量。学习时要不断问自己：这个变量会影响哪个判断？如果没有它，模型会错误预测什么？如果回答不上来，它可能只是噪声。
