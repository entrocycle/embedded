# 主机实验：双槽配置日志与断电一致性

这个实验用 C11 内存数组模拟 Flash，在 macOS、Linux 或 WSL 上枚举配置更新的每个断电点。它把“断电安全”从一句设计主张变成可重复失败、可自动判定的证据。

完整源码、逐步练习和评价量规位于仓库的 [`labs/host-config-journal`](https://github.com/entrocycle/embedded/tree/main/labs/host-config-journal)。在仓库根目录执行：

```bash
cd labs/host-config-journal
make test
make sanitize
```

## 先预测，再运行

运行前先回答：

1. 一次记录更新包含多少个可断电的原子步骤？
2. 任一步骤后重启，允许读到哪些值？哪些结果绝不允许出现？
3. CRC、提交标记和序列号各自证明什么，又不能证明什么？
4. 写入函数返回失败时，新记录是否一定没有持久化？

把预测写进实验记录，再运行测试。看到 `7/7 tests passed` 只说明实现满足当前模型中的断言；还要解释每组测试排除了哪个竞争假设。

## 证据边界

主机模型能够验证：

- 记录编码、CRC 检查和序列号回绕逻辑。
- 两槽轮换与“最后提交”的状态转换。
- 模型定义的每个 erase/program 步骤后的恢复不变量。
- 非法输入、数据损坏和 sanitizer 能观察到的软件错误。

它不能直接证明：

- 真实 Flash 的擦除与编程粒度、对齐和耗时。
- 电压缓慢下降时 CPU、brown-out 和 Flash 的行为。
- cache、DMA、中断和多任务并发下的可见性与互斥。
- 磨损寿命、恶意篡改、防回滚和量产器件差异。

完成主机实验后，按 [测试与验证方法](../06-debug-optimization/testing-verification.md) 的目标板验证阶段，把模型假设转换成数据手册条目、实板故障注入和可复核记录。

## 最终产出

保留四项内容：初始预测、一次被测试推翻的判断、断电点统计证据、迁移到具体芯片仍需验证的假设。实验的价值不是记住双槽实现，而是学会区分“代码看起来合理”“模型内得到验证”和“产品约束下证据充分”。

## 从主机模型迁移到真实设备

主机实验的价值在于把“断电一致性”抽象成可枚举状态，但它不能替代目标板验证。迁移到真实 MCU 或 Linux 设备时，要重新检查：

- Flash 擦除和写入是否只能从 1 写成 0，program unit 是否要求对齐。
- 掉电时电压下降曲线是否允许完成当前写入，还是可能产生半写、乱写或读回不稳定。
- 写入缓存、文件系统缓存、块设备 cache 是否让 `write()` 成功与介质落盘之间存在距离。
- 配置区擦写寿命是否能承受实际更新频率，是否需要磨损均衡。
- 恢复默认配置在产品语义上是否安全，还是应该停机、降级或等待人工介入。

这类迁移训练能帮助读者区分“不变量”和“实现假设”。不变量是任意时刻不能加载半写配置；实现假设则包括写入粒度、掉电行为、CRC 位置和启动扫描规则。平台变化时，不变量通常保留，假设必须重验。
