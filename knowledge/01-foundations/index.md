# 基础能力

嵌入式基础不是 C 语法、Linux 命令和硬件名词的集合，而是建立一条可以双向推理的链路：从源码推到编译产物、CPU 访问和电路行为，也能从异常地址、总线波形和复位现象反推软件假设。

本模块是 MCU/RTOS 与 Embedded Linux/BSP 的共同前置。它不要求一开始掌握所有架构细节，但要求读者停止把内存、构建、并发和硬件条件当成框架会自动处理的背景。

## 模块契约

### 进入条件

- 已完成 [环境准备](../00-learning-path/environment.md) 的主机闭环。
- 能用严格警告编译 C 程序，并能阅读第一条编译错误。
- 已选择第一条系统路线，但本模块对两条路线都必修。

### 完成后应能

- 为 C 对象标注地址、长度、生命周期、所有权和访问上下文。
- 区分 `volatile`、原子性、临界区、锁、cache 维护和内存屏障。
- 从源码追到预处理、目标文件、ELF、map、镜像和加载地址。
- 解释 `.data`、`.bss`、栈、堆、只读数据和启动初始化的关系。
- 从原理图和手册确认电源、复位、时钟、引脚、电气与外设约束。
- 用编译产物、反汇编、手册条目或测量证据验证判断。

### 本模块不解决

- 不系统教授通用 C 语言全部语法和标准库。
- 不替代数字电路、模拟电路和 PCB 设计课程。
- 不在没有目标平台保证时承诺原子访问宽度、cache 一致性或寄存器行为。

## 贯穿任务：解释一段数据的完整旅程

选择一个 32 位传感器采样值，从物理信号一直追到最终变量：

```text
传感器/ADC
  -> 总线或寄存器
  -> 外设状态和 DMA/CPU 访问
  -> RAM buffer
  -> C 对象和类型
  -> 协议编码或日志
```

对每一步记录：

- 数据表示、字节序、对齐和单位。
- 生产者、消费者和所有权移交。
- 可能发生的截断、越界、竞态和过期。
- 可以观察它的寄存器、内存、波形或测试。
- 当前证据可以排除什么，仍不能排除什么。

这个任务会在 MCU、RTOS、Linux 和网络章节不断扩展。

## 必做实践

1. 运行 `labs/ring-buffer`，先预测空、满和 wrap，再故意破坏满检查。
2. 编译包含全局、静态、常量和局部对象的程序，检查符号和段。
3. 对比 `-O0` 与 `-O2` 的反汇编，找出不能仅凭源码顺序推断的地方。
4. 选择一块板卡，画出电源树、复位、时钟、启动脚和调试接口。
5. 为一个 I2C/SPI/UART 外设建立手册笔记，包含电气、时序、寄存器和 Errata。

## 证据包

模块完成时至少提交：

- `object-map.md`：五类 C 对象的位置、生命周期和访问者。
- `firmware-map.md`：段大小、关键符号、加载地址和 RAM/Flash 预算。
- `board-bringup.md`：板卡版本、电源、复位、时钟、调试口和第一个外设。
- `ring-buffer-report.md`：不变量、失败注入、测试结果和目标板待验证假设。
- 一条从软件现象追到手册或电路条件的完整证据链。

## 阶段门 G1

- [ ] 能解释一个对象为什么位于 `.data`、`.bss`、`.rodata` 或栈。
- [ ] 能从 ELF/map 判断 Flash 与 RAM 占用，而不只看源文件大小。
- [ ] 能指出读改写何时发生竞争，以及专用 SET/CLEAR 寄存器为何更安全。
- [ ] 能说明 `volatile` 不提供原子性、互斥、cache 一致性或跨核顺序。
- [ ] 能从手册找到目标外设的时钟、引脚、状态位、错误位和时序限制。
- [ ] 能把主机测试结论与目标板仍待验证的架构假设分开。

如果对象和段无法解释，回补 C 与编译链接；如果并发工具职责混淆，回补 CPU 与内存；如果只能照 SDK 配置外设，回补手册与原理图方法。

## 深挖与迁移

完成后尝试回答：

1. 同一份 C 源码换编译器、优化级别、ABI 或 CPU 后，哪些行为由标准保证，哪些必须重新验证？
2. 主机上的互斥与原子经验迁移到 ISR、DMA 和双核 MCU 时，模型要增加哪些参与者？
3. 某批板卡低温启动失败时，源码、链接、时钟、电源和复位证据应按什么顺序收集？
4. 如果协议结构体不能直接映射到线上的字节，序列化层应承担哪些边界检查和版本责任？

本模块真正的出口不是“会 C”，而是能把代码放回真实机器中理解。
