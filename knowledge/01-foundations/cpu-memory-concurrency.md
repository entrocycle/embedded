# CPU、内存映射与并发基础

寄存器、DMA、中断、RTOS 锁和 Linux 驱动看似分属不同章节，底层却共享同一组问题：CPU 按什么顺序观察内存，外设如何与 CPU 交换数据，多个执行上下文如何共同修改状态。缺少这层模型时，代码常常“不开优化能跑、打开优化就坏”或“单步正常、全速异常”。

## 学习目标与边界

完成本章后应能：

- 区分地址、存储对象、总线事务和外设副作用。
- 解释 `volatile`、原子操作、临界区、锁和内存屏障分别解决什么问题。
- 根据 CPU、DMA、cache 和中断之间的所有权画出数据流。
- 用反汇编、寄存器、波形和压力测试验证并发假设。

本章建立通用模型，不替代具体芯片的架构手册。寄存器访问宽度、cache 维护范围、屏障要求和原子指令支持必须以目标平台手册为准。

## 从地址到一次访问

C 代码中的指针只是一个数值解释。一次读写是否有效，还取决于地址映射、访问宽度、权限、对齐和总线响应。

```text
C 表达式
  -> 编译器生成 load/store
  -> CPU 地址与权限检查
  -> cache / 总线矩阵
  -> SRAM、Flash 或外设
  -> 返回数据或触发副作用
```

同一个地址空间可能包含：

| 区域 | 典型性质 | 访问风险 |
|---|---|---|
| Flash | 非易失、写擦除受限 | 等待周期、擦写粒度、执行时写入限制 |
| SRAM | 读写快、掉电丢失 | 栈越界、并发覆盖、ECC 错误 |
| 外设 MMIO | 访问会触发硬件事务 | 读清零、写 1 清零、保留位、访问宽度限制 |
| 外部内存 | 经控制器或总线访问 | cache、一致性、时序、映射属性 |

因此，`*(uint32_t *)addr = value` 只有在你能回答以下问题时才是完整设计：

- `addr` 对应什么资源，当前时钟和电源域是否开启？
- 允许 8、16 还是 32 位访问，是否要求对齐？
- 该次读写有什么副作用？
- 是否可能被 ISR、另一个核或 DMA 同时访问？
- 写入何时真正到达设备，后续动作是否依赖它完成？

## MMIO 与 `volatile`

外设寄存器通常通过 `volatile` 指针访问：

```c
#define UART_SR (*(volatile uint32_t *)0x40011000u)

while ((UART_SR & UART_TX_READY) == 0u) {
}
```

`volatile` 告诉编译器每次抽象机访问都不可随意省略或合并，适合“值可能在正常 C 数据流之外变化”的对象，例如 MMIO 和由 ISR 更新的简单标志。

但它不保证：

- 复合操作原子。
- 多核或多任务互斥。
- 跨对象的先后关系满足协议。
- CPU cache 与 DMA 自动一致。
- 硬件总线事务已经完成。

```c
volatile uint32_t count;
count++;  /* 通常是 load -> add -> store，不是不可分割操作 */
```

若任务和 ISR 都执行 `count++`，两次更新可能丢失。修复方案取决于上下文和平台：短临界区、C11 原子、RTOS 原语或单一所有者模型，而不是再多加一个 `volatile`。

## 读改写风险

设置寄存器某一位常写成：

```c
REG |= MASK;
```

这会产生“读整个寄存器、修改、写回整个寄存器”。以下情况可能出错：

- 其他上下文在读写之间改变了其他位。
- 寄存器包含 write-one-to-clear 位，写回读到的 `1` 会意外清标志。
- 保留位不允许写入非复位值。
- 硬件为该功能提供专用 SET/CLEAR 寄存器。

优先使用芯片提供的原子置位/清零寄存器。若只能读改写，要查手册语义，并分析并发上下文。

## 原子性、可见性与顺序

并发问题至少有三个维度，不能互相替代：

| 维度 | 问题 | 常用机制 |
|---|---|---|
| 原子性 | 操作会不会被观察到一半，更新会不会丢失 | 原子指令、临界区、锁 |
| 可见性 | 一个执行者的修改何时被另一个看到 | 原子语义、cache 维护、同步原语 |
| 顺序性 | 数据和状态标志是否按协议顺序被观察 | acquire/release、内存屏障、设备屏障 |

典型生产者/消费者协议：

```c
payload = new_value;
ready = 1;
```

消费者看到 `ready == 1` 时，必须也能看到对应的 `payload`。在简单单核 MCU 上它可能长期“看起来没问题”，但编译器优化、多核、cache 或较弱内存模型会暴露漏洞。正确方式是使用队列、锁或带 release/acquire 语义的原子变量建立 happens-before 关系。

```c
#include <stdatomic.h>

static int payload;
static atomic_bool ready;

void publish(int value)
{
    payload = value;
    atomic_store_explicit(&ready, true, memory_order_release);
}

bool try_consume(int *value)
{
    if (!atomic_load_explicit(&ready, memory_order_acquire)) {
        return false;
    }
    *value = payload;
    return true;
}
```

注意：并非所有嵌入式工具链和目标都能无锁实现所有 C11 原子类型。用 `atomic_is_lock_free`、反汇编和库实现确认成本；ISR 中尤其不能意外调用会拿锁的运行库函数。

## 中断不是线程，但仍然并发

单核 CPU 在任一瞬间只执行一条指令，并不代表代码没有并发问题。ISR 可以在任务的任意可抢占点插入。

```c
/* task */
head = (head + 1u) % SIZE;

/* ISR */
head = (head + 1u) % SIZE;
```

两个上下文共享写 `head` 会丢更新。更好的环形缓冲设计是单生产者只写 `head`，单消费者只写 `tail`，并明确：

- 索引读写在该平台是否天然原子。
- 写入数据与发布新索引之间的顺序。
- 满和空如何区分。
- 溢出时丢新、丢旧还是报告故障。

关中断可以构成单核 MCU 的短临界区，但临界区时间必须有上界。它不能替代多核锁，也不能解决 DMA 一致性。

## DMA、cache 与所有权

DMA 是另一个内存访问主设备。一个 buffer 在生命周期中应有明确所有者：

```text
CPU_PREPARE
  -> cache clean（若需要）
  -> DMA_OWNED
  -> DMA completion
  -> cache invalidate（若需要）
  -> CPU_OWNED
```

常见错误：

- TX 前未 clean，DMA 读到 RAM 中的旧数据。
- RX 后未 invalidate，CPU 继续读 cache 中的旧数据。
- DMA 运行时 CPU 修改同一 cache line 的其他对象，维护操作造成误伤。
- buffer 在栈上，函数返回后 DMA 仍在使用。
- 描述符和数据的发布顺序不符合 DMA 控制器要求。

cache 维护通常要求地址按 cache line 对齐并覆盖完整范围。不要机械调用“全 cache clean”，它可能掩盖所有权错误并带来不可接受的实时开销。

## 屏障解决什么问题

屏障有不同层次：

- 编译器屏障：限制编译器重排，不一定产生 CPU 指令。
- CPU 内存屏障：约束处理器观察内存访问的顺序。
- 设备或同步屏障：确保特定访问在后续动作前完成。
- 指令同步屏障：修改控制寄存器、向量表或代码后刷新执行流水线。

CMSIS 中常见 `__DMB()`、`__DSB()`、`__ISB()`，但不能凭名称随处添加。先从架构手册找出协议要求，再选择屏障；错误同步往往仍能在当前测试中“工作”，却留下性能损失或不可移植假设。

## 字节序、对齐和协议边界

不要把 C 结构体直接当作持久化或网络协议：

```c
typedef struct {
    uint8_t type;
    uint32_t value;
} message_t;
```

它可能包含填充，整数也受字节序影响。编译器、ABI 或字段变化会让格式不兼容。可靠做法是显式序列化：

- 规定每个字段的宽度、字节序、单位和范围。
- 验证长度后再解析字段。
- 版本化格式，并定义未知字段的行为。
- 对外部输入做溢出检查，例如 `offset + length` 本身可能溢出。

`packed` 只能改变布局，不能解决字节序、版本和未对齐访问风险。

## 实验一：观察编译器

准备三组代码：普通变量、`volatile` 变量、C11 原子变量。在 `-O0`、`-Og`、`-O2` 下编译并反汇编：

```bash
arm-none-eabi-gcc -mcpu=cortex-m4 -mthumb -O2 -S ordering.c
arm-none-eabi-objdump -dr ordering.o
```

记录：

1. 轮询是否被优化。
2. `count++` 对应几条指令。
3. 不同 memory order 是否生成屏障。
4. 某原子类型是否调用运行库。

反汇编只能证明当前编译器、选项和目标的实现，不能把偶然结果误写成 C 语言保证。

## 实验二：DMA 所有权故障

在带 D-cache 的目标上建立 DMA loopback 或外设收发实验：

1. 使用对齐 buffer，按手册执行 clean/invalidate，记录基线。
2. 故意省略维护操作，增加压力和数据模式变化。
3. 故意让两个对象共享 cache line，观察维护范围错误。
4. 记录 DMA 状态、buffer 地址、cache line 大小和首个错误位置。

若目标没有 D-cache，不应伪造该现象；改做 ISR 与任务共享环形缓冲实验，并说明平台差异。

## 深入思考

1. 如果 `volatile` 不能解决并发，为什么设备寄存器仍然普遍需要它？
2. 单生产者/单消费者环形缓冲何时可以无锁？增加第二个生产者后，哪个不变量先失效？
3. 全局关中断能让代码变“线程安全”吗？从中断延迟、多核和 DMA 三个角度反驳这个说法。
4. 为什么 cache 一致性 bug 常在加日志或单步调试后消失？这类现象能提供什么线索，不能证明什么？
5. 若一个协议字段在 MCU 上是 32 位自然对齐，是否应直接把网络 buffer 转成结构体指针？列出至少四个失败条件。

4. 用“原子性、可见性、顺序性”重新分析一个任务与 ISR 的共享变量。

## 工程洞察

并发问题之所以难，不是因为语法复杂，而是因为多个执行主体对“当前状态”的理解不一致。CPU、DMA、ISR、任务、编译器、cache 和外设寄存器都可能参与同一条数据路径；如果没有所有权模型，系统只能依赖运气。

分析共享数据时，至少区分三件事：

- 原子性：一次读写是否会被中断或另一个主体观察到中间状态。
- 可见性：一个主体写入的数据，另一个主体何时能看见。
- 顺序性：多个读写是否会按设计者想象的顺序发生。

`volatile` 只影响编译器对访问的保留和合并，不能自动提供互斥、内存屏障、cache 维护或事务语义。`REG |= MASK` 看似单条语句，可能是读、改、写三步；如果寄存器含有 write-one-to-clear 位，或另一个上下文同时修改同一寄存器，就会出现隐藏破坏。

开放问题：

1. 一个标志位由 ISR 设置、任务清除。即使用了 `volatile`，为什么仍可能丢事件？
2. DMA 写入 buffer 后触发中断，CPU 在中断里立刻读取。带 cache 的系统为什么可能读到旧数据？
3. 如果一个 32 位计数器在 8 位 MCU 上被 ISR 更新，主循环如何安全读取？禁中断、双读校验和原子库各有什么代价？
