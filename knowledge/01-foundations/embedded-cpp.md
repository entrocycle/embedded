# 嵌入式 C++：对象生命周期与零开销抽象

C++ 在嵌入式中的价值，不是把桌面应用写法搬到 MCU，而是用类型、作用域、构造析构、模板和 RAII 更准确地表达资源、生命周期与接口约束。它也会引入静态初始化、异常、RTTI、动态分配、ABI 和代码体积等新风险，因此“能不能用 C++”不是信仰问题，而是工具链、平台和产品约束下的工程决策。

## 本章契约

### 前置知识

- 能解释 C 对象的存储期、作用域、链接属性和内存段。
- 理解栈、堆、构造顺序、函数指针和 C ABI 的基础概念。
- 能查看 ELF 符号、map 文件和反汇编。

### 学完后应能

- 区分“语言抽象本身零额外开销”和“某种用法会生成运行时成本”。
- 用 RAII 管理锁、IRQ 状态、文件描述符和临时外设事务，同时识别析构上下文风险。
- 控制全局对象构造、异常、RTTI、虚函数、模板和动态分配的成本。
- 设计稳定的 C/C++ 边界，理解名称改编、对象布局和 ABI 兼容风险。
- 用 map、反汇编、栈/heap 测量和故障测试验证选择，而不是按经验争论。

### 本章边界

本章不系统教授现代 C++ 全部特性，也不主张所有嵌入式项目改用 C++。裸机、RTOS、Linux 用户态和 Linux 内核的可用语言子集不同；Linux 内核主线仍以 C 为主，不能将本章实践直接迁移到内核驱动。

## 先从约束决定语言策略

| 场景 | C++ 可能带来的价值 | 主要风险 |
|---|---|---|
| 小型裸机 MCU | 类型安全寄存器、静态多态、RAII | 启动构造、代码体积、库支持 |
| RTOS 固件 | 任务/锁/队列封装、状态机、所有权 | 析构阻塞、异常跨任务、heap 策略 |
| Linux 用户态 | 标准库、容器、线程、测试生态 | 动态分配、延迟尾部、依赖和 ABI |
| Bootloader/安全启动 | 强类型解析、范围检查 | 运行时和攻击面应尽量小 |
| Linux 内核 | 不属于常规选择 | 内核构建、运行时和编码约束不支持通用 C++ |

决策至少记录：编译器和标准版本、是否启用异常/RTTI、标准库实现、动态分配策略、全局构造规则、代码体积预算和 C ABI 边界。

## “零开销”到底是什么意思

零开销原则通常表达两点：不使用的特性不应付费；使用某个抽象时，其成本不应高于手写等价低层实现。但它不表示任何 C++ 代码都没有成本。

例如一个 `constexpr` 位域函数和内联模板可能生成与 C 宏相同的机器码，并额外获得类型检查：

```cpp
#include <cstdint>

template<std::uint32_t Mask, unsigned Shift>
constexpr std::uint32_t field(std::uint32_t value)
{
    return (value << Shift) & Mask;
}

static_assert(field<0x000000f0u, 4u>(3u) == 0x30u);
```

但下面的设计可能产生显著成本：

- 虚调用需要 vptr/vtable，并可能阻碍内联。
- 异常需要 unwind 表、运行时和明确的跨边界策略。
- `std::function` 可能发生类型擦除和动态分配，具体取决于对象与实现。
- 容器扩容会分配内存、复制/移动对象并产生不可预测尾延迟。
- iostream、locale 和格式化库可能引入大量代码。
- 模板在多个类型上实例化可能造成代码膨胀。

成本必须通过目标工具链产物测量。只凭源码风格判断“高级”或“低效”都不可靠。

## 对象生命周期是核心模型

C++ 增加了构造和析构，资源可以和对象生命周期绑定，但也增加了隐式执行路径。

| 对象 | 构造时机 | 析构时机 | 嵌入式风险 |
|---|---|---|---|
| 自动对象 | 进入作用域 | 离开作用域，包括提前返回 | 栈占用、析构是否阻塞 |
| 静态局部对象 | 首次执行到声明 | 程序结束时，固件中通常不重要 | 线程安全 guard 和首次调用延迟 |
| 全局对象 | `main()` 前 | 运行时退出 | 初始化顺序、硬件尚未就绪 |
| 动态对象 | `new` | `delete` | 失败、碎片、所有权和实时性 |

### 全局构造顺序问题

不同翻译单元的动态初始化顺序通常不能依赖。更危险的是全局构造函数可能在时钟、heap、日志或外设准备好之前运行。

更稳妥的策略：

- 优先使用常量初始化、`constexpr` 和平凡对象。
- 将硬件初始化放在显式 `init()` 阶段，返回可检查错误。
- 避免全局构造函数访问外设、锁、日志和动态内存。
- 用链接 map 和启动调用链确认 `__libc_init_array` 或等价过程。

构造函数不能返回错误。可能失败的硬件初始化通常不适合完全藏在构造中。

## RAII：把资源释放绑定到作用域

RAII 的价值是让所有退出路径都恢复资源状态。例如保存并恢复中断屏蔽状态：

```cpp
class IrqGuard {
public:
    IrqGuard() noexcept : previous_(read_primask()) { disable_irq(); }
    ~IrqGuard() noexcept { write_primask(previous_); }

    IrqGuard(const IrqGuard&) = delete;
    IrqGuard& operator=(const IrqGuard&) = delete;

private:
    std::uint32_t previous_;
};
```

这个示例表达了嵌套恢复不变量，但真实实现需使用目标架构 intrinsic，并评审：

- 析构是否一定在允许恢复中断的上下文发生？
- guard 能否被移动、跨任务保存或动态分配？这里通过禁止复制减少误用，但还要控制作用域。
- 临界区最坏持续时间是否有上限？RAII 保证退出，不保证时间足够短。
- 多核系统中屏蔽本核中断是否足以保护共享数据？通常不够。

适合 RAII 的资源还包括 mutex、文件描述符、映射内存、事务片选和临时电源域引用。析构函数应尽量 `noexcept`、有界、不可阻塞；在 ISR 或实时路径中尤其需要明确。

## 静态多态与动态多态

### 虚函数

虚函数适合运行期需要替换实现、对象数量可控且代码清晰收益明显的场景。成本包括对象 vptr、间接调用、vtable 和可能的链接保留。

评审时不要简单禁止，而应测量：

- 每个对象增加多少 RAM。
- vtable 和未使用虚函数占多少 Flash。
- 热路径能否接受间接调用和不可内联。
- 对象销毁是否需要虚析构。
- ABI 是否跨动态库、Bootloader/App 或固件版本边界。

### 模板与 CRTP

模板可以在编译期选择实现并保持内联，但每种实例化都可能复制代码。适合类型少、性能敏感和接口可在编译期决定的场景。

不要把静态多态当作绝对优于虚函数。若十种类型实例化一段大函数，Flash 成本可能高于共享虚实现。用 map 和 `nm --size-sort` 比较真实结果。

## 异常和错误语义

很多 MCU 项目使用 `-fno-exceptions`，原因包括代码体积、运行时支持、最坏延迟和团队策略。但关闭异常并不会自动得到良好错误处理。

无异常项目仍需定义：

- 构造不能完成时如何表示无效对象。
- 分配失败、驱动超时和协议错误如何传播。
- `noexcept` 函数内部出现不可恢复错误时如何处理。
- 第三方库是否在禁用异常配置下仍然正确。
- 标准库或 operator `new` 失败的具体行为。

候选方式包括错误码、结果类型、工厂函数和状态对象。重点是调用方不能无意忽略失败，错误中保留足够上下文，并且失败路径可测试。

## RTTI、转换与类型边界

若不使用 `dynamic_cast` 和 `typeid`，可考虑 `-fno-rtti`，但应确认第三方库要求。关闭 RTTI 不是安全边界，错误的 `static_cast` 仍可造成未定义行为。

协议和持久化数据不要直接依赖 C++ 对象布局：

- vptr、padding、继承布局和 ABI 可能变化。
- 对象可能不满足平凡可复制条件。
- 字节序、对齐和版本兼容仍需显式编码。

使用独立 wire/storage schema 和编码函数，不把 `sizeof(Class)` 当协议长度。

## 动态分配策略

“C++ 一定用 heap”是误解，语言和标准库可以在静态内存策略下使用。但必须审计哪些组件会隐式分配。

常见策略：

1. 完全禁止运行期动态分配，并在链接或运行时捕获 `new`。
2. 只在初始化阶段分配，调度启动后冻结 heap。
3. 使用固定容量容器、对象池或 arena。
4. Linux 用户态允许动态分配，但监控失败、增长和尾延迟。

固定容量把“分配失败”转化为可提前预算的“容量满”，但容量满仍需业务策略。静态分配也可能浪费 RAM；禁止 heap 不是免费可靠性。

## C 与 C++ 边界

中断向量、厂商 HAL、RTOS hook 和跨模块稳定 ABI 常需要 C linkage：

```cpp
extern "C" void USART1_IRQHandler(void)
{
    uart_irq_dispatch();
}
```

边界原则：

- 对外使用固定宽度整数、字节 buffer 和显式长度。
- 不让 STL 类型、异常或 C++ 对象布局穿过 C ABI。
- 明确对象由哪一侧创建和销毁，避免 allocator 不匹配。
- 不让异常穿越 `extern "C"` 边界。
- 若 Bootloader 和 App 分别构建，函数表 ABI 必须版本化并验证地址范围。

## 编译与产物验证

建立至少四种构建对比：

```text
C 基线
C++ 最小特性
C++ + 虚函数
C++ + 异常/标准库组件
```

在同一工具链、优化和链接选项下记录：

- `.text/.rodata/.data/.bss`。
- 最大栈和动态内存行为。
- 构造表、vtable、typeinfo 和异常表符号。
- 热路径反汇编与调用方式。
- 启动时间和最坏执行时间。

常用检查：

```bash
arm-none-eabi-size firmware.elf
arm-none-eabi-nm -C --size-sort firmware.elf
arm-none-eabi-readelf -S firmware.elf
arm-none-eabi-objdump -d -C firmware.elf
```

名称 demangle 只帮助阅读，不改变 ABI。版本变化后必须重新测量，不能永久引用一次结果。

## 故障注入

- 让内存池耗尽，检查对象创建和上层降级。
- 在不同翻译单元创建相互依赖全局对象，观察并修复初始化顺序依赖。
- 故意让 RAII guard 作用域过大，用 GPIO 或 trace 测量临界区。
- 向固定容量容器写满，验证数据语义与指标。
- 构造一个 C/C++ allocator 不匹配，解释为何接口必须规定释放方。
- 改变编译器、LTO 或标准库版本，比较 ABI 与固件大小。

## 设计检查清单

- [ ] C++ 标准、异常、RTTI、线程安全静态初始化等策略已记录。
- [ ] 全局构造不访问尚未准备的硬件和服务。
- [ ] 资源类不可被意外复制，移动和销毁语义清楚。
- [ ] 析构有界、不可抛异常，执行上下文可接受。
- [ ] 动态分配与容量满有可测试策略。
- [ ] C ABI 不泄漏 C++ 对象布局和异常。
- [ ] 代码体积、栈、heap 和热路径成本由产物测量支持。
- [ ] 协议和持久化格式独立于对象内存布局。

## 练习

1. 用 C 宏、`constexpr` 函数和模板分别实现寄存器位域，比较类型错误和反汇编。
2. 为 mutex 或 IRQ 状态实现最小 RAII guard，注入提前返回并证明资源恢复。
3. 比较虚接口与模板接口在三种实现类型下的 Flash、RAM 和调用代码。
4. 写一个不使用异常的 `Result<T, Error>` 或项目已有等价接口，讨论它是否会被调用者忽略。
5. 为 C 驱动库设计一层 C++ 包装，但保持底层 ABI 和 buffer 所有权明确。

## 深入思考

1. “禁止 C++”会消除哪些风险，又会把哪些生命周期和类型约束转回人工管理？
2. RAII 让释放更可靠，为什么仍不能证明实时性和死锁安全？
3. 固定容量容器比动态容器更可预测，但容量选择错误时，失败会以什么业务形式出现？
4. 编译期抽象减少运行期开销后，为什么可能增加构建时间、代码尺寸和诊断复杂度？

## 项目产出

在现有项目中选择一个资源，例如 SPI 事务、mutex、文件描述符或电源域，提交：

- C 与 C++ 两种接口的生命周期和错误契约。
- 产物大小与热路径反汇编对比。
- 一次失败注入和回归结果。
- 采用或拒绝 C++ 方案的决策记录及重新评审条件。

本章的目标不是让读者站队，而是把语言选择变成有约束、有测量、有失败策略的工程决策。
