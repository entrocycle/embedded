# 寄存器与外设驱动方法

外设驱动的本质是按数据手册要求配置寄存器，让硬件按预期收发信号。HAL、LL、SDK 都是在这件事上做抽象。

## 本章任务与边界

完成本章后，你应能从需求和手册设计一个有超时、有错误语义、有所有权契约的驱动，按轮询、中断、DMA 的复杂度逐级实现，并用寄存器、波形和压力数据验证主路径与恢复路径。

本章代码包含伪代码和跨平台接口示例，不可在未核对时钟、引脚、寄存器特殊语义、Errata 与执行上下文前直接用于产品。HAL 返回成功也不自动证明总线、对端设备和业务动作成功。

完成证据：一个真实外设的接口契约、成功波形、至少三种错误注入，以及轮询/中断/DMA 的选择记录。驱动完成后继续进入 [裸机架构](baremetal-architecture.md)，验证它不会让主循环无界阻塞。

## 本章学习方式

学习外设驱动时，不要只追求“某个外设能跑”。真正要训练的是从需求到证据的闭环：

```text
需求
  ↓
数据手册约束
  ↓
寄存器配置
  ↓
信号或数据流
  ↓
异常处理
  ↓
验证证据
```

每写一个驱动，都要能回答：它占用哪些引脚、时钟、DMA、中断和 buffer；它什么时候超时；它如何报告错误；它如何证明自己真的按手册工作。

## 驱动开发步骤

一个外设驱动通常按以下步骤开发：

1. 明确需求：输入输出、速率、精度、延迟、功耗。
2. 阅读手册：时钟、引脚复用、寄存器、状态位、中断/DMA。
3. 设计接口：初始化、读写、控制、回调、错误码。
4. 写最小路径：先轮询，再中断，再 DMA。
5. 加异常处理：超时、忙状态、错误标志、复位恢复。
6. 验证信号：用串口日志、逻辑分析仪、示波器确认。
7. 工程化：封装、测试、文档、移植层。

### 从需求反推驱动形态

同一个外设会因为需求不同而采用完全不同的实现。

| 需求 | 更可能的实现 | 关键风险 |
|---|---|---|
| 上电早期输出少量日志 | UART 轮询发送 | 阻塞时间、波特率是否已稳定 |
| 命令行交互 | UART 中断 + 环形缓冲 | 丢字节、行解析、回显阻塞 |
| 高速连续数据 | UART/SPI DMA | 半包处理、cache、buffer 所有权 |
| 低功耗周期采样 | 传感器中断唤醒 + 定时器 | 唤醒源、GPIO 漏电、采样稳定时间 |
| 控制闭环 | 定时器/PWM/ADC 同步 | 抖动、延迟、优先级 |

因此驱动设计不能脱离指标。没有速率、延迟、功耗、错误处理和验证方式，就很难判断实现是否合适。

## GPIO

GPIO 是理解外设的入口。一个引脚至少涉及：

- 端口时钟。
- 模式：输入、输出、复用、模拟。
- 输出类型：推挽、开漏。
- 上拉/下拉。
- 输出速度。
- 复用功能编号。

输出控制伪代码：

```c
void led_init(void)
{
    rcc_enable_gpioa();
    gpio_set_mode(GPIOA, 5, GPIO_MODE_OUTPUT);
    gpio_set_output_type(GPIOA, 5, GPIO_PUSH_PULL);
}

void led_set(bool on)
{
    if (on) {
        GPIOA->BSRR = 1u << 5;
    } else {
        GPIOA->BSRR = 1u << (5 + 16);
    }
}
```

优先使用原子置位/复位寄存器，避免读改写影响其他引脚。

GPIO 调试不要只看 LED 亮灭。更可靠的证据包括：

- 输出寄存器和输入寄存器是否一致。
- 引脚复用是否被其他外设占用。
- 上拉/下拉是否符合外部电路。
- 复位期间引脚默认状态是否安全。
- 用示波器确认边沿、毛刺和电平幅值。

## UART

UART 是最重要的调试通道。

关键参数：

- 波特率。
- 数据位。
- 校验位。
- 停止位。
- 硬件流控。

推荐从三种模式逐步实现：

| 模式 | 优点 | 缺点 | 适用 |
|---|---|---|---|
| 轮询 | 简单 | 阻塞 CPU | 启动早期日志 |
| 中断 | 响应及时 | ISR 和缓冲管理复杂 | 命令行、低速通信 |
| DMA | CPU 占用低 | 半包/空闲中断处理复杂 | 高速数据流 |

UART 接收推荐使用环形缓冲：

```c
typedef struct {
    uint8_t *buf;
    uint16_t size;
    volatile uint16_t head;
    volatile uint16_t tail;
} ring_buffer_t;

bool rb_push(ring_buffer_t *rb, uint8_t ch)
{
    uint16_t next = (uint16_t)((rb->head + 1u) % rb->size);
    if (next == rb->tail) {
        return false;
    }
    rb->buf[rb->head] = ch;
    rb->head = next;
    return true;
}
```

环形缓冲还要明确满了怎么办：

| 策略 | 行为 | 适用 |
|---|---|---|
| 丢新数据 | 保留旧数据，返回 overflow | 命令行、不能破坏已有帧 |
| 覆盖旧数据 | 保留最新数据 | 传感器流、状态流 |
| 阻塞等待 | 等消费者取走 | 任务上下文，不适合 ISR |
| 扩大缓存 | 降低溢出概率 | RAM 足够且峰值可预测 |

ISR 中通常不能阻塞，所以必须把 overflow 作为可观测事件记录下来。

## SPI

SPI 重点是时序匹配。

需要确认：

- 主从模式。
- CPOL/CPHA。
- MSB/LSB first。
- 片选控制时序。
- 最大时钟频率。
- 读写命令格式。

调试 SPI 必须看波形：

- SCK 是否存在。
- CS 是否在整个传输期间有效。
- MOSI 命令是否正确。
- MISO 是否有响应。
- CPOL/CPHA 是否与芯片手册一致。

SPI 的一个常见误区是只看代码配置，不看从设备手册的时序图。很多器件要求 CS 拉低后等待 `tCSS`，最后一个时钟后保持 `tCSH`，读数据前还要 dummy byte。驱动接口如果只暴露 `spi_transfer(tx, rx, len)`，上层就必须明确谁负责 CS 和这些间隔。

## I2C

I2C 常见问题多，调试时关注：

- 上拉电阻是否存在且阻值合理。
- 地址是 7 位还是 8 位写法。
- ACK/NACK 出现在哪个阶段。
- 总线是否被从设备拉死。
- 时钟频率是否超过器件能力。

典型读寄存器流程：

```text
START
  slave_addr + W
  register_addr
RESTART
  slave_addr + R
  read data
STOP
```

I2C 驱动必须能区分不同失败阶段：

| 阶段 | 现象 | 可能原因 |
|---|---|---|
| START 失败 | 总线忙或超时 | SDA/SCL 被拉低、外设未复位 |
| 地址 NACK | 从设备无响应 | 地址错误、电源、上拉、7/8 位混淆 |
| 寄存器地址后 NACK | 命令不支持 | 寄存器地址错误、设备状态不允许 |
| 读阶段异常 | 少读、多读或超时 | 长度、状态机或时序错误 |
| STOP 后总线不释放 | 从设备异常 | 需要总线恢复或电源复位 |

把所有错误都返回 `-1` 会让后续定位非常痛苦。

ACK 的发送方取决于传输方向：地址和写数据后由从设备 ACK/NACK；读数据时由主机在每个字节后回应。主机读取最后一个字节后发送 NACK，再发 STOP 或 repeated START，是正常结束信号，不应记录成“从设备读数据失败”。抓波形时要同时标注方向、字节序号和 ACK 的发送者。

## ADC

ADC 不只是“读一个数”。要关注：

- 参考电压。
- 分辨率。
- 采样时间。
- 输入阻抗。
- 校准。
- 多通道扫描顺序。
- DMA 缓冲区。

工程处理：

- 采样前保证信号稳定。
- 多次采样平均或滤波。
- 对关键电压做校准。
- 避免数字噪声影响模拟输入。

## DMA

DMA 让外设和内存之间直接搬运数据，适合：

- UART 高速收发。
- ADC 连续采样。
- SPI 屏幕刷新。
- 音频数据流。

关键概念：

- 方向：外设到内存、内存到外设、内存到内存。
- 地址递增：外设通常不递增，内存通常递增。
- 数据宽度：byte/halfword/word。
- 模式：普通、循环、双缓冲。
- 中断：半传输、传输完成、错误。

常见错误：

- DMA buffer 放在不可访问区域。
- cache 未清理/失效导致数据不一致。
- 数据宽度配置错误。
- DMA 和 CPU 同时访问同一 buffer 未同步。

### DMA buffer 生命周期

DMA 驱动设计的核心不是启动传输，而是管理 buffer 状态。

```text
FREE -> DMA_OWNED -> READY -> CPU_OWNED -> FREE
```

- `FREE`：可以交给 DMA。
- `DMA_OWNED`：CPU 不应修改，cache 需要按方向处理。
- `READY`：DMA 完成，等待任务处理。
- `CPU_OWNED`：CPU 解析数据，完成后释放。

如果没有这个状态模型，很容易出现 CPU 读到半包、DMA 覆盖未处理数据、cache 中旧数据被当成新数据的问题。

带 D-cache 的目标还要明确 cache line 所有权。TX 通常在交给 DMA 前 clean；RX 不能只在完成后 invalidate，因为 DMA 开始前若该 line 存在 CPU dirty 数据，后续回写可能覆盖 DMA 新数据。应使用芯片或 OS 的 DMA API 按方向执行“准备设备访问”和“恢复 CPU 访问”，并满足地址、长度和 cache line 对齐要求。DMA 拥有期间 CPU 不得读写该 buffer，也不要让无关对象共享同一 cache line。

## Watchdog

看门狗不是“定时喂一下”这么简单。它应该检测系统是否真的健康。

建议：

- 不要在中断中无条件喂狗。
- 由健康监控模块汇总关键任务心跳。
- 某个关键任务卡死时不喂狗，让系统复位。
- 记录复位原因，启动后上报或保存。

## 驱动接口设计

一个可维护驱动应包含：

```c
typedef enum {
    SENSOR_OK = 0,
    SENSOR_ERR_TIMEOUT,
    SENSOR_ERR_BUS,
    SENSOR_ERR_PARAM,
} sensor_status_t;

typedef struct {
    int (*read_reg)(uint8_t reg, uint8_t *buf, uint16_t len);
    int (*write_reg)(uint8_t reg, const uint8_t *buf, uint16_t len);
    void (*delay_ms)(uint32_t ms);
} sensor_port_t;

sensor_status_t sensor_init(const sensor_port_t *port);
sensor_status_t sensor_read(float *value);
```

好处：

- 驱动逻辑和具体平台解耦。
- 方便单元测试。
- 方便从 STM32 移植到 ESP32、Linux 用户态或 RTOS。

### 接口文档最低要求

驱动头文件不应该只列函数名。至少写清楚：

- 初始化前置条件：时钟、GPIO、电源、复位脚由谁配置。
- 并发规则：是否可重入，是否允许 ISR 调用，是否需要外部锁。
- 超时单位：tick、毫秒还是硬件计数。
- 错误码语义：总线错误、设备无响应、参数错误要区分。
- 资源占用：中断号、DMA 通道、静态 buffer、任务栈估算。
- 验证方式：可读 ID、波形、错误计数、压力测试。

## 故障注入清单

| 外设 | 故障 | 期望行为 | 观测证据 |
|---|---|---|---|
| GPIO | 经限流电阻改变负载或上拉 | 进入已定义保护行为，不损坏 IO | 电流、温升、输入寄存器、电平波形 |
| UART | 波特率错误或 RX 溢出 | 记录帧错/溢出，不死机 | 错误计数、逻辑分析仪 |
| SPI | CPHA 配错 | 读 ID 失败，返回明确错误 | MOSI/MISO 波形 |
| I2C | 拔掉传感器 | 地址 NACK，驱动超时返回 | I2C 波形、错误码 |
| ADC | 输入悬空 | 数据被标记不可用或波动异常 | 采样分布、原理图 |
| DMA | buffer 未对齐 | 初始化失败或断言 | 错误日志、配置检查 |

故障注入不能制造新的硬件危险。禁止把推挽输出直接硬短到 GND、VDD 或另一个反向输出；应先查绝对最大额定值、单引脚和端口总电流，用限流电阻、电子负载或带保护的测试夹具逐步施加负载，并监测电流与温升。涉及市电、电机和电池大电流时，应由具备相应资质和防护条件的人员完成。

## 深入思考

1. 为什么一个“平台无关传感器驱动”仍然需要 `delay_ms`、`read_reg`、`write_reg` 这类平台回调？
2. 如果 UART 日志和 AT 模组共用一个串口，会出现哪些架构问题？
3. I2C 总线被从设备拉死时，软件有哪些恢复手段？哪些必须依赖硬件电源控制？
4. DMA 看起来降低 CPU 占用，但为什么可能增加系统复杂度和延迟？

## 练习

1. 写一个平台无关的 I2C 温度传感器驱动接口。
2. 用逻辑分析仪抓一次 I2C 读寄存器波形，标注地址、ACK、数据。
3. 将 UART 接收从轮询改为中断环形缓冲。
4. 给你的驱动增加错误计数结构体，并设计一个调试命令读取这些计数。

## 工程洞察

外设驱动不是寄存器配置清单，而是硬件能力、软件接口和错误恢复之间的契约。好的驱动应让上层知道它能保证什么，不能保证什么；同时把硬件异常转化为可诊断、可恢复、可统计的状态，而不是让所有失败都变成 `-1` 或死循环。

驱动设计要明确五类契约：

- 资源契约：使用哪些时钟、引脚、DMA、中断、buffer 和电源状态。
- 时间契约：调用是否阻塞，超时是多少，ISR 与回调在哪个上下文执行。
- 并发契约：是否可重入，是否允许多任务访问，谁持有总线锁。
- 错误契约：NACK、timeout、overrun、CRC 错、bus busy 分别如何返回和恢复。
- 观测契约：有哪些计数器、最后错误、状态转移和调试命令。

驱动抽象的深度应由复用需求和硬件差异决定。过薄会让上层到处依赖寄存器细节，过厚会隐藏关键时序和错误信息。工程判断在于找到能保护上层语义、又不丢失诊断能力的边界。
