# MCU 裸机开发

MCU 裸机阶段训练对启动、时间、外设和物理信号的直接控制。即使后续使用 HAL、RTOS 或厂商 SDK，这里的模型仍是定位启动失败、中断异常、DMA 丢数、总线超时和低功耗问题的基础。

“裸机”不意味着不做架构和错误处理。主循环、ISR、DMA 和硬件本身已经构成并发系统；如果阻塞、所有权和恢复没有被命名，复杂性只是被藏在全局变量和延时函数里。

## 模块契约

### 进入条件

- 通过基础阶段门 G1。
- 有一块带调试器和串口的 Cortex-M 或同类 MCU 开发板。
- 能找到芯片 datasheet、reference manual、编程手册、Errata 和板卡原理图。

### 完成后应能

- 从复位向量解释 CPU 如何进入 `Reset_Handler` 和 `main()`。
- 配置并验证时钟、GPIO、UART、Timer、中断、I2C/SPI、ADC/PWM 和 DMA 主路径。
- 为驱动定义输入输出、超时、错误码、buffer 所有权和执行上下文。
- 联合寄存器、调试器、逻辑分析仪和示波器定位外设问题。
- 设计非阻塞主循环与中断协作，处理过载和错误恢复。
- 建立低功耗能量模型和 Bootloader/OTA 的启动不变量。

### 本模块不解决

- 不要求脱离厂商 HAL 重写所有外设；要求能穿透抽象验证关键行为。
- 不把某个 STM32、ESP32 或 nRF SDK 的 API 当成跨平台标准。
- Bootloader 和低功耗在本模块建立机制，产品级安全与回归在后续模块完成。

## 推荐顺序

| 阶段 | 章节 | 新增复杂度 | 关键证据 |
|---|---|---|---|
| M0 | [启动流程与最小系统](mcu-startup.md) | 复位、向量表、运行时初始化 | PC/SP/VTOR、map、启动日志 |
| M1 | [时钟、中断与 DMA](clock-interrupt-dma.md) | 时间基准、异步事件、数据搬运 | 测频、中断探针、所有权图 |
| M2 | [寄存器与外设驱动](peripheral-driver.md) | 多种总线和接口契约 | 寄存器、波形、超时和错误码 |
| M3 | [裸机架构](baremetal-architecture.md) | 非阻塞状态机、时间和过载 | 主循环延迟与事件队列证据 |
| M4 | [传感器与执行器](sensor-actuator.md) | 物理量、校准、保护和反馈 | 数据链、校准、故障与安全状态 |
| M5 | [低功耗设计](low-power.md) | 状态能量与唤醒边界 | 电流曲线和能量预算 |

启动、时钟和 UART 应最先完成，因为它们提供后续实验的观察基础。DMA 应在轮询和中断主路径正确后加入；否则更高吞吐会同时隐藏更多状态。

## 项目里程碑

### P0：可解释的最小固件

- 自己的构建命令产生 ELF、map 和 bin/hex。
- 调试器能在复位入口和 `main()` 停下。
- 启动日志包含版本、构建标识和复位原因。
- GPIO 输出能用寄存器和波形双重验证。

### P1：可恢复的传感器采集器

- 通过 I2C 或 SPI 读取器件 ID 和数据。
- 驱动区分 NACK、timeout、bus error、ID mismatch 等语义。
- 断开设备、错误地址和总线占用不会让主循环永久卡死。
- 波形、错误计数和恢复记录可以互相对应。

### P1+：执行器安全路径

如果项目包含电机、继电器、加热器或阀门，还必须定义：

- 上电、复位、通信丢失和传感器失效时的安全输出。
- 命令接收、软件写寄存器和物理动作完成分别由什么证据确认。
- 过流、过温、限位、反向电动势等保护由硬件还是软件承担。

## 必做实验

| 实验 | 改变变量 | 必须记录 |
|---|---|---|
| 启动链 | App 链接地址或向量表位置 | PC、SP、VTOR、map 和 fault |
| UART 时钟 | 主频、总线分频或波特率 | 寄存器和逻辑分析仪位宽 |
| I2C/SPI | 地址、CPOL/CPHA、上拉或频率 | 手册预期、波形、错误标志 |
| 中断时延 | 临界区长度和系统负载 | GPIO 探针的最大响应时间 |
| DMA 过载 | buffer 数量和消费速度 | 所有权、丢失、overrun 和恢复 |
| 低功耗 | 唤醒周期和外设状态 | 分状态电流、时间和平均功耗 |
| 配置/OTA | 写入或切换步骤中复位 | 恢复镜像、配置版本和启动决策 |

## 驱动验收契约

每个核心驱动至少写清：

```text
初始化前置：时钟、引脚、电源、复位和设备状态
接口语义：同步/异步、输入输出和返回值
时间边界：最长阻塞、超时来源和取消方式
所有权：buffer 何时属于 CPU、DMA、ISR 或调用者
错误模型：硬件错误、协议错误、资源错误和恢复动作
上下文：主循环、ISR 或任务中哪些 API 可调用
观测：寄存器、错误计数、日志和波形触发点
低功耗：睡眠前后谁保存、关闭和恢复状态
```

“返回 0 表示成功”不够。成功到底表示写入数据寄存器、总线事务完成、对端 ACK，还是物理动作完成，必须明确。

## 阶段门 G2-G3

- [ ] 能从上电追到第一条应用日志，并解释每一阶段的资源状态。
- [ ] 至少一个总线驱动具有成功波形和三种失败证据。
- [ ] ISR 有最坏执行时间或 GPIO 探针测量，不执行无界阻塞。
- [ ] DMA buffer 的分配者、所有者、完成事件、cache 和超时路径明确。
- [ ] 主循环中的每个等待都有边界，过载策略不是无限扩大 buffer。
- [ ] 复位原因、外设错误和关键计数可在下一次启动后诊断。
- [ ] 对低功耗或 OTA 至少完成一项定量/故障实验，而不只调用 API。

如果没有早期日志和调试器证据，先回补启动；波形与软件状态无法对应时回补时钟/外设；一加入 DMA 就不稳定时先退回中断路径并重画所有权。

## 向下一阶段迁移

进入 RTOS 后，保留本模块的驱动接口、超时、波形和错误计数。RTOS 只改变调用和等待方式，不会自动修复硬件时序与所有权。

进入联网或产品阶段前，继续追问：

- 网络任务阻塞时，本地采集和安全控制是否继续满足截止期？
- 远程配置会怎样改变定时器、DMA 和功耗预算？
- OTA 新固件能否理解旧配置，回滚旧固件能否理解新配置？
- 量产器件批次、温度和电源变化会让哪些实验结论失效？

真正掌握裸机开发，是能解释和控制硬件失败，而不只是让硬件工作一次。
