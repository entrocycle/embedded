# 裸机架构：主循环、状态机与时间驱动

裸机系统最初往往是一个 `while (1)`，随着传感器、通信、按键、存储和执行器增加，阻塞延时、全局状态和隐式调用顺序会迅速失控。本章连接“会写外设驱动”和“需要 RTOS”之间的空白：先用非阻塞状态机、合作式调度和明确所有权管理复杂度，再用证据判断是否真的需要抢占式 RTOS。

## 本章契约

### 前置知识

- 已完成 MCU 启动、时钟、中断和至少一个外设驱动。
- 理解环形缓冲、超时、无符号计数回绕和 ISR 最小职责。

### 学完后应能

- 把阻塞流程改写为运行至完成的状态机。
- 使用单调时间和 wrap-safe 比较设计周期与 deadline。
- 设计合作式调度器，量化每个任务的最坏运行时间与调用间隔。
- 用事件队列连接 ISR 与主循环，并定义过载行为。
- 区分状态、事件、命令和资源所有权，避免全局变量耦合。
- 根据截止期、阻塞和复杂度证据决定继续裸机还是引入 RTOS。

## 从阻塞代码看隐含假设

```c
sensor_start();
delay_ms(100);
value = sensor_read();
send_uart(value);
```

这段代码隐含：100 ms 内没有其他事情必须响应；传感器一定在 100 ms 后完成；UART 不会长时间阻塞；失败后重复整个流程可接受。功能增加后，这些假设会相互冲突。

阻塞不是绝对错误。启动阶段等待晶振稳定、极简系统执行短而有界的 SPI 事务，都可能合理。关键是知道谁被阻塞、多久、是否影响截止期、超时后如何恢复。

## 运行至完成状态机

将等待拆为状态和事件：

```text
IDLE --period_due--> START_CONVERSION
START_CONVERSION --started--> WAIT_READY
WAIT_READY --ready--> READ_RESULT
WAIT_READY --deadline--> RECOVER
READ_RESULT --ok--> PUBLISH
READ_RESULT --error--> RECOVER
PUBLISH --done--> IDLE
RECOVER --reset_done--> IDLE
```

每次调用状态机只完成有界工作，然后返回主循环。状态机需要明确：

- 状态进入时执行什么，退出条件是什么。
- 事件来自轮询、ISR、DMA、计时器还是其他模块。
- 每个等待的 deadline 和超时恢复。
- 非法事件是忽略、计数、断言还是进入安全状态。
- 复位后哪些状态可以恢复，哪些事务必须放弃。

状态机的价值不是 `switch`，而是把隐式等待和失败变成可观察模型。

## 时间模型与回绕

周期任务常见错误是直接比较绝对时间：

```c
if (now >= deadline) { /* wrap 后可能错误 */ }
```

若时间基于固定宽度无符号计数，并保证比较跨度小于半个计数范围，可使用模运算意义下的差值：

```c
static bool time_reached(uint32_t now, uint32_t deadline)
{
    return (int32_t)(now - deadline) >= 0;
}
```

这不是对任意跨度都成立。契约必须说明 tick 宽度、频率、最大 timeout 和转换规则。还要区分：

- **相对延时**：从现在再等 10 ms，容易累计周期漂移。
- **绝对周期**：下次 deadline 基于上一个计划时刻，更适合稳定周期。
- **超时**：资源等待的失败边界。
- **截止期**：完成动作仍有业务意义的最晚时间。

合作式系统无法在一个函数未返回时调度其他工作，因此最长运行至完成时间会直接限制全系统响应。

## 合作式调度器

最小调度循环可以按到期时间调用短任务：

```c
for (;;) {
    uint32_t now = monotonic_ms();

    sensor_step(now);
    command_step(now);
    network_step(now);
    logger_step(now);
    watchdog_step(now);

    enter_idle_if_possible();
}
```

代码简单，但必须量化：

```text
主循环最坏一圈时间 = 各 step 最坏执行时间之和 + 中断干扰
事件最坏响应时间 <= 当前 step 剩余时间 + 后续 step + 中断干扰
```

若某个 `network_step` 内部 DNS、串口 AT 命令或 Flash 擦除可阻塞数秒，整个模型失效。需要把操作拆成异步阶段、交给专用硬件/上下文，或引入 RTOS 隔离阻塞。

## ISR 与事件队列

ISR 负责确认硬件事件、采集最小状态并通知主循环。事件结构应尽量小且语义稳定：

```c
typedef struct {
    uint16_t type;
    uint16_t source;
    uint32_t timestamp;
    uint32_t data;
} app_event_t;
```

设计事件队列前回答：

- 队列使用复制还是指针，payload 生命周期由谁管理？
- 同类事件能否合并，例如“状态已变化”通常可合并，脉冲计数可能不可丢。
- 满时丢最新、丢最旧、置位汇总、进入降级还是触发保护？
- ISR 与主循环对索引的原子性和可见性由什么保证？
- 时间戳在哪一层采集，回绕如何处理？

仓库的 `labs/ring-buffer` 只证明顺序逻辑；ISR 并发和目标 CPU 内存模型必须在实板验证。

## 状态、事件、命令不要混用

| 类型 | 含义 | 合适表示 | 满/重复处理 |
|---|---|---|---|
| 状态 | 当前事实 | 最新值或 bit | 旧值通常可覆盖 |
| 事件 | 已发生的事实 | 带 ID/时间记录 | 依据重要性保留或计数 |
| 命令 | 请求产生副作用 | request + result | 过期、去重、授权 |
| 令牌 | 资源或完成通知 | 计数/信号 | 不能凭空复制 |

把按键状态和按键边沿都放进一个布尔变量，会丢失快速事件；把“UART 可发送”当作永久状态，可能错误重复使用完成事件。数据类型不是 C 类型，而是业务时间语义。

## 资源所有者

裸机模块最容易通过全局变量彼此调用。更稳妥的规则是每项资源有单一所有者：

| 资源 | 所有者例子 | 其他模块如何使用 |
|---|---|---|
| I2C 控制器 | bus manager | 提交有超时事务 |
| UART TX | logger/transport | 入队消息并处理满 |
| 配置 Flash | config service | 请求更新，不直接写地址 |
| 执行器 | control state machine | 提交带范围与期限命令 |
| 网络模组 | connection state machine | 发布业务事件，不直接发 AT |

单一所有者减少并发写，但可能成为瓶颈。需要测量服务时间、队列深度和优先级，而不是把所有资源都集中到一个“大管理器”。

## 看门狗与进展证明

主循环每圈喂狗只能证明 CPU 还在循环，不能证明传感器、控制和通信取得有意义进展。

为关键状态机定义进展条件：

- 控制任务在截止期内完成一次周期。
- 传感器要么成功采样，要么产生可接受的错误和恢复动作。
- 事件队列没有持续满，最老事件年龄未超限。
- 网络可以离线，但连接状态机仍按退避推进且不阻塞本地控制。

看门狗复位前保存哪个状态机、哪个 deadline、队列水位和外设错误，有助于区分软件卡死与外部设备长期不可用。

## 何时引入 RTOS

| 现象 | 裸机可继续改进 | RTOS 可能更合适 |
|---|---|---|
| 少量短周期任务 | 合作式调度足够清晰 | 不需要仅为“并发”引入 |
| 多个独立阻塞库 | 可逐个异步化但成本高 | 任务可隔离阻塞上下文 |
| 严格优先级需求 | 中断 + 状态机可处理少数路径 | 多周期/截止期需要抢占与分析 |
| 多团队组件 | 所有权和回调可控 | 任务/队列边界更易集成，但锁风险增加 |
| 调试与内存极受限 | 裸机状态空间更小 | RTOS 元数据和栈可能不可接受 |

引入 RTOS 的决策记录应包含：当前主循环最大延迟、阻塞来源、任务/栈预算、调试能力和团队经验。RTOS 会解决什么，也会引入什么新失败模式。

## 必做实验

1. 把一个带 `delay_ms()` 的传感器流程改成状态机，比较主循环最大一圈时间。
2. 使用 8 位或模拟短周期 tick 强制回绕，验证 timeout 比较。
3. 对每个 `step()` 用 GPIO 拉高/拉低，测量最坏执行时间和响应延迟。
4. 暂停事件消费者制造队列满，验证数据语义和 overflow 指标。
5. 让一个外设永不完成，确认状态机超时、恢复且其他模块继续推进。

## 阶段验收

- [ ] 主循环没有未记录的无界等待。
- [ ] 每个状态机有合法转移、deadline、错误和恢复路径。
- [ ] 时间比较在计数回绕约束下正确，最大 timeout 有文档。
- [ ] ISR 与主循环之间的数据所有权和队列满策略明确。
- [ ] 最长 step 和主循环最坏响应由测量支持。
- [ ] 看门狗检查业务进展而不只是循环活性。
- [ ] 是否引入 RTOS 的结论有当前系统证据。

## 深入思考

1. 状态机减少阻塞后，为什么状态数量、非法转移和取消路径会成为新复杂度？
2. 合作式调度避免抢占竞态，却为什么仍然存在 ISR 并发和长期延迟风险？
3. 将所有外设操作集中到服务模块会改善所有权，但什么时候会造成优先级和吞吐瓶颈？
4. 如果日志 step 在串口拥塞时变慢，丢日志、阻塞或压缩分别会影响哪些诊断能力？

本章的出口是让裸机复杂度可测、可解释，并让“是否需要 RTOS”成为工程结论而不是习惯选择。
