# 应用层协议设计：边界、版本、幂等与演进

TCP、UART、CAN、MQTT 和 BLE 提供不同的传输能力，但产品仍要定义自己的应用语义：消息在哪里结束，字段如何编码，错误怎样报告，旧版本如何共存，重复命令是否产生副作用，资源耗尽时如何失败。

协议设计不是先画一张字段表。它是一份跨设备、云端、工具、测试和未来版本长期执行的契约，一旦部署就很难随意修改。

## 本章契约

### 前置知识

- 理解字节流与消息边界、字节序、整数溢出、CRC/MAC 的区别。
- 理解 timeout、重试、状态机和基本安全边界。
- 已运行 `labs/byte-stream-parser` 或等价分片实验。

### 学完后应能

- 根据底层传输选择 framing、同步和有界解析策略。
- 为状态、事件、命令和文件定义不同消息语义。
- 设计显式版本、能力协商和向前/向后兼容规则。
- 区分传输重试、业务 request ID、幂等和持久化去重。
- 设计结构化错误、deadline、流控和资源配额。
- 用属性测试、分片测试、畸形输入和跨版本矩阵验证协议。

## 从业务语义开始

先为每类操作写清：

| 问题 | 示例 |
|---|---|
| 操作是什么 | 设置采样周期、启动泵、读取状态 |
| 成功定义 | 参数已持久化、任务已接受、物理反馈确认 |
| 可否重复 | 设置绝对值可幂等，执行一次脉冲通常不可幂等 |
| 有效期限 | 命令 10 秒后失效，配置长期有效 |
| 授权主体 | 用户、运维、产测或云服务 |
| 失败结果 | 拒绝、稍后重试、状态未知或部分完成 |
| 查询方式 | 用 request ID 查询最终结果 |

只有定义了这些，才能决定消息字段和重试行为。

## Framing：怎样找到消息边界

| 方案 | 优点 | 主要风险 | 适用 |
|---|---|---|---|
| 固定长度 | 简单、可预测 | 浪费空间、演进困难 | 固定控制帧 |
| 长度前缀 | 二进制高效、易分片 | 头损坏后难同步、必须限长 | TCP、可靠内部链路 |
| 分隔符+转义 | 人类可读或易同步 | 转义与最坏膨胀 | UART/文本协议 |
| COBS/SLIP | 明确帧界和有界膨胀 | 需编码解码 | 串口、数据报封装 |
| TLV | 可跳过未知字段 | 解析复杂、长度嵌套 | 需要演进的二进制消息 |
| CBOR/Protobuf | schema 与工具生态 | 代码/RAM、兼容规则 | 复杂设备与云协议 |

长度字段必须在任何分配、复制和加法之前检查：

- 最小/最大长度。
- `header + payload + trailer` 的整数溢出。
- 嵌套字段总长度和递归深度。
- 单连接未完成帧占用时间和内存。
- EOF、timeout 和取消时半帧如何释放。

在无可靠边界的串口上，还要定义头损坏后的重新同步；在 TCP 上虽然传输不丢字节，应用、内存损坏或错误端仍可能发送畸形长度，最安全行为常是关闭会话而非无限扫描。

## 编码：不要序列化内存布局

协议字段应明确：

- 字节序、宽度、符号和单位。
- 缩放与范围，例如温度使用 milli-Celsius。
- 字符串编码、最大长度和是否允许 NUL。
- 可选字段、默认值和未知字段处理。
- 浮点 NaN/Inf、精度和跨平台表示策略。

不要直接发送 C/C++ 结构体：padding、对齐、endianness、ABI 和未初始化填充都会造成兼容与信息泄漏风险。使用显式编码/解码，并让 decoder 在写输出对象前完成验证，或保证失败时对象状态明确。

## 消息头设计

一个通用思考模板：

```text
magic / sync       可选，帮助识别和重同步
protocol_version   协议或 envelope 版本
message_type       消息语义
flags              只定义已知 bit 的处理规则
payload_length     有界长度
request_id         业务关联与去重
sequence           流顺序或诊断，不自动等于 request ID
deadline/time      过期语义，依赖可信时间模型
payload
integrity/auth     CRC 检错或 MAC/AEAD 认证
```

不是字段越多越好。每个字段都需要说明由谁生成、何时回绕、重启后是否持久、攻击者能否修改、旧版本如何处理。

## 版本与兼容

版本兼容至少分四种：

| 变化 | 兼容风险 | 推荐策略 |
|---|---|---|
| 新增可选字段 | 旧端是否忽略未知字段 | 明确 skip 规则和默认语义 |
| 修改字段含义 | 同一字节被两种方式解释 | 使用新字段/消息类型，不静默改语义 |
| 删除字段 | 旧端仍要求 | 保持兼容窗口或能力协商 |
| 改变事务流程 | 状态机不一致 | 协商能力或独立版本流程 |

单个 `version=2` 不自动解决兼容。还需决定：

- 握手失败是降级、拒绝还是只启用共同能力。
- 新设备连旧云、旧设备连新云分别如何测试。
- OTA 回滚后协议和持久化队列是否仍可解释。
- 灰度期间两个版本如何共存和诊断。

能力协商比简单按版本比较灵活，但能力组合会扩大测试矩阵。只协商真正可独立启用的能力。

## request ID、sequence 与幂等

三者职责不同：

- **sequence**：表示流中顺序、检测缺口或选择新旧记录，通常会回绕。
- **request ID**：关联请求、响应和重试，应在重试期间保持不变。
- **幂等键/去重记录**：确保同一业务动作只产生一次副作用，需要定义保存范围和寿命。

假设设备执行开阀后断电，ACK 未发送。云端重试同一 request ID：

1. 设备必须在动作前还是动作后持久化去重状态？
2. 持久化动作结果与执行器物理状态如何保持一致？
3. 若无法原子地同时保证，系统是否应返回“结果未知”并读取反馈？
4. 去重表满、过期或设备恢复出厂后会怎样？

“QoS 1 + request ID”仍不能自动实现物理动作 exactly once。需要业务状态机和反馈设计。

## 错误模型

错误响应至少区分：

| 类别 | 例子 | 调用方动作 |
|---|---|---|
| 格式错误 | 长度、字段、版本非法 | 不重试同一内容，记录安全事件 |
| 未授权 | 身份或权限不足 | 刷新凭据或停止，不盲目重试 |
| 冲突 | 当前状态不允许 | 查询状态、重新决策 |
| 临时不可用 | 资源忙、网络依赖失败 | 按建议退避重试 |
| 永久不支持 | 能力/硬件不支持 | 降级或更改产品逻辑 |
| 结果未知 | 请求可能已经执行 | 使用 request ID 查询，不直接重复副作用 |

错误码需要稳定机器语义；人类文本可用于诊断但不应驱动程序逻辑。错误中不要泄漏密钥、内部地址和敏感状态。

## deadline、timeout 与时钟

- timeout 是一方愿意等待多久。
- deadline 是动作最晚何时仍有价值。
- timestamp 表示某个时钟域下的时间事实。

设备时钟可能未同步、回拨或被攻击。对实时命令可使用服务器签发时间 + 有效窗口 + nonce，但设备必须有可信时间来源或单调会话时钟。不能在时间不可信时假装过期判断可靠。

命令进入设备后仍应检查本地 deadline，避免排队和重连后执行过时动作。

## 流控与资源耗尽

协议必须在过载时有边界：

- 单帧最大长度。
- 单连接 inflight 请求数。
- 解码 CPU/内存预算。
- 每类消息速率和优先级。
- 未完成事务超时。
- 输出队列满和慢客户端处理。

TCP backpressure 只能让发送方最终变慢，不能定义业务应丢状态、保告警还是拒绝命令。应用层仍需容量和数据语义。

## 完整性、认证与保密

- CRC 检测随机传输/存储损坏，不抵抗主动篡改。
- MAC/AEAD 验证密钥持有方并保护完整性，AEAD 还提供保密。
- TLS 保护连接通道，但 broker、网关或应用层转发后是否仍需端到端签名取决于信任边界。
- 重放保护需要 nonce、sequence、窗口或持久状态，认证本身不自动防重放。

安全字段设计必须结合密钥生命周期和失败恢复；不要自创密码算法。

## 验证策略

### 解析属性

- 任意分片和合并不改变合法消息结果。
- 编码后解码得到同一语义值。
- 任意输入都不会越界、无限循环或无界分配。
- decoder 失败后状态符合契约。

### 兼容矩阵

```text
old device <-> old service
old device <-> new service
new device <-> old service
new device <-> new service
upgrade/rollback with queued old messages
```

### 故障注入

- 截断每一个字节位置。
- 修改每个长度、类型和 flags 边界。
- 重复、乱序、延迟和过期。
- request 执行后、响应前断电。
- 去重表满和 sequence 回绕。
- 慢客户端与输入洪泛。

模糊测试适合发现解析器安全问题，但业务状态机仍需模型和序列测试。

## 协议评审清单

- [ ] 每类消息的业务成功、重复和过期语义明确。
- [ ] framing 能处理分片、合并、畸形和重同步/关闭。
- [ ] 所有长度、计数和嵌套有上限与溢出检查。
- [ ] 编码独立于内存布局，单位和范围明确。
- [ ] 版本变化和未知字段有双向兼容策略。
- [ ] request ID、sequence 和去重职责分开。
- [ ] 错误区分可重试、不可重试、冲突和结果未知。
- [ ] deadline 建立在明确时钟信任模型上。
- [ ] 流控、配额和慢消费者策略明确。
- [ ] CRC、认证、加密和重放保护边界没有混淆。
- [ ] 跨版本、断电和恶意输入测试进入回归。

## 综合练习

为“远程执行一次校准”设计协议：校准可能持续 30 秒，设备中途断电，云端可能重试，结果需要审计，旧固件不支持新校准参数。

提交：

- 消息与状态机。
- request ID、deadline 和去重生命周期。
- 进度、完成、失败和结果未知语义。
- 版本/能力协商。
- 断电点和恢复结果。
- 认证、授权和审计字段。
- 测试矩阵及评价标准。

## 深入思考

1. 协议越“灵活”为什么可能越难验证、越难安全解析？
2. 忽略未知字段有利于兼容，什么时候会让安全关键约束被旧端忽略？
3. 业务 exactly once 为什么常常需要把协议、持久化和物理反馈放在同一模型中？
4. 若设备无法获得可信墙上时间，如何设计命令过期和重放窗口？

本章的出口不是设计一张漂亮字段表，而是让跨版本、跨网络和跨复位的业务行为仍然可解释。
