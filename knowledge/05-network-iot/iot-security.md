# IoT 安全与设备生命周期

IoT 安全的目标不是让设备“绝对安全”，而是建立合理的信任边界，降低批量攻击、固件篡改、设备冒充、密钥泄露和恶意降级的风险。

## 本章任务与边界

完成本章后，你应能识别资产、主体、入口和信任边界，设计每设备身份、密钥存储、TLS 校验、命令授权、安全启动、签名 OTA、调试口与退役生命周期，并说明每项控制的残余风险。

本章不提供自创密码方案，也不等价于安全认证。算法、参数、证书、随机数、安全存储和漏洞响应需采用成熟库、芯片安全能力与当前官方指南，并接受独立安全评审。TLS 只保护通道，不能替代本地授权与安全状态。

完成证据：项目威胁模型、凭据生命周期、重放/证书过期/时间错误/签名失败实验，以及密钥泄漏或设备退役后的响应推演。安全结论应记录攻击者能力和未覆盖风险。

## 威胁模型

先明确攻击面：

| 攻击面 | 风险 |
|---|---|
| 固件包 | 被篡改、植入后门、降级到漏洞版本 |
| 通信链路 | 中间人攻击、窃听、重放 |
| 设备身份 | 克隆设备、伪造上报 |
| 本地调试口 | 读取 Flash、修改固件 |
| 云端接口 | 未授权命令、批量控制 |
| 配网过程 | Wi-Fi 密码泄露、绑定错误 |
| 外部存储 | 配置和密钥明文暴露 |

## 设备身份

每台设备应有唯一身份：

- SN 或 device_id。
- 设备证书或唯一密钥。
- 硬件唯一 ID 作为辅助，不应单独作为安全凭据。

常见方案：

| 方案 | 特点 |
|---|---|
| 预烧录对称密钥 | 成本低，云端校验简单，密钥泄露影响面需控制 |
| X.509 设备证书 | 标准化，适合 TLS 双向认证 |
| 安全芯片 | 私钥不可导出，安全性高，成本增加 |
| 首次激活下发凭据 | 灵活，但激活链路必须安全 |

## 密钥存储

不要把所有设备共用密钥硬编码在固件中。

优先级：

1. 安全芯片/TrustZone/OTP。
2. MCU 读保护 + 加密存储。
3. Flash 分区加密和访问控制。
4. 最低限度：每设备不同密钥。

密钥管理要考虑：

- 生成。
- 烧录。
- 绑定。
- 轮换。
- 吊销。
- 泄露后的影响范围。

## TLS

设备使用 TLS 时要做：

- 校验证书链。
- 校验服务器域名。
- 使用正确系统时间或证书有效期策略。
- 禁用过时协议和弱加密套件。
- 妥善处理证书更新。

常见错误：

- 为了调试关闭证书校验后忘记恢复。
- 只加密不认证，仍可能被中间人攻击。
- 所有设备使用同一 client key。

## 命令安全

云端命令下发要保证：

- 设备只接受授权主题或连接中的命令。
- 命令带 request_id，响应可追踪。
- 重要命令需要幂等。
- 危险命令需要额外确认或权限。
- 防重放：timestamp、nonce、递增序号。

命令响应示例：

```json
{
  "request_id": "cmd-001",
  "code": 0,
  "message": "ok"
}
```

## Secure Boot

Secure Boot 确保设备只运行被授权的固件。

基本链路：

```text
ROM/Bootloader 信任根
  ↓ 验证
Bootloader
  ↓ 验证签名
Application
```

关键：

- 公钥或 hash 存在不可轻易修改的位置。
- 私钥只在构建/签名环境中，不进入设备。
- 验签失败不启动应用。
- 防版本回退。

## OTA 安全

OTA 最低要求：

- 固件完整性校验。
- 固件签名验证。
- 版本号防回退。
- 下载通道加密。
- 断电恢复。
- 失败回滚。

签名校验应在 Bootloader 或可信执行路径中完成。App 自己说“我验证过了”不够可靠。

## 调试口策略

量产设备要定义调试口策略：

| 策略 | 适用 |
|---|---|
| 完全关闭 | 高安全产品 |
| 授权开启 | 售后维修、研发诊断 |
| 保留串口日志但限制命令 | 普通 IoT 设备 |
| 开发版本开放，量产版本关闭 | 常见做法 |

关闭调试口前要确认：

- OTA 可恢复。
- 产测流程完成。
- 售后诊断有替代路径。
- 读保护等级不可逆风险已评估。

## 设备生命周期

```text
生产
  ↓ 写入 SN/密钥/校准
出厂测试
  ↓
激活绑定
  ↓
运行
  ↓
升级/配置变更
  ↓
售后诊断
  ↓
解绑/转移/报废
```

每个阶段要记录：

- 设备身份。
- 固件版本。
- 硬件版本。
- 激活状态。
- 密钥状态。
- 最近升级结果。
- 故障码。

## 工程洞察

IoT 安全不是上线前加 TLS，而是贯穿设备身份、制造、激活、运行、升级、报废的生命周期设计。攻击者不一定只来自网络，也可能来自物理接触、调试口、供应链、固件提取、云端凭据泄露或二手设备转移。

安全设计要先做资产和信任边界分析：

- 资产：设备私钥、证书、用户数据、控制命令、固件镜像、生产测试权限和调试接口。
- 入口：网络服务、BLE 配网、串口、JTAG/SWD、OTA 包、USB、文件系统和本地按键。
- 信任边界：芯片安全区、Bootloader、应用、云端、手机 App、产测工具和售后工具。
- 生命周期：出厂注入、首次绑定、证书轮换、OTA、恢复出厂、转移所有权和报废擦除。

安全的难点在于它会和调试、量产、售后、成本和可用性冲突。例如关闭调试口提高安全性，却增加现场定位成本；证书过期策略严格，可能让长期离线设备无法恢复；安全启动保护固件完整性，但也要求 OTA 回滚和密钥轮换设计更严谨。

开放问题：

1. 如果设备私钥在产测阶段泄露，系统如何发现、吊销和恢复？
2. 恢复出厂设置应该删除哪些数据，保留哪些出厂身份？为什么？
3. 远程命令使用 TLS 传输后，为什么仍需要权限、nonce、时间窗和业务幂等？
