# IoT 安全与设备生命周期

IoT 安全的目标不是让设备“绝对安全”，而是建立合理的信任边界，降低批量攻击、固件篡改、设备冒充、密钥泄露和恶意降级的风险。

## 威胁模型

先明确攻击面：

| 攻击面 | 风险 |
|---|---|
| 固件包 | 被篡改、植入后门、降级到漏洞版本 |
| 通信链路 | 中间人攻击、窃听、重放 |
| 设备身份 | 克隆设备、伪造上报 |
| 本地调试口 | 读取 Flash、修改固件 |
| 云端接口 | 未授权命令、批量控制 |
| 配网过程 | Wi-Fi 密码泄露、绑定错误 |
| 外部存储 | 配置和密钥明文暴露 |

## 设备身份

每台设备应有唯一身份：

- SN 或 device_id。
- 设备证书或唯一密钥。
- 硬件唯一 ID 作为辅助，不应单独作为安全凭据。

常见方案：

| 方案 | 特点 |
|---|---|
| 预烧录对称密钥 | 成本低，云端校验简单，密钥泄露影响面需控制 |
| X.509 设备证书 | 标准化，适合 TLS 双向认证 |
| 安全芯片 | 私钥不可导出，安全性高，成本增加 |
| 首次激活下发凭据 | 灵活，但激活链路必须安全 |

## 密钥存储

不要把所有设备共用密钥硬编码在固件中。

优先级：

1. 安全芯片/TrustZone/OTP。
2. MCU 读保护 + 加密存储。
3. Flash 分区加密和访问控制。
4. 最低限度：每设备不同密钥。

密钥管理要考虑：

- 生成。
- 烧录。
- 绑定。
- 轮换。
- 吊销。
- 泄露后的影响范围。

## TLS

设备使用 TLS 时要做：

- 校验证书链。
- 校验服务器域名。
- 使用正确系统时间或证书有效期策略。
- 禁用过时协议和弱加密套件。
- 妥善处理证书更新。

常见错误：

- 为了调试关闭证书校验后忘记恢复。
- 只加密不认证，仍可能被中间人攻击。
- 所有设备使用同一 client key。

## 命令安全

云端命令下发要保证：

- 设备只接受授权主题或连接中的命令。
- 命令带 request_id，响应可追踪。
- 重要命令需要幂等。
- 危险命令需要额外确认或权限。
- 防重放：timestamp、nonce、递增序号。

命令响应示例：

```json
{
  "request_id": "cmd-001",
  "code": 0,
  "message": "ok"
}
```

## Secure Boot

Secure Boot 确保设备只运行被授权的固件。

基本链路：

```text
ROM/Bootloader 信任根
  ↓ 验证
Bootloader
  ↓ 验证签名
Application
```

关键：

- 公钥或 hash 存在不可轻易修改的位置。
- 私钥只在构建/签名环境中，不进入设备。
- 验签失败不启动应用。
- 防版本回退。

## OTA 安全

OTA 最低要求：

- 固件完整性校验。
- 固件签名验证。
- 版本号防回退。
- 下载通道加密。
- 断电恢复。
- 失败回滚。

签名校验应在 Bootloader 或可信执行路径中完成。App 自己说“我验证过了”不够可靠。

## 调试口策略

量产设备要定义调试口策略：

| 策略 | 适用 |
|---|---|
| 完全关闭 | 高安全产品 |
| 授权开启 | 售后维修、研发诊断 |
| 保留串口日志但限制命令 | 普通 IoT 设备 |
| 开发版本开放，量产版本关闭 | 常见做法 |

关闭调试口前要确认：

- OTA 可恢复。
- 产测流程完成。
- 售后诊断有替代路径。
- 读保护等级不可逆风险已评估。

## 设备生命周期

```text
生产
  ↓ 写入 SN/密钥/校准
出厂测试
  ↓
激活绑定
  ↓
运行
  ↓
升级/配置变更
  ↓
售后诊断
  ↓
解绑/转移/报废
```

每个阶段要记录：

- 设备身份。
- 固件版本。
- 硬件版本。
- 激活状态。
- 密钥状态。
- 最近升级结果。
- 故障码。

## 安全检查清单

- [ ] 每台设备是否有唯一身份？
- [ ] 密钥是否每设备唯一？
- [ ] TLS 是否校验证书和域名？
- [ ] OTA 包是否签名？
- [ ] 是否防止固件降级？
- [ ] Debug 口量产策略是否明确？
- [ ] 云端命令是否可追踪和幂等？
- [ ] 配网过程是否保护用户 Wi-Fi 密码？
- [ ] 设备解绑/报废是否清除敏感信息？
- [ ] 构建签名私钥是否隔离保存？
