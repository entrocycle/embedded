# 连接状态机与弱网设计

联网设备不能只验证“实验室里连上一次”。真实现场会遇到弱网、DNS 失败、证书过期、云端重启、路由器断电、蜂窝切换、设备掉电和 Flash 写满。连接状态机的目标，是让设备在这些情况下可恢复、可观测、可控制。

## 本章任务与边界

完成本章后，你应能将链路、地址、时间、TLS、协议会话和业务就绪拆成分层状态，设计带抖动的指数退避、离线数据策略和命令恢复，并用状态年龄、失败分类和缓存水位诊断弱网。

本章不提供适合所有运营商和协议栈的固定 timeout。重试参数必须依据网络尾部时延、服务限流、设备功耗与规模测试；状态机能让失败收敛，但不会自动解决错误的数据语义和身份安全。

完成证据：状态/事件/守卫表、DNS/TLS/broker/断网四类注入、重连能量或频率预算，以及恢复后缓存和命令语义验证。

## 分层状态

把联网拆成几层，不要用一个 `connected` 布尔值表示全部：

```text
LINK_DOWN
  -> LINK_UP
  -> IP_READY
  -> TIME_SYNCED
  -> TLS_READY
  -> CLOUD_CONNECTED
  -> ONLINE
```

每一层都有自己的失败原因和恢复动作。

## 典型状态机

```text
BOOT
  -> LOAD_CONFIG
  -> NET_IF_UP
  -> GET_IP
  -> SYNC_TIME
  -> CONNECT_BROKER
  -> AUTHENTICATE
  -> ONLINE
  -> DEGRADED
  -> BACKOFF
  -> RECOVER
```

关键设计：

- 每个状态有进入动作、退出动作和超时。
- 每个失败有错误码。
- 重试使用退避，不要高频重连。
- 在线和离线都要有可用行为。
- 状态变化要写日志和诊断指标。

## 退避策略

简单退避：

```text
1s -> 2s -> 4s -> 8s -> 16s -> 30s -> 60s
```

建议加入随机抖动，避免大量设备同时重连：

```text
delay = min(base * 2^retry, max_delay) + random(0, jitter)
```

重试计数要按失败类型区分：

- DNS 失败。
- TCP 失败。
- TLS 失败。
- 鉴权失败。
- MQTT CONNACK 失败。
- 心跳超时。

鉴权失败通常不应该无限高频重试，而应进入降级或等待配置更新。

## 数据上报

上报要考虑：

- 周期数据。
- 事件数据。
- 告警数据。
- 设备状态。
- OTA 结果。

弱网下要定义缓存策略：

| 数据类型 | 离线缓存策略 |
|---|---|
| 最新状态 | 只保留最新 |
| 告警 | 必须保留，带上限 |
| 高频传感器 | 降采样或聚合 |
| 日志 | 环形缓存 |
| OTA 结果 | 优先上报直到确认 |

缓存必须有容量上限。不能为了“数据不丢”把 RAM 或 Flash 写爆。

## 命令下发

命令下发要关注幂等和确认：

- 命令 ID。
- 超时时间。
- 参数校验。
- 执行中、成功、失败状态。
- 重复命令处理。
- 设备离线时命令过期策略。

例如“打开继电器”最好带目标状态和命令 ID，而不是裸操作：

```json
{
  "cmd_id": "abc-123",
  "target": "relay1",
  "state": "on",
  "expires_at": 1710000000
}
```

## MQTT 设计要点

- Client ID 要稳定且唯一。
- Keepalive 不宜过短。
- QoS 不是越高越好，QoS 1 已能满足大多数上报确认。
- retained message 要谨慎使用。
- Last Will 可用于设备离线通知。
- topic 命名要版本化。

示例 topic：

```text
devices/{sn}/telemetry
devices/{sn}/event
devices/{sn}/command/down
devices/{sn}/command/up
devices/{sn}/ota/status
```

## 时间同步

TLS、证书和日志都依赖时间。

设备应处理：

- RTC 无效。
- NTP 失败。
- 蜂窝网络提供时间。
- 时间回跳。
- 证书尚未生效或已过期。

没有可靠时间前，不要直接判定证书永久失败；要把“时间未同步”作为独立状态。

## 可观测指标

建议维护：

- 当前连接状态。
- 最近一次状态变化时间。
- 失败原因计数。
- 当前退避时间。
- MQTT 重连次数。
- 上报队列深度。
- 离线缓存大小。
- TLS 证书错误码。

这些指标要能通过串口命令、本地 API 或远程诊断读取。

## 弱网测试矩阵

- [ ] 启动时无网络。
- [ ] DNS 不通。
- [ ] TCP 可连但 TLS 失败。
- [ ] MQTT 连接后云端主动断开。
- [ ] 心跳包丢失。
- [ ] 网络频繁抖动。
- [ ] 离线缓存达到上限。
- [ ] 设备时间错误。
- [ ] 证书过期。
- [ ] 路由器重启后自动恢复。

## 工程洞察

连接状态机的目标不是“尽快重连”，而是在弱网和故障条件下保护设备资源、业务语义和可观测性。没有状态机的重连代码通常会退化成散落的 flag、延时和重试，最后在证书错误、网络不可达、服务器拒绝和本地配置损坏时做出同样动作。

设计连接状态机时要区分：

- 可恢复故障：临时无网络、DNS 失败、broker 重启、信号差。
- 需人工或云端干预故障：鉴权失败、证书过期、设备被禁用、配置错误。
- 本地降级状态：离线缓存、本地控制、只读模式、低功耗等待。
- 终止或保护状态：连续失败过多、数据区损坏、命令风险不可接受。

退避策略不是单纯省流量，它还保护电池、Flash、日志和云端服务。离线缓存也不是越大越好，它必须尊重数据语义：状态值可以覆盖，告警可能必须保留，命令回执需要和 request ID 绑定。弱网设计的成熟度，体现在失败时系统是否还能被理解和控制。
