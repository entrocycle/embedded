# 综合项目：环境监测节点

本项目用于把 MCU、RTOS、传感器、网络、日志、低功耗和 OTA 思路串起来。你可以用 STM32 + Wi-Fi 模组，也可以用 ESP32 直接实现。

## 项目目标

实现一个环境监测节点：

- 周期采集温湿度、光照或气压。
- 本地串口输出日志。
- 使用 MQTT 上报云端或本地 broker。
- 支持远程修改上报周期。
- 支持本地配置保存。
- 具备看门狗和异常恢复。
- 预留 OTA 状态机。

## 需求假设与指标

先给项目设定约束，否则无法判断架构是否合理。

| 指标 | 建议目标 | 影响 |
|---|---:|---|
| 默认采样周期 | 1-10s | 决定传感器任务周期和数据量 |
| 默认上报周期 | 10-60s | 决定网络功耗和缓存深度 |
| 断网缓存 | 至少 10 分钟 | 决定 RAM/Flash 缓存策略 |
| 单次命令响应 | 2s 内 | 决定 MQTT 任务和命令队列优先级 |
| 看门狗窗口 | 2-5s | 决定任务心跳和最大阻塞时间 |
| 配置写入寿命 | 大于预期修改次数 10 倍 | 决定 Flash wear leveling |

这些数值不是固定答案。你应该根据产品场景调整，并在复盘里解释为什么这样定。

## 硬件建议

| 模块 | 示例 |
|---|---|
| 主控 | STM32F4/F1/L4、ESP32 |
| 传感器 | SHT30/BME280/BH1750 |
| 通信 | ESP8266 AT、ESP32 Wi-Fi、以太网 |
| 调试 | ST-Link/J-Link、USB 转串口 |
| 可选 | OLED、外部 Flash、按键、LED |

## 系统架构

```text
┌─────────────────────────────────────┐
│ app                                 │
│  app_state / command / telemetry    │
├─────────────────────────────────────┤
│ services                            │
│  logger / config / mqtt / watchdog  │
├─────────────────────────────────────┤
│ drivers                             │
│  sht30 / wifi_modem / led / button  │
├─────────────────────────────────────┤
│ bsp                                 │
│  clock / uart / i2c / gpio / flash  │
└─────────────────────────────────────┘
```

依赖规则：

- `app` 不直接访问 `GPIOA`、`I2C1`、`USART2` 这类硬件名。
- `drivers` 不直接发布 MQTT，也不直接写配置。
- `services` 负责日志、配置、MQTT、看门狗等横切能力。
- `bsp` 只表达板级资源和平台适配，避免包含业务状态。

## 架构取舍

| 决策 | 可选方案 | 推荐选择 | 原因和代价 |
|---|---|---|---|
| 采集与上报 | 同一任务 / 分任务 | 分任务 + 队列 | 解耦弱网阻塞，但增加队列满处理 |
| 传感器驱动 | 直接 HAL / driver + port | driver + port | 便于移植和测试，但接口设计更多 |
| 日志 | 同步串口 / 异步队列 | 运行期异步，早期同步 | 降低阻塞，但要处理日志丢弃 |
| 配置存储 | 单副本 / 双副本 | 带版本和 CRC 的双副本 | 抗断电，但占用更多 Flash |
| 看门狗 | 任意位置喂狗 / 健康汇总 | watchdog_task 汇总心跳 | 能发现任务卡死，但需要设计心跳语义 |

## RTOS 任务设计

| 任务 | 优先级 | 触发 | 职责 |
|---|---:|---|---|
| sensor_task | 3 | 周期 | 采集传感器并发队列 |
| mqtt_task | 2 | 队列/网络事件 | 连接、订阅、发布 |
| command_task | 2 | MQTT 下行 | 解析命令并更新配置 |
| logger_task | 1 | 日志队列 | 异步输出日志 |
| watchdog_task | 4 | 周期 | 检查任务心跳并喂狗 |

任务设计要补充阻塞点和证据：

| 任务 | 主要阻塞点 | 必须记录的指标 |
|---|---|---|
| sensor_task | I2C/SPI 超时、传感器转换时间 | 采样耗时、失败计数、队列满次数 |
| mqtt_task | 连接、发布、订阅、DNS/TLS | 状态、重连次数、发布耗时、缓存深度 |
| command_task | 配置写 Flash、命令 ack | 非法命令次数、配置版本、写入结果 |
| logger_task | 串口输出、日志队列 | 丢日志次数、队列高水位 |
| watchdog_task | 不应长时间阻塞 | 每个任务心跳年龄、复位原因 |

## MQTT 主题

```text
env/{device_id}/telemetry
env/{device_id}/event
env/{device_id}/command/down
env/{device_id}/command/up
env/{device_id}/ota/down
env/{device_id}/ota/up
```

遥测 payload：

```json
{
  "seq": 1024,
  "ts": 12345678,
  "temperature": 24.6,
  "humidity": 52.1,
  "light": 380.0
}
```

命令 payload：

```json
{
  "cmd": "set_report_interval",
  "request_id": "abc-001",
  "value_ms": 5000
}
```

命令处理要求：

- 每条命令必须有 `request_id`，设备回复 ack，避免云端无法判断是否生效。
- 设置类命令要做范围校验，例如上报周期不能小于传感器和网络能承受的下限。
- 重复命令要幂等，同一个 `request_id` 重发不应导致配置反复写 Flash。
- 下行命令失败时要返回错误码，而不是只返回失败字符串。

## 实现阶段

### 阶段 1：最小板级 bring-up

目标：

- 串口日志可用。
- LED 可控。
- 系统 tick 可用。

验收：

- 上电输出版本号、编译时间、复位原因。
- LED 以 1Hz 闪烁。

思考：

- 如果没有串口输出，下一步先查电源、时钟、下载、复位还是 UART 引脚？证据是什么？
- 复位原因如果不准确，会如何影响后续现场诊断？

### 阶段 2：传感器驱动

目标：

- I2C/SPI 传感器可读取。
- 驱动有错误码和超时。
- 用逻辑分析仪确认总线时序。

验收：

- 周期打印温湿度。
- 断开传感器后能输出明确错误码，不死机。

思考：

- 驱动返回 `timeout`、`nack`、`id_mismatch` 的恢复策略是否应该相同？
- 传感器数据异常时，是驱动负责滤波，还是上层业务负责判断？

### 阶段 3：RTOS 化

目标：

- 采集、日志、通信分任务。
- 使用队列传递遥测数据。
- 看门狗任务检查心跳。

验收：

- 任一低优先级任务卡住，不影响高优先级健康监控发现异常。
- 任务栈 high water mark 有记录。

思考：

- `watchdog_task` 优先级设高会发现什么问题？又可能掩盖什么问题？
- 如果日志队列满了，是否允许阻塞传感器任务？

### 阶段 4：MQTT 上报

目标：

- 连接 broker。
- 上报遥测数据。
- 订阅命令主题。
- 断网自动重连。

验收：

- 本地 Mosquitto 能收到数据。
- 断开 Wi-Fi 后进入重连退避，恢复后继续上报。

思考：

- 断网期间遥测数据全部缓存、只缓存最新值、还是只缓存告警？为什么？
- MQTT QoS 1 是否一定比 QoS 0 更可靠？它会带来哪些重复和阻塞问题？

### 阶段 5：配置和命令

目标：

- 支持远程修改上报周期。
- 配置带版本和 CRC。
- 重启后配置仍生效。

验收：

- 下发命令后设备回复 ack。
- 配置损坏时恢复默认值。

思考：

- 配置写 Flash 时断电，启动后如何选择有效副本？
- 命令下发成功但 ack 丢了，云端重发时设备如何避免重复写入？

### 阶段 6：OTA 预留

目标：

- 设计 OTA 状态机。
- 预留分区和版本检查接口。
- 实现下载/校验的 mock 流程。

验收：

- 能从 `IDLE -> CHECK -> DOWNLOAD -> VERIFY -> READY` 走通状态。
- 校验失败能回到安全状态。

思考：

- 新固件下载完成后，什么时候才允许切换启动标记？
- 如果新固件启动后 MQTT 连不上，算升级成功还是失败？

## 故障注入

至少验证：

| 故障 | 注入方法 | 预期行为 | 必须保留的证据 |
|---|---|---|---|
| 传感器无响应 | 拔掉传感器或断电 | 返回明确错误码，系统继续运行 | I2C 波形、错误计数 |
| I2C 总线异常 | 拉低 SDA/SCL 或配置错地址 | 超时并尝试总线恢复 | 超时日志、恢复次数 |
| Wi-Fi 断开 | 关闭 AP | 进入退避重连，不阻塞采集 | 状态机日志、缓存深度 |
| MQTT broker 不可达 | 停止 broker | 发布失败并缓存/丢弃按策略执行 | 重连次数、丢弃计数 |
| 队列满 | 减小队列或暂停消费者 | 按策略丢弃或背压，不死锁 | 队列高水位、overflow 计数 |
| Flash 配置 CRC 错 | 手工破坏配置 | 加载默认值或旧副本 | 配置版本、CRC 检查日志 |
| 任务卡死 | 在任务中注入死循环 | 看门狗复位并记录任务心跳 | 复位原因、心跳快照 |
| 非法上报周期 | 下发 0ms 或极小值 | 拒绝命令并 ack 错误码 | command ack、错误码 |

## 证据清单

项目完成时至少归档：

- 一段上电串口日志，包含版本、复位原因和配置版本。
- 一张 I2C/SPI 读传感器 ID 的波形图或解析记录。
- 一份任务表，包含优先级、周期、栈大小、high water mark。
- 一份弱网或断网测试记录，包含状态机转移和缓存深度。
- 一份配置损坏恢复记录，包含 CRC 错误和默认值恢复。
- 一份故障复盘，说明至少一个真实或模拟问题的根因和预防措施。

## 项目复盘问题

- 哪些模块和硬件绑定太紧？
- 哪些错误码不够可定位？
- 哪些日志对定位有帮助，哪些是噪声？
- 任务优先级是否有证据支撑？
- OTA 如果断电，系统如何恢复？
- 量产时如何写入 SN 和校准参数？

## 项目洞察

环境监测节点看似是入门项目，真正价值在于它包含了完整产品链路的缩影：传感器采集、滤波、任务调度、低功耗、网络连接、离线缓存、远程配置、OTA、日志和产测。不要把它做成“读数上报 demo”，要把它当作训练系统边界的样机。

项目中最值得深挖的不是某个驱动，而是跨模块问题：

- 传感器数据是状态还是事件？缓存满时能否覆盖？
- 网络断开时设备继续采样、降频采样还是进入低功耗等待？
- 云端下发采样周期后，旧配置、新配置和正在进行的采样如何一致切换？
- OTA 升级后配置 schema 变化，旧数据和旧固件回滚如何处理？
- 任务卡死、传感器超时、网络重连风暴和存储写满如何被看门狗和诊断系统区分？

完成项目后，建议写一份“如果要卖 10000 台”的扩展评审。把成本、产测、证书、批次差异、远程维护、数据隐私和售后纳入讨论，项目就会从功能练习升级为工程训练。
