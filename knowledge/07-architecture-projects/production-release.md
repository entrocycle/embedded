# 量产、测试与发布

从 Demo 到产品，中间差的是可重复制造、可测试、可追溯、可升级、可诊断。量产问题不提前设计，后期会用大量人工补洞。

## 本章任务与边界

完成本章后，你应能定义硬件/固件/配置/身份版本，设计产测、校准、烧录、密钥注入和追溯流程，建立发布包、灰度、回滚、现场诊断和退役策略，并为误烧、断电、重复工位和返修建立防错。

量产流程涉及硬件安全、密钥权限、法规和供应链，不能直接复制教学命令。测试通过只证明已覆盖项目，无法保证未测焊接缺陷、器件替代、老化和现场环境。

完成证据：一台设备从空板到可出货的可重放记录、失败工位隔离、SN/校准/证书追踪、发布签名/hash、升级回滚和退役演练。

## 版本管理

至少区分：

- 硬件版本：PCB 版本、BOM 版本。
- Bootloader 版本。
- App 固件版本。
- 协议版本。
- 配置版本。
- 产测程序版本。

固件版本建议包含：

```text
MAJOR.MINOR.PATCH+BUILD
git_commit
build_time
target_board
dirty_flag
```

启动日志示例：

```text
bootloader=1.2.0 app=2.4.3+1024 board=hw_b git=abc1234 reset=watchdog
```

## 产测流程

```text
上电
  ↓
下载产测固件
  ↓
写入 SN/密钥/校准参数
  ↓
电源/时钟/存储自检
  ↓
外设测试
  ↓
通信测试
  ↓
功耗测试
  ↓
下载正式固件
  ↓
最终校验
  ↓
上传测试记录
```

## 产测项目

| 项目 | 目的 |
|---|---|
| 电源电压 | 发现焊接、电源芯片、短路问题 |
| 晶振/RTC | 确认时钟可靠 |
| Flash/RAM | 确认存储可读写 |
| GPIO | 确认关键控制脚 |
| UART/I2C/SPI | 确认总线和外设 |
| 传感器 | 确认读数范围 |
| 无线 | 确认 MAC、RSSI、联网 |
| 功耗 | 发现漏电和模组未睡眠 |
| 按键/LED/显示 | 确认交互部件 |

## SN 与校准

SN 设计要求：

- 唯一。
- 可追溯批次。
- 可被设备和云端读取。
- 不依赖外包装。

校准数据要求：

- 带版本。
- 带 CRC。
- 有默认值和失效处理。
- 记录校准时间和工位。

## 配置分区

配置分区建议双副本：

```text
config_slot_a
  version
  sequence
  length
  payload
  crc

config_slot_b
  ...
```

启动时选择：

1. CRC 正确。
2. sequence 最新。
3. version 可迁移。

写入时先写新副本，再切换 sequence，降低断电损坏风险。

## 发布前测试

功能测试：

- 正常采集/控制/通信。
- 配置修改。
- 异常输入。

稳定性测试：

- 72/168 小时长稳。
- 高低温。
- 电源插拔。
- 弱网。
- 外设断开。

升级测试：

- 正常升级。
- 下载中断电。
- 写入中断电。
- 新固件失败回滚。
- 降级策略。

安全测试：

- 非法固件。
- 过期版本。
- 错误证书。
- 未授权命令。

## 发布包内容

一次正式发布应包含：

- 固件 `.bin/.hex`。
- ELF 和 map 文件归档。
- 符号文件。
- 版本说明。
- Flash 分区说明。
- 配置默认值。
- 已知问题。
- 升级路径。
- 回滚路径。
- 测试报告。
- SBOM、第三方许可证与源码/声明义务。
- 工具链、依赖、构建配置和容器 digest。
- 安全公告、已知漏洞评估和支持期限。

## BOM 与器件变更

硬件 BOM 变化可能让“软件完全相同”的设备产生不同启动、时序、校准、功耗和故障行为。对替代料和批次变化建立：

- 器件型号、修订、供应商和适用板卡序列范围。
- 驱动/设备树/校准兼容性。
- Errata、电气、时序和温度差异评审。
- 需要重跑的 bring-up、波形、功耗、长稳和产测项目。
- 现场诊断可读取的硬件/BOM variant。

软件不要只靠编译宏假设硬件版本。启动时读取 strap、EEPROM、OTP 或生产配置，并验证组合合法；遇到未知组合应拒绝危险功能或进入可诊断降级状态。

### 产线防错与幂等

产线步骤会超时、重复、断电和人工重试。工位命令需要幂等或明确事务状态：

- 同一 SN 不得被写入两台设备，重试不得生成新身份。
- 密钥注入成功但回执丢失时，不能无条件再次写入或导出密钥。
- 校准中断后不得把半完成数据标记为有效。
- 测试失败设备要物理/系统隔离，不能通过后续步骤覆盖失败状态。
- 返修后保留原记录和变更链，不伪装成首次生产。

工装、产测固件、脚本和服务端同样要版本化、最小权限和审计。产测日志中不得包含私钥或可重放认证材料。

## 现场诊断

现场设备至少要能提供：

- 设备 SN。
- 固件版本。
- 复位原因。
- 最近错误码。
- 网络状态。
- 升级状态。
- 关键计数器。
- 最近一次异常时间。

可通过：

- 串口命令。
- BLE 诊断。
- 云端日志。
- 本地故障文件。

## 工程洞察

量产发布是对前面所有设计假设的总审计。开发板上功能正常，只说明单件样机在有限条件下工作；量产要求系统在供应链波动、批次差异、产线误操作、现场异常和长期维护中仍可追溯、可恢复、可诊断。

发布前要特别关注：

- 可追溯：每台设备的硬件版本、固件版本、SN、校准数据、测试结果和烧录记录能否关联。
- 可恢复：产测失败、升级失败、配置损坏、证书错误和首次联网失败后是否有明确流程。
- 可诊断：现场报修时能否拿到版本、复位原因、错误计数、日志和关键状态。
- 可回滚：新版本失败时是否能回到旧版本，配置和数据是否兼容。
- 可审计：发布包是否包含 hash、签名、map、符号、构建环境和测试报告。

量产阶段不适合依赖个人经验和口头流程。任何需要“某个人记得怎么做”的步骤，都应该被脚本、清单、工装、权限系统或文档替代。
