# 项目阶梯：两条系统路线，一个交付终点

系统化学习需要项目承接，但项目阶梯不是固定 demo 清单。它的设计原则是：每一级只增加少量主导复杂度，同时保留前一级已经建立的启动、错误、测试和诊断能力。

本教程在共同基础后分成两条项目路线：MCU/RTOS 与 Embedded Linux/BSP。两条路线不互为前置，最终都可以进入联网、安全、OTA、CI、量产和现场维护的产品交付阶段。

## 本章任务与边界

完成本章后，你应能选择与目标岗位匹配的项目路线，为每一级定义进入条件、新增复杂度、不变量、证据包和退出门，并证明后续项目真正复用了前一阶段资产。

项目名称和硬件可以替换，阶段门不能用截图或口头描述替代。一个控制器、数据记录器、网关或 HMI 都能承载同样能力；涉及危险电源、机械和执行器时，必须增加专业安全评审与独立保护。

## 总体结构

```text
                 M0 最小固件 -> M1 采集控制 -> M2 裸机架构
                /                               |
C0 共同基础 ---+                                v
                \                         M3 RTOS 系统 -> M4 MCU IoT/OTA --+
                 L0 交叉开发 -> L1 Linux 系统 -> L2 驱动与服务 -> L3 网关 --+
                                                                         |
                                                                         v
                                                    X1 架构与验证 -> X2 产品交付
```

所有阶段贯穿：版本、错误码、测试、证据、故障注入、资源预算和复盘。它们不是最后 X 阶段才加入的工作。

## 共同完成标准

| 维度 | 功能性完成 | 工程性完成 |
|---|---|---|
| 目标 | 功能可以演示 | 条件、阈值、失败和安全状态可判定 |
| 实现 | 代码能运行 | 接口、所有权、并发、超时和版本清楚 |
| 资源 | 没有明显溢出 | RAM/Flash/栈/存储/带宽/功耗有预算和水位 |
| 调试 | 有日志 | 有原始证据、分层观测和故障上下文 |
| 测试 | 手动成功一次 | 边界、状态序列、故障注入和回归 |
| 构建 | 本机能编译 | 干净环境可重建，产物可追溯 |
| 发布 | 有 bin/镜像 | manifest、hash/签名、符号、测试和回滚 |
| 复盘 | 描述做了什么 | 解释假设、取舍、残余风险和迁移条件 |

每一级至少达到工程性完成中的大多数要求，才能继续叠加复杂度。

## C0：共同基础项目

目标：在没有目标板依赖的情况下建立严格构建、边界测试、证据和版本意识。

### 任务

- 运行 `labs/ring-buffer`，理解空满、wrap 和参考模型。
- 运行 `labs/byte-stream-parser`，验证分片、合并、畸形和截断。
- 运行 `labs/host-config-journal`，穷举断电点和恢复不变量。
- 对一个程序做对象、符号、段和反汇编分析。

### 退出证据

- 工具版本、完整命令和退出码。
- 主动破坏一项保护后的首个失败测试。
- 修复后的同条件回归。
- 三个主机结论分别有哪些目标板假设尚未证明。

### 保留资产

- 严格编译选项和主机测试框架。
- ring buffer、parser、配置 schema/测试。
- 实验记录模板和问题调查模板。

## 路线 M：MCU 与 RTOS

### M0：可解释的最小固件

新增复杂度：复位、链接地址、时钟、GPIO、UART 和调试器。

任务：

- 从启动文件与链接脚本构建 ELF/map/bin。
- 在复位入口和 `main()` 断点。
- 输出版本、板卡、构建标识和复位原因。
- 用 GPIO 和 UART 建立最早可用观测。

退出证据：

- SP/PC/VTOR 与 map 一致。
- `.data/.bss` 初始化验证。
- GPIO/UART 寄存器和波形。
- 错误链接地址或时钟配置的故障复盘。

保留不变量：后续加入任何框架后，仍能从复位解释到应用入口，并在启动早期获得版本和复位证据。

### M1：可恢复的采集与控制

新增复杂度：I2C/SPI/ADC、真实传感器、校准、执行器和物理失败。

任务：

- 读传感器 ID 和数据，建立单位、范围和质量标志。
- 驱动返回有区分度的超时、NACK、总线和设备错误。
- 若有执行器，定义命令、保护、安全状态和独立反馈。
- 使用逻辑分析仪/示波器对照手册时序。

退出证据：

- 成功波形与至少三种失败波形/状态。
- 驱动接口的时间、所有权、错误和上下文契约。
- 传感器断开、错误地址、总线占用和异常值测试。
- 执行器软件输出与物理反馈的证据边界。

保留资产：板级 bring-up、驱动测试、波形基线、错误码和安全状态。

### M2：有界裸机系统

新增复杂度：多个功能并行、时间回绕、事件过载和资源所有权。

任务：

- 把阻塞流程改成运行至完成状态机。
- 建立合作式调度、单调时间与 deadline。
- 使用事件队列连接 ISR 和主循环。
- 量化主循环最坏一圈时间和事件响应。

退出证据：

- 状态/事件/守卫/超时表。
- GPIO 探针测得的 step 时间和最大响应。
- 队列满、外设永不完成和 tick 回绕实验。
- 是否需要 RTOS 的决策记录。

保留不变量：所有等待有界，过载可见，关键本地动作不依赖网络或日志。

### M3：可分析的 RTOS 系统

新增复杂度：抢占调度、任务栈、队列/锁、优先级反转和内核配置。

任务：

- 从 M2 的时间和所有权模型推导任务，而不是按文件拆分。
- 设计优先级、deadline、WCET、阻塞和栈预算。
- 选择队列、通知、mutex、事件组和服务任务。
- 建立任务心跳、看门狗、trace 和资源水位。

退出证据：

- 任务参数表、通信图和阻塞链。
- 响应时间分析与目标板 trace 对照。
- 队列满、优先级反转、栈不足、任务卡死和外设超时注入。
- 裸机与 RTOS 版本的同条件指标比较。

保留不变量：M0 启动仍可解释，M1 驱动错误与波形仍有效，M2 过载语义没有被 RTOS API 隐藏。

### M4：联网、低功耗与可信升级

新增复杂度：弱网、数据语义、TLS、持久化、功耗和多镜像。

任务：

- 建立分层连接状态机、退避和离线缓存。
- 区分状态、事件、命令和诊断数据。
- 实现 request ID、过期、幂等/去重和结果状态。
- 建立低功耗状态能量模型。
- 设计配置和 OTA 的提交、校验、首次启动确认与回滚。

退出证据：

- DNS/TLS/broker/断网和恢复积压测试。
- 缓存容量、重连能量和平均功耗预算/实测。
- 配置与 OTA 各阶段断电矩阵。
- 新旧固件、配置 schema 和协议兼容表。
- 设备身份、密钥和调试口生命周期设计。

推荐整合项目：[环境监测节点](project-environment-node.md)。

## 路线 L：Embedded Linux 与 BSP

### L0：可追溯交叉开发

新增复杂度：目标架构、sysroot、动态链接器、部署和远程调试。

任务：

- 用匹配工具链/sysroot 交叉构建程序。
- 检查 ELF 架构、解释器、依赖和 build ID。
- 部署并使用 `strace`/gdbserver/core 建立诊断闭环。

退出证据：

- 工具链、sysroot、RootFS 和程序版本关系。
- 一个架构/动态链接器错误的分层定位。
- 运行二进制与归档符号一致的远程调试或 core 解析。

保留资产：工具链配置、部署脚本、符号归档和版本命令。

### L1：可解释、可复现的 Linux 系统

新增复杂度：BootROM/SPL/U-Boot、Kernel、DTB、RootFS 和 init。

任务：

- 构建或固定 BSP 与 RootFS 配置。
- 保存并标注完整启动日志。
- 证明实际加载的 Kernel、DTB、RootFS 与源码/配置一致。
- 定义只读系统、持久数据、身份和临时数据边界。

退出证据：

- 启动阶段、镜像、cmdline、分区和 mount 记录。
- 错误 DTB/root 参数/服务依赖的分层故障。
- 两次干净构建比较和镜像 manifest/hash。

保留不变量：任何现场镜像都能追到工具链、源码、配置和依赖。

### L2：设备模型与受管理服务

新增复杂度：设备树 binding、probe 生命周期、用户态权限、IPC 和 systemd。

任务：

- 接入标准子系统或实现合理的 platform/总线驱动。
- 处理 probe defer、错误路径、IRQ/work/DMA 与 remove/suspend。
- 构建非 root 用户态服务，设计配置、日志、健康和有限重启。

退出证据：

- DTS 源码、schema、DTB 和 runtime DT 一致性。
- driver binding、probe 中途失败和异步清理测试。
- systemd 权限、依赖、退出码、重启限速和 watchdog 证据。
- `strace` 区分路径、权限、设备和 ioctl 问题。

保留不变量：用户接口可见时底层资源有效，服务重启不能抹掉根因或破坏数据。

### L3：可维护的边缘网关

新增复杂度：本地持久化、规则、弱网、安全、应用/系统更新和长期运维。

任务：

- 区分关键事件与普通遥测的存储和上传语义。
- 建立连接状态机、TLS、幂等和离线恢复。
- 设计最小权限诊断、core/符号和健康指标。
- 实现应用或系统更新、确认和回滚。
- 建立 30 天资源趋势、磁盘满和只读故障策略。

退出证据：

- 本地 deadline 在云端不可用和恢复上传压力下仍满足。
- 存储容量/寿命、积压恢复和资源趋势报告。
- 服务崩溃、配置损坏、磁盘满、时间错误和弱网矩阵。
- 应用/系统更新逐阶段断电与回滚。
- 发布包含 SBOM、许可证、符号、测试和残余风险。

推荐整合项目：[Embedded Linux 边缘网关](project-linux-gateway.md)。

## 汇合阶段：产品工程

### X1：需求、架构与验证

两条路线在这里使用相同的方法：

- 从用户场景提取可判定需求、假设和系统不变量。
- 维护上下文、模块、数据、时间、错误和信任边界图。
- 建立资源预算、风险分析和需求-测试-证据追踪。
- 把主机、仿真、目标板、HIL、长稳和现场证据分级。

退出证据：至少一个关键需求从风险、实现、故障测试追到原始证据；至少三个架构决策记录候选方案、代价和重新评审条件。

### X2：可量产、可发布、可维护

新增复杂度：供应链、批次差异、产测、身份、发布、灰度、漏洞响应和退役。

任务：

- 建立 CI、HIL 与发布门禁。
- 生成 manifest、SBOM、许可证、hash/签名和符号归档。
- 设计 SN、校准、证书/密钥注入、工位防错和追溯。
- 验证真实发布包的烧录、升级、回滚和现场诊断。
- 建立依赖漏洞评估、补丁、灰度和应急响应。
- 设计设备转移所有权与退役。

退出证据：

- 从空板/空镜像到出货的可重放记录。
- 一项产线失败被隔离且不会误判通过。
- 发布包可从源码与构建环境重现。
- 一次升级事故、凭据泄漏或依赖漏洞的桌面演练。
- 已知问题与残余风险清单。

## 跨级复用账本

每升一级，更新一张表：

| 资产 | 来源阶段 | 当前阶段如何复用 | 是否修改 | 为什么 |
|---|---|---|---|---|
| 启动版本日志 | M0 | M4 OTA 确认与现场诊断 | 增加 boot state | 需要解释回滚 |
| ring buffer 测试 | C0 | M1 UART RX | 加 ISR 目标板测试 | 主机不证明并发 |
| parser | C0 | M4/L3 网络 | 增加 schema/version | 产品协议演进 |
| 配置日志 | C0 | M4 配置服务 | 替换 Flash primitive | 目标原子粒度不同 |
| build ID/符号 | L0 | L3 现场 core | 自动归档 | 支持发布版本 |

若某项资产每个项目都被重写，说明接口、测试或平台层没有形成真正复用。

## 项目文档最低结构

```text
目标、范围与使用者
可判定需求、硬约束和假设
系统不变量与最高风险
硬件/平台/工具链/依赖版本
上下文、模块、数据、时间、错误和信任边界图
接口契约、状态机和资源预算
实现里程碑与决策记录
测试矩阵、故障注入和原始证据
构建、发布、升级、回滚和现场诊断
已知问题、残余风险和迁移复盘
```

## 项目深度量规

| 等级 | 项目表现 |
|---|---|
| L0 演示 | 主路径出现过，依赖个人环境 |
| L1 可解释 | 架构、机制和资源能说明 |
| L2 可验证 | 主路径、边界和原始证据可复核 |
| L3 可工程 | 故障、预算、回归、发布和诊断形成闭环 |
| L4 可迁移 | 约束变化后能重建模型并评审原结论 |

M0/L0 项目至少达到 L2；M3/L2 以后至少达到 L3；X2 应包含 L4 约束变化评审。

## 三个强制迁移题

每个完整项目都改变三项条件重新设计：

1. 资源减半：RAM、Flash、存储或带宽减少 50%。
2. 规模放大：设备数量或离线时间扩大 100 倍。
3. 风险升级：从监测改为控制可能造成损失的执行器。

回答哪些不变量保持、哪些预算先失效、哪些接口需要改变、哪些测试必须重跑。迁移题没有唯一答案，但不能只说“换更强硬件”。

## 最终判断

项目阶梯有效的标志，不是仓库数量越来越多，而是：

- 旧能力在新约束下仍被维护。
- 测试、接口、诊断和构建资产不断复用。
- 每一级只增加可解释的新复杂度。
- 失败方式越来越可控，证据越来越接近真实产品。
- 最终交付在作者不在场时仍可构建、运行、调查和回滚。

这才是项目驱动学习的系统性。
