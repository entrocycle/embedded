# 项目阶梯

系统化学习需要项目承接。项目阶梯的设计目标，是让每个项目都复用上一阶段能力，同时引入一个新的工程难点。

项目不是章节练习的集合，而是把多个约束同时放到一个系统里：硬件资源、实时性、功耗、异常、升级、测试、维护和表达。每个项目都要留下证据，而不是只留下“已经跑通”的截图。

## 阶梯总览

| 阶段 | 项目 | 新增能力 | 主要产出 |
|---|---|---|---|
| P0 | 裸机最小系统 | 启动、GPIO、UART、构建 | 可下载固件、串口日志 |
| P1 | 传感器采集器 | I2C/SPI、ADC、数据手册 | 传感器驱动、波形记录 |
| P2 | RTOS 环境节点 | 任务、队列、定时器、看门狗 | 任务设计、栈统计 |
| P3 | 联网数据上报 | TCP/IP、MQTT、状态机 | 断线重连、离线缓存 |
| P4 | Bootloader + OTA | 分区、签名、回滚 | 升级流程、失败矩阵 |
| P5 | 嵌入式 Linux 网关 | U-Boot、DTS、RootFS、服务 | 可启动镜像、systemd 服务 |
| P6 | 产品化闭环 | 产测、日志、诊断、CI | 发布包、测试报告 |

## 每个项目的深度标准

| 维度 | 最低要求 | 深度要求 |
|---|---|---|
| 功能 | 主路径能跑通 | 失败路径有状态机和恢复策略 |
| 代码 | 模块能编译 | 接口边界、错误码、资源占用清楚 |
| 调试 | 能看串口日志 | 有波形、dump、trace 或指标数据 |
| 测试 | 手动验证一次 | 有故障注入矩阵和回归步骤 |
| 文档 | 有 README | 有架构图、任务表、复盘和取舍说明 |
| 表达 | 能介绍做了什么 | 能解释为什么这么设计和如何证明可靠 |

## P0 裸机最小系统

目标：

- 能从空工程启动到 `main()`。
- 能控制 GPIO。
- 能输出串口日志。
- 能生成 ELF/HEX/BIN 和 map 文件。

验收：

- 说明启动文件、链接脚本和内存布局。
- 串口输出固件版本和复位原因。
- map 文件中 Flash/RAM 占用清楚。

思考：

- 如果串口没有任何输出，你如何区分是启动失败、时钟错误、引脚复用错误还是波特率错误？
- 为什么版本号、编译时间和复位原因应该在最早期输出？

## P1 传感器采集器

目标：

- 通过 I2C/SPI 读取传感器。
- 实现平台无关驱动接口。
- 对采样数据做简单滤波。

验收：

- 抓取一次总线波形并标注地址、寄存器和 ACK。
- 驱动包含 ID 检查、超时和错误码。
- 输出采样周期和错误计数。

思考：

- 传感器驱动如何做到平台无关？哪些部分必须留给 BSP？
- 如果采样数据偶发跳变，如何区分是传感器噪声、总线错误、供电问题还是滤波策略问题？

## P2 RTOS 环境节点

目标：

- 把采集、日志、控制、通信拆成任务。
- 用队列或事件组通信。
- 加入任务心跳和看门狗。

验收：

- 任务表包含优先级、周期、栈大小。
- 能故意阻塞一个任务并触发看门狗复位。
- 有栈 high watermark 记录。

思考：

- 任务优先级是根据什么证据定的，而不是根据感觉定的？
- 看门狗喂狗策略如何避免“系统已经坏了但仍然继续喂狗”？

## P3 联网数据上报

目标：

- 使用 Wi-Fi、以太网或蜂窝模组联网。
- MQTT 或 HTTP 上报数据。
- 支持断线重连和离线缓存。

验收：

- 有明确连接状态机。
- 网络断开后不丢关键告警。
- 队列满和缓存满行为明确。
- 云端或本地 broker 可看到设备状态。

思考：

- 断网期间哪些数据必须保留，哪些数据可以丢弃？
- 重连退避如何兼顾云端压力、设备功耗和恢复速度？

## P4 Bootloader + OTA

目标：

- 设计固件分区。
- 支持版本检查、下载、校验和跳转。
- 支持失败回滚。

验收：

- 写出升级状态机。
- 测试下载失败、校验失败、写入失败、断电、新固件启动失败。
- 启动后能报告当前版本和上次升级结果。

思考：

- 新固件第一次启动后，什么时候才算“确认成功”？
- 如果 Bootloader 自身需要升级，风险边界和恢复策略会如何变化？

## P5 嵌入式 Linux 网关

目标：

- 构建 Bootloader、Kernel、DTS、RootFS。
- 写一个基础驱动或启用一个外设。
- 写用户态服务采集数据并上报。

验收：

- 串口启动日志完整。
- 设备树修改可解释。
- systemd 服务可自动重启。
- 日志和状态命令可用于定位问题。

思考：

- 哪些逻辑应该放在内核驱动，哪些应该放在用户态服务？
- 如果远程设备无法 SSH 登录，你还需要哪些本地或云端诊断信息？

## P6 产品化闭环

目标：

- 建立 CI 构建和基础测试。
- 设计产测流程。
- 加入现场诊断和远程日志。
- 输出发布包。

验收：

- 发布包包含固件、hash、版本说明和烧录说明。
- 产测能写 SN、校准、测试关键外设。
- 长稳、断电、弱网、OTA 回归通过。
- 有问题复盘模板和故障码表。

思考：

- 发布包如何保证可追溯：源码、工具链、配置、依赖和产物 hash 如何关联？
- 产测项目如何区分“硬件问题”“校准问题”和“软件问题”？

## 项目文档模板

每个项目都要留下：

```text

已知问题：
设计取舍：
复盘：
```

## 复盘评分

可以用下面的问题检查项目是否真的有深度：

- 是否能从一张架构图讲清楚数据流、控制流和错误流？
- 是否能拿出至少一个波形、dump、trace、功耗曲线或长稳记录？
- 是否主动制造过失败，而不是只验证成功路径？
- 是否能解释一个关键取舍，例如轮询/中断/DMA、队列/事件组、用户态/内核态、A/B 分区/单分区？
- 是否能把项目讲给面试官、硬件工程师和测试工程师听，并各自说到他们关心的证据？

## 面试表达

做项目时同步积累面试材料：

- 架构图。
- 状态机图。
- 一次最难问题的定位过程。
- 波形、日志或性能数据。
- 失败场景和处理策略。

## 练习

1. 从 P0 到 P3 选择一个项目，补齐项目文档模板。
2. 为 P4 OTA 项目列出至少 10 个失败场景。
3. 把 P2 RTOS 项目的任务表整理成面试可讲版本。
