# 架构与项目

项目能力是嵌入式知识的汇合点。一个能长期维护的嵌入式项目，需要清晰的分层、稳定的接口、可定位的问题路径和可重复的构建交付。

## 工程分层

```text
Application   业务逻辑、状态机、产品功能
Service       日志、配置、OTA、通信管理、数据缓存
Middleware    RTOS、协议栈、文件系统、加密库
Driver        传感器、Flash、屏幕、无线模组
BSP/HAL       板级初始化、时钟、GPIO、外设适配
Hardware      MCU/SoC、电源、传感器、执行器
```

分层的目标不是增加目录，而是控制依赖方向。

建议按 [项目阶梯](project-ladder.md) 从裸机最小系统逐步推进到联网、OTA、Linux 网关和产品化闭环；同时用 [工具链、构建与 CI](toolchain-ci.md) 保证构建和发布可复现。

## 深挖问题

- 目录分层是否真的减少依赖，还是只把文件移动到了不同文件夹？
- 一个模块的接口是否写清楚错误码、超时、并发上下文和资源占用？
- CI 证明了什么，没有证明什么？哪些必须通过硬件在环或长稳测试验证？
- 发布包能否追溯到源码、工具链、配置、测试报告和现场诊断信息？

## 阶段产出

- 一张项目架构图，标出依赖方向、数据流和错误流。
- 一份模块接口文档，包含输入输出、错误码、资源、并发和验证方法。
- 一份发布包清单，包含固件、hash、map、符号、版本说明、升级和回滚路径。
