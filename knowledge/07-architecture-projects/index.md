# 架构、项目与产品交付

架构的目的不是把代码分进更多目录，而是让变化和失败有明确边界；项目的目的不是展示功能，而是把需求、资源、测试、发布和维护放进同一组约束；交付的目的不是产生一个固件文件，而是让设备能够被追溯、量产、升级、诊断和回滚。

本模块是前面技术路线的汇合点。你需要选择一个真实项目，将此前的启动、驱动、并发、网络、存储、调试和测试产物逐步整合，而不是重新写一个只追求演示效果的 demo。

## 模块契约

### 进入条件

- 已完成 MCU/RTOS 或 Embedded Linux/BSP 至少一条主线的系统阶段。
- 已有一个可运行项目和至少一份真实故障调查记录。
- 能用测试、波形、trace、抓包或构建产物支持关键结论。

### 完成后应能

- 把模糊产品需求改写为指标、约束、假设和不变量。
- 画出数据流、控制流、错误流、资源所有权和信任边界。
- 设计稳定依赖方向、模块接口、状态机和错误传播。
- 对 RAM/Flash、实时性、功耗、存储寿命和带宽做预算。
- 建立可复现构建、分层测试、CI 和硬件在环策略。
- 设计产测、校准、身份注入、版本、发布、回滚和退役。
- 让现场设备提供足够证据，使问题可以远程分流和收敛。
- 用决策记录说明取舍、残余风险和重新评审条件。

### 本模块不解决

- 不追求与需求无关的通用“完美架构”。
- 不把抽象层数量、设计模式数量或任务数量当成架构质量。
- 不用 CI 绿灯替代目标板时序、功耗、长稳和现场验证。

## 项目从需求开始

不要从选 MCU、RTOS 或云协议开始。先写一页项目约束：

| 类别 | 示例问题 |
|---|---|
| 用户价值 | 谁在什么场景下依赖设备完成什么？ |
| 安全状态 | 失去电源、网络、传感器或控制软件时，物理输出应怎样？ |
| 时间 | 哪些路径有截止期，哪些只是吞吐目标？ |
| 数据 | 哪些是状态、事件、命令、配置和审计记录？ |
| 资源 | 成本、RAM、Flash、功耗、带宽、寿命和开发时间上限？ |
| 环境 | 温度、振动、干扰、断网、维护可达性和设备规模？ |
| 生命周期 | 如何生产、激活、升级、诊断、转移所有权和退役？ |
| 假设 | 哪些条件尚未验证，错误后会推翻哪个设计？ |

每个模糊词都要改写。例如“稳定运行”应拆为负载、持续时间、允许复位、资源趋势、故障恢复时间和数据一致性要求。

## 五张项目图

一个项目至少维护以下五张图，图中对象必须能对应代码、指标或测试：

1. **上下文与信任边界**：设备、用户、云、产测、调试和外部接口。
2. **模块依赖图**：业务、服务、中间件、驱动、BSP/系统和硬件。
3. **数据流图**：状态、事件、命令、配置、日志和固件如何流动。
4. **控制与时间图**：ISR、任务、服务、状态机、截止期和阻塞。
5. **错误与恢复图**：错误从哪里产生，如何转换、隔离、恢复和记录。

只画一张“模块框图”通常无法表达时间、失败和信任关系。

## 工程评审门

### R1：需求与风险

- 指标可判定，假设有验证计划。
- 本地安全状态和关键不变量明确。
- 最高风险不是按个人兴趣排序，而是按影响与证据缺口排序。

### R2：架构与预算

- 依赖方向和资源所有者明确。
- 接口包含超时、错误、并发和生命周期。
- RAM/Flash、栈、吞吐、功耗、存储寿命至少有初步预算。
- 对关键方案保留备选与重新评审条件。

### R3：实现与集成

- 最小垂直路径先贯通并留证据。
- 驱动、服务和业务错误语义不被简单压成一个失败码。
- 日志、指标和故障注入点与架构同时实现。

### R4：验证与发布

- 需求、风险、测试和证据可追踪。
- 主机、仿真、目标板、HIL、长稳和现场试验各自边界清楚。
- 构建和发布产物可从干净环境重现并签名/hash。
- 回滚和恢复使用真实发布包验证。

### R5：量产与运维

- 产测能区分硬件、校准、身份和软件问题。
- 每台设备的硬件、固件、配置、证书和测试结果可关联。
- 现场能获得版本、复位、错误、健康、升级和关键状态。
- 设备退役、密钥撤销和数据清除有策略。

## 综合项目证据包

项目“完成”时至少归档：

```text
需求与可判定指标
约束、假设、风险和系统不变量
五张项目图
模块接口和资源预算
构建环境、依赖、manifest 和产物 hash
测试矩阵、原始证据与故障注入记录
长稳、功耗、弱网或实时性报告
版本、升级、回滚和配置迁移记录
产测、校准和现场诊断说明
关键决策记录、已知问题和残余风险
```

证据包不是为了增加文档量，而是让项目在作者不在场时仍能被理解、构建、验证和维护。

## 阶段门 G6

- [ ] 项目目标、硬约束、假设和安全不变量可以被不同角色复核。
- [ ] 架构图中的模块与代码依赖一致，错误流和信任边界没有被省略。
- [ ] 关键资源和时间预算有公式、测量或保守假设。
- [ ] 至少三种故障注入进入自动或可重复回归。
- [ ] 发布包可追溯到源码、工具链、配置、依赖和测试结果。
- [ ] 量产设备身份、校准数据、配置和证书有独立生命周期。
- [ ] 升级、回滚和现场诊断在接近真实部署条件下验证。
- [ ] 已知问题和残余风险公开，不用“后续优化”掩盖交付边界。

## 架构迁移题

完成项目后，强制改变三个约束重新评审：

1. 设备数量从 100 台变成 10 万台。
2. 网络从稳定 Wi-Fi 变成可能离线 48 小时的蜂窝连接。
3. MCU RAM 减半或 Flash 删除一半。
4. 传感器更换供应商且校准格式变化。
5. 产品需要支持五年 OTA 和旧版本回滚。
6. 设备控制从监测变成可能伤害人员或设备的执行动作。

观察哪些模块可以局部修改，哪些假设会推翻架构。可迁移架构不是不需要变化，而是变化路径和风险可以被解释。

本模块的最终出口，是把“我做过一个项目”升级为“我能用证据交付并长期维护一个系统”。
