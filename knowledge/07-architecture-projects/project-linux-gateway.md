# 综合项目：可维护的 Embedded Linux 边缘网关

本项目把交叉构建、启动链、设备树、驱动、用户态服务、网络、存储、安全、更新和现场诊断放入同一系统。目标不是“板子启动后上报一条 MQTT”，而是交付一个可追溯、可恢复、可验证的边缘网关。

可以使用 QEMU + 虚拟设备完成软件主线，再在 Raspberry Pi、i.MX6ULL、STM32MP1、RK356x 或其他 Linux 板上验证设备树、真实外设、功耗和更新。不同平台命令会变化，本项目以契约和证据为验收标准。

## 项目目标

网关连接一个本地传感器或 MCU 节点，完成：

- 从串口、RS-485、CAN、I2C/SPI 或模拟输入源采集数据。
- 将原始输入规范化为带质量、序列和时间的内部事件。
- 本地持久化关键事件，并通过 MQTT/HTTP 上报。
- 支持远程配置、诊断和受控应用/系统更新。
- 网络与云端不可用时继续执行本地采集和安全策略。
- 服务崩溃、存储耗尽、协议异常和升级失败时可恢复、可诊断。
- 从源码、工具链、BSP 配置和依赖重建发布镜像。

## 进入条件

- 已完成 [Linux 主机到目标机开发闭环](../04-embedded-linux/linux-development-workflow.md)。
- 能标注一份完整启动日志，并能证明实际加载的 DTB/RootFS。
- 理解驱动 probe/remove 生命周期和 systemd 服务管理。
- 已学习 [需求、风险与证据追踪](requirements-risk.md) 与 [应用层协议设计](../05-network-iot/application-protocol-design.md)。

## 需求与边界

先采用一组可修改的基线约束：

| 指标 | 基线目标 | 为什么重要 |
|---|---:|---|
| 本地采集周期 | 1 s | 决定设备/服务吞吐与时间模型 |
| 关键事件持久化 | 至少 7 天 | 决定容量、写放大和上传恢复 |
| 普通遥测离线保留 | 最新 24 h | 与关键事件采用不同语义 |
| 本地事件处理 deadline | 200 ms | 网络不能阻塞本地路径 |
| 服务异常恢复 | 30 s 内 | 包含重启限速和状态恢复 |
| 系统启动到业务就绪 | 45 s 内 | 需要分解启动关键路径 |
| 应用更新回滚 | 2 次失败内 | 防止无限重启新版本 |
| 系统可用周期 | 30 天无人工干预 | 需要资源趋势和故障注入 |

这些不是通用正确答案。根据设备用途重新定义，并记录假设。例如“7 天离线”若改为 90 天，会改变存储容量、恢复带宽、Flash 寿命和运维承诺。

## 系统不变量

项目至少维护：

1. 云端、DNS 或网络不可用时，本地采集和安全动作仍可运行。
2. 用户服务可访问设备接口时，底层驱动与资源已经就绪。
3. 任何已提交关键事件在确认上传前不会静默丢失；容量耗尽时进入显式降级并告警。
4. 任意更新失败后，设备至少保留一个可启动系统和一个可启动应用版本。
5. 设备身份、私钥和校准数据不会被普通系统更新覆盖。
6. 所有现场数据能关联硬件、内核、DTB、RootFS、应用和配置版本。

为每条不变量指定维护模块、故障注入和原始证据。

## 参考架构

```text
外部设备/MCU/传感器
        |
        v
Linux driver / standard subsystem
        |
        v
collector service -----> health/diagnostics
        |
        v
bounded event bus
   |              |
   v              v
local store    rules engine -----> local action
   |
   v
uploader state machine ---- TLS ---- cloud

update agent ---- signed bundle ---- boot/update slots
```

### 建议进程边界

| 组件 | 职责 | 不应承担 |
|---|---|---|
| 驱动/标准子系统 | 寄存器、中断、DMA、稳定硬件语义 | JSON、云协议、业务重试 |
| collector | 读取、校验、时间与质量标记 | 无限缓存、更新系统 |
| store | 原子持久化、保留、容量与恢复 | 决定所有业务优先级 |
| rules | 本地业务状态机和安全策略 | 直接操作私有寄存器 |
| uploader | 连接、退避、批量、确认与幂等 | 阻塞本地采集 |
| update agent | 包验证、安装、确认与回滚 | 持有设备长期私钥明文 |
| diagnostics | 聚合版本、状态、指标和受控命令 | 无认证 shell |

是否拆成多个进程取决于隔离、部署、资源和团队边界。单进程也必须保持模块与故障边界；多进程会引入 IPC、启动顺序和一致性成本。

## 数据契约

内部事件至少包含：

```text
schema_version
source_id
event_id
source_sequence
monotonic_timestamp
wall_timestamp + time_quality
value + unit
data_quality
severity
payload
```

要区分：

- 采样时单调时间与上报时墙上时间。
- 数据有效、过期、估算、校准失效和传感器故障。
- 普通状态、关键事件、命令结果和诊断日志。
- 上游设备 sequence 与网关 event ID。

不要让 JSON 直接成为内部所有模块的共享内存模型。先定义语义，再选择本地编码和云端格式。

## 存储设计

将数据分区或逻辑分类：

| 数据 | 一致性 | 保留 | 更新影响 |
|---|---|---|---|
| 系统镜像 | 签名、只读 | 当前 + 回滚 | A/B 或恢复分区 |
| 应用 | 签名/校验 | 当前 + 回滚 | 原子切换版本目录 |
| 设备身份 | 高保护 | 设备生命周期 | 普通更新不得覆盖 |
| 配置 | 原子更新、schema | 当前 + 上一有效 | 支持迁移与回滚 |
| 关键事件 | 持久、可审计 | 产品定义 | 满时显式降级 |
| 普通遥测 | 可覆盖/批量 | 有界窗口 | 满时按策略删除 |
| 日志/core | 限额、脱敏 | 时间/容量窗口 | 不能挤占关键数据 |

必须计算：

```text
日写入量 = 事件速率 x 单条落盘大小 + 索引/日志开销
恢复上传时间 = 离线数据量 / 可用净上传吞吐
存储寿命 ~= 可写总量 / 日写入量（再考虑写放大与裕量）
```

文件系统和数据库的 `write()` 成功不等于掉电后持久。需要理解事务、`fsync`、目录项、journaling 和介质缓存边界，并以目标存储验证。

## 分阶段实现

### 阶段 L0：可追溯启动

目标：从干净环境构建或获得匹配 BSP，启动到 shell，并保存版本证据。

产出：

- 分阶段启动日志。
- Bootloader、Kernel、DTB、RootFS 的 hash/版本。
- 内核 cmdline、分区和挂载表。
- 构建工具链、配置和依赖清单。

故障：错误 DTB、错误 root 参数、缺失驱动。要求能从日志区分层次。

### 阶段 L1：设备接入

目标：通过标准子系统或合理驱动接入数据源。

产出：

- 设备树 binding 与 schema 检查。
- 运行时 DT、driver binding、IRQ/设备节点证据。
- 用户态读取工具和错误码。
- probe 中途失败、设备无响应和 remove/suspend 设计。

若使用串口/SocketCAN 等现有子系统，不为展示驱动能力重复造字符设备；可以深入一个虚拟/简单 platform 驱动训练生命周期。

### 阶段 L2：受管理采集服务

目标：collector 通过 systemd 启动、读取数据并输出结构化事件。

产出：

- unit 依赖、用户/组、设备权限和安全选项。
- 配置 schema、原子更新和错误回退。
- 服务启动、退出码、重启限速和 watchdog 策略。
- `strace`、journal 和健康接口。

故障：权限丢失、配置损坏、设备暂时缺失、服务崩溃和重启风暴。

### 阶段 L3：有界本地存储

目标：关键事件与普通遥测使用不同保留策略。

产出：

- 容量、写放大和寿命预算。
- 断电/进程 kill 后恢复测试。
- 存储满、只读挂载、I/O 错误和数据库损坏策略。
- 队列年龄、水位、丢弃和关键事件未确认计数。

故障恢复不能无限重试写坏盘，必要时进入只读/降级并保留本地告警。

### 阶段 L4：联网与弱网

目标：上传服务在弱网中退避、批量、确认并恢复积压。

产出：

- 分层连接状态机。
- TLS 身份与证书验证。
- event ID、确认、重试和服务端去重语义。
- DNS/TLS/broker 故障与抖动测试。
- 离线 7 天后的恢复上传吞吐和本地 deadline 证据。

验证上传压力不会抢占本地采集、耗尽内存或填满日志。

### 阶段 L5：本地规则与命令

目标：远程配置/命令与本地安全策略共存。

产出：

- 授权、request ID、deadline 和结果状态机。
- 配置版本、迁移和回滚。
- 云端命令与本地手动/安全状态的优先级规则。
- 命令执行后 ACK 前崩溃的恢复证据。

对执行器使用独立反馈；用户态服务返回成功不代表物理动作完成。

### 阶段 L6：更新与产品交付

目标：应用或系统更新可验证、确认和回滚。

产出：

- manifest、签名、硬件兼容和防回滚策略。
- 应用与系统 A/B/版本目录方案。
- 健康确认条件，不能只以“进程启动”确认。
- 各安装阶段断电、磁盘满和新版本不兼容测试。
- 发布包、SBOM、许可证、符号、测试报告和回滚说明。

## systemd 服务契约

unit 至少评审：

- `After` 与 `Requires/Wants` 是否表达真实依赖。
- 使用专用非 root 用户和最小设备权限。
- 文件系统写路径、临时目录和只读保护。
- capability、system call、namespace 等隔离是否适用。
- `Restart`、`StartLimit*` 与退出码是否避免风暴。
- watchdog 通知是否代表有意义进展。
- 停止时 timeout、事务 flush 和强制 kill 后果。

安全加固不能盲目复制；限制必须通过功能和恢复测试，且记录为什么需要开放某项能力。

## 可观测性设计

### 必备身份

- 硬件/板卡/BOM 版本。
- Bootloader、Kernel、DTB、RootFS、应用和配置版本。
- 设备 ID、启动 ID、build ID 和升级状态。

### 必备健康指标

- 数据源错误与最后成功时间。
- IPC/队列水位和最老事件年龄。
- 存储使用、I/O 错误和未确认事件。
- 网络状态、重连次数和上传延迟。
- 服务重启、core、内存/句柄/线程趋势。
- 系统 uptime、温度、磁盘和时钟质量。

日志使用结构化事件 ID、限速和敏感信息脱敏。指标用于趋势，日志用于上下文，core/trace 用于深层证据；三者不能互相替代。

## 故障注入矩阵

| 故障 | 注入 | 预期 | 证据 |
|---|---|---|---|
| 数据源断开 | 拔线/停虚拟源 | 有界错误，不阻塞其他路径 | 驱动/服务状态和计数 |
| 服务崩溃 | signal/故意 fault | 保存 core，有限重启并恢复状态 | build ID、core、journal |
| 配置损坏 | 截断/错误 schema | 拒绝并回到旧有效配置 | 配置版本与错误码 |
| 磁盘满 | 限额/填充 | 优先保关键事件，显式降级 | 水位、删除/拒绝记录 |
| 文件系统只读 | remount/mock 错误 | 停止写风暴，继续安全功能 | errno、状态转移 |
| DNS/TLS 失败 | 规则/错误证书 | 分类退避，不影响本地 deadline | 状态机、重连能量 |
| 网络恢复 | 恢复服务 | 有界批量上传，不耗尽资源 | backlog 与吞吐曲线 |
| 更新中断电 | 每阶段复位 | 启动旧或完整新版本 | boot/update 状态 |
| 新应用不健康 | 健康检查失败 | 自动回滚且保留诊断 | 版本、失败原因、回滚 |
| 时间错误 | 清 RTC/阻断同步 | 标记 time quality，不伪造顺序 | 单调/墙上时间记录 |

## 性能与容量验证

至少建立：

- 启动时间分解，而不只记录总时长。
- 输入到本地规则动作的 P50/P99/最大延迟及负载条件。
- 离线积压恢复期间 CPU、内存、磁盘和本地 deadline。
- 30 天资源趋势：RSS、句柄、线程、磁盘、日志和队列。
- 更新包大小、写入时间、峰值临时空间和回滚时间。

平均指标不能掩盖尾部；采样工具的开销和时间基准必须记录。

## 安全评审

- 启动和更新包有签名与硬件兼容检查。
- 每设备身份与私钥有受控制造、轮换、撤销和退役。
- 服务以最小权限运行，诊断接口认证、授权、限速、审计。
- 输入 parser 有长度、类型、深度、速率和资源上限。
- 远程命令区分授权与本地安全策略。
- SBOM、许可证和依赖漏洞有响应流程。
- 日志、core 和诊断包不泄漏密钥与用户敏感数据。

## 项目复盘问题

1. 如果禁止 SSH 和远程 shell，当前诊断能力还剩多少？
2. 如果云服务停止 30 天，存储、时间、证书和恢复上传会怎样？
3. 如果底层 MCU 协议升级但网关回滚，双方如何协商和保留旧队列？
4. systemd 把服务拉起是否可能掩盖持续损坏状态？什么条件应停止重启？
5. 若 eMMC/NAND 出现 I/O 错误或变只读，哪些本地安全功能仍必须运行？
6. 如果目标板更换 SoC，哪些 BSP 层变化，哪些服务、协议和测试应保持？
7. 如果设备规模扩大到十万台，日志、证书、OTA 灰度和重连什么先成为瓶颈？

## 完成定义

- [ ] 从干净环境可重建并证明运行版本。
- [ ] 启动链、设备树、驱动和用户服务有端到端证据。
- [ ] 本地路径不依赖云端可用，并满足 deadline。
- [ ] 存储满、服务崩溃、弱网和时间错误有有界行为。
- [ ] 应用/系统更新在真实发布包和断电条件下可回滚。
- [ ] 30 天长稳中资源趋势有阈值，没有静默恶化。
- [ ] 现场诊断不依赖开发者登录设备临时猜测。
- [ ] 发布包包含 SBOM、许可证、符号、测试和残余风险。

完成这个项目，意味着你交付的不只是一台能启动的 Linux 板，而是一套能够长期构建、运行、更新和调查的设备系统。
