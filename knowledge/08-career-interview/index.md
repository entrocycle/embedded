# 职业发展与工程表达

职业能力不是把知识清单背熟，也不是把项目包装成更大的规模。真正有说服力的表达来自可复核工程证据：你面对什么约束，建立了什么模型，如何定位问题，为什么选择这个方案，怎样验证，以及留下了什么可以复用的资产。

本模块用于识别岗位差距、整理项目证据和训练技术沟通。它不能替代前面的实践；没有真实实验、故障和取舍时，任何面试模板都会迅速暴露为空洞叙述。

## 模块契约

### 进入条件

- 至少完成一个阶段项目，拥有代码、证据和复盘。
- 至少独立调查过一个非显然故障。
- 能诚实区分个人负责、团队协作、参考实现和未完成部分。

### 完成后应能

- 根据目标岗位拆解必需、加分和岗位无关能力。
- 用事实和证据评估能力，而不是按“了解/熟悉/精通”自评。
- 在 3 分钟内讲清项目目标、架构、约束和结果。
- 在 20 分钟追问中解释机制、故障、取舍、验证和残余风险。
- 对不会的问题建立推理边界，而不是猜测或背诵绝对结论。
- 将面试暴露的缺口转化为实验和项目计划。

## 推荐顺序

1. [岗位路线与能力准备](role-roadmap.md)：选择岗位，建立带证据的差距矩阵。
2. [面试题库](question-bank.md)：用问题检查机制、边界和迁移，不按答案背诵。
3. 回到对应技术章节和项目补证据，再更新项目表达。

## 岗位证据矩阵

| 方向 | 核心证据 | 常见虚假进度 |
|---|---|---|
| MCU/RTOS | 启动、波形、任务/阻塞、实时、低功耗、故障恢复 | 只列 HAL 和 RTOS API |
| Linux/BSP | 启动日志、运行时 DT、驱动生命周期、系统构建 | 只会改 DTS 和复制驱动模板 |
| IoT 产品 | 状态机、抓包、弱网、幂等、安全、OTA | 只展示云端收到数据 |
| 测试/可靠性 | 测试预言、故障注入、HIL、长稳和复盘 | 只列测试工具和覆盖率 |
| 架构/资深 | 需求、预算、取舍、演化、发布和现场数据 | 只画分层图和使用设计模式 |

## 项目表达的证据链

推荐结构不是固定话术，而是一条可追问链：

```text
场景与目标
  -> 约束与个人职责
  -> 架构与关键接口
  -> 最难的不确定性
  -> 假设、实验和原始证据
  -> 方案选择与放弃方案
  -> 指标结果与证据边界
  -> 回归、诊断和后续改进
```

每个数字都应能回答测量条件，每个“优化”都应有优化前基线，每个“稳定”都应有负载、时长和失败定义。

## 三层回答标准

面对任何技术问题，尽量给出三层答案：

1. **事实层**：对象是什么，基本机制是什么。
2. **边界层**：它不能保证什么，在哪些条件下失效。
3. **工程层**：在真实约束下如何选择、验证、诊断和恢复。

例如回答 `volatile`：

- 事实：约束编译器对该对象的抽象机访问，常用于 MMIO 和异步变化对象。
- 边界：不保证原子性、互斥、多变量一致性、cache 一致性或跨核顺序。
- 工程：根据 ISR/DMA/多核参与者选择架构保证、临界区、原子操作、屏障或所有权协议，并用反汇编与压力测试验证假设。

## 必做产出

- 一份目标岗位 JD 能力拆解，标注证据链接和缺口。
- 一个项目的一页架构与证据摘要。
- 两份完整故障复盘：一个底层问题，一个系统问题。
- 三个关键取舍记录，包含放弃方案和重新评审条件。
- 一次模拟追问记录，列出答错、答浅和证据不足的地方。
- 一个 4-8 周补缺计划，每项以实验或项目产物验收。

## 退出标准

- [ ] 简历中每个核心技术词都能被深入追问到机制和边界。
- [ ] 项目规模、指标和个人贡献没有模糊夸大。
- [ ] 能拿出 map、波形、trace、抓包、测试或发布记录支持项目结论。
- [ ] 能讲清一次错误假设和如何被证据推翻。
- [ ] 能解释当前方案代价和在新约束下如何重选。
- [ ] 面对未知问题能暴露假设并提出验证路径，而不是伪造确定性。

## 职业成长的判断方式

从初级到高级，不只是负责模块越来越大，而是管理不确定性的能力在提高：

- 初级：在明确边界内实现并验证功能。
- 中级：独立定位跨模块问题，设计接口和失败路径。
- 高级：管理约束、风险、演化和交付，帮助团队建立可复用机制。

最有效的职业准备，是不断把个人经验变成代码、测试、工具、清单、文档和决策。面试只是这些资产的一次表达场景。
