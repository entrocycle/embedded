# 面试题库

题库不是背答案，而是训练表达：先讲结论，再讲机制，再讲边界和工程证据。页面中的短答只给出回答骨架，不是完整标准答案；面试者应主动说明适用平台和假设。

## 使用方式与评分

先遮住短答，按四轮作答：

1. **边界**：用一句话定义概念解决什么、不解决什么。
2. **机制**：画出资源、时序、状态或调用链。
3. **反例**：给出一个常见说法会失效的条件。
4. **证据**：引用自己做过的日志、波形、反汇编、trace、测试或量化结果。

| 等级 | 表现 |
|---|---|
| 0 背诵 | 只复述本页短答或 API 名称 |
| 1 解释 | 能讲机制，但没有边界和实际证据 |
| 2 应用 | 能结合项目说明取舍、失败路径和观测 |
| 3 迁移 | 面对约束变化能修正结论，并提出验证方案 |

每个问题都可继续追问：“什么情况下不成立？”“你如何证明？”“如果资源减半怎么办？” 如果回答无法承受这些追问，说明掌握仍停留在术语层。

## C 语言

### `volatile` 能保证线程安全吗？

不能。`volatile` 只保证每次访问都从内存或寄存器读取/写入，防止编译器优化掉访问。它不保证原子性、顺序一致性，也不提供互斥。多任务共享变量仍需要临界区、锁或原子操作。

### `static` 有哪些作用？

- 修饰局部变量：延长生命周期到整个程序运行期。
- 修饰全局变量/函数：限制链接可见性，只在当前文件可见。
- 工程上常用于隐藏模块内部状态，减少全局命名污染。

### 栈溢出会有什么现象？

可能表现为随机 HardFault、变量被破坏、任务异常、返回地址错误。RTOS 中可开启栈溢出检测和 high water mark；裸机中可在栈区填充哨兵值并周期检查。

## MCU 与外设

### MCU 上电后如何执行到 `main()`？

上电复位后，CPU 从向量表读取初始栈顶和 Reset_Handler 地址。Reset_Handler 拷贝 `.data` 到 RAM，清零 `.bss`，初始化系统时钟和 C 运行环境，然后调用 `main()`。

### 为什么外设访问前要使能时钟？

多数 MCU 为了省电，外设时钟默认关闭。时钟未使能时访问外设寄存器可能无效，甚至触发总线错误。初始化外设第一步通常是打开对应总线和外设时钟。

### DMA 有什么好处和风险？

好处是减少 CPU 搬运，提高吞吐，适合 UART、ADC、SPI、音频等数据流。风险是 buffer 生命周期、对齐、cache 一致性、半传输处理和 CPU/DMA 同时访问导致的数据竞争。

## RTOS

### 任务优先级怎么设计？

按实时性和阻塞风险设计。硬实时控制和安全监控优先级高；网络、文件系统、日志等可能阻塞的任务优先级低。高优先级任务应短、确定、少阻塞。

### 队列满了怎么办？

取决于数据类型。遥测数据可丢旧保新；告警事件应持久化或短暂阻塞；控制命令应返回 busy；日志可按等级丢弃。关键是统计队列满次数，不能悄悄吞掉。

### ISR 和任务如何通信？

常用 FromISR API：二值信号量、队列、任务通知。ISR 做最少工作，清标志、取必要数据、通知任务，复杂处理放任务中。

## Linux

### 设备树的作用是什么？

设备树描述硬件资源，如寄存器地址、中断、GPIO、clock、总线设备。驱动通过 compatible 匹配设备，避免把板级硬件写死在内核代码里。

### platform driver 的 probe 做什么？

probe 在设备和驱动匹配后执行，通常获取资源、映射寄存器、申请中断、获取 clock/reset/gpio、初始化硬件，并注册字符设备或接入标准子系统。

### 用户态和内核态怎么划分？

内核态负责硬件访问、中断、DMA 和标准子系统接入；用户态负责策略、业务、网络协议、配置和 UI。复杂业务尽量不要放内核。

## 网络与 IoT

### MQTT QoS 0/1/2 区别？

- QoS 0：最多一次，可能丢，开销低。
- QoS 1：至少一次，不丢但可能重复，需要业务去重。
- QoS 2：MQTT 链路使用多步握手去重交付，流程和持久状态开销更大。

QoS 不保证数据库写入或执行器动作端到端恰好一次。继续追问：客户端在动作完成、ACK 发出前断电，重启后如何处理重复命令？合格回答应引出业务 `request_id`、幂等语义和持久化去重。

### 断线重连怎么设计？

用状态机管理网络和 MQTT 连接，失败后指数退避，限制最大重试频率。重连期间缓存关键数据，但要有上限。连接恢复后重新订阅主题并恢复上报。

### OTA 安全怎么保证？

使用 TLS 保护下载链路，固件包做 hash 和签名，Bootloader 使用内置公钥验证签名，防止版本回退，升级失败可回滚。

## 调试

### HardFault 怎么定位？

保存异常栈中的 PC、LR、SP、xPSR，并读取 CFSR/HFSR/BFAR/MMFAR。根据 PC 定位出错指令，结合 map/elf/objdump 判断是空指针、非法访问、栈溢出还是总线错误。

### I2C 没有 ACK 怎么查？

确认供电、上拉、电平、地址、SCL/SDA 是否接反、设备启动延时、总线是否被拉死。用逻辑分析仪看地址阶段是否 ACK，并确认 7 位/8 位地址处理。

### 低功耗电流偏高怎么查？

先分模块测电流，再逐个关闭外设和 GPIO。检查悬空输入、外设时钟、无线模组 sleep、传感器 power-down、调试器影响、IO 反向供电和 RTOS tick 唤醒。

## 项目追问

### 你项目中最难的问题是什么？

回答结构：

1. 背景和影响。
2. 复现条件。
3. 初步假设。
4. 证据和排除过程。
5. 根因。
6. 修复和验证。
7. 后续预防。

### 如何证明你的系统稳定？

回答要有指标：

- 长稳测试时长。
- 重启次数。
- 错误码统计。
- 栈和 heap 最小值。
- 队列高水位。
- 弱网/断电/外设异常测试结果。

这些指标仍不能抽象地“证明稳定”。回答还应限定硬件、固件、环境、工作负载、样本数和停止条件，并明确未覆盖风险。长稳 168 h 只证明在已测试条件下未观察到故障，不能直接外推整个产品寿命。

## 证据化练习

任选三个问题，为每个问题提交：一张机制图、一个反例、一份来自实验的原始证据，以及一次约束变化后的修正版答案。由同伴按上面的 0-3 级量规评分，并记录最先答不下去的追问。

## 如何把题库用深

题库不是用来背固定答案的，而是用来训练“可追问回答”。每个问题至少准备三版：

- 30 秒版：说清核心结论和最重要边界。
- 3 分钟版：补机制、例子、失败场景和调试证据。
- 10 分钟版：展开到项目经验、取舍、反例和迁移问题。

回答时优先避免绝对化表达。例如“`volatile` 保证变量最新”不如说“`volatile` 防止编译器省略或合并访问，但不保证原子性、互斥或跨核内存顺序”。这种表达方式能自然引出深层讨论，也能暴露你对边界的理解。

建议把每道题都改写成工程场景：

- 不是“什么是 DMA”，而是“DMA 接收偶发旧数据，如何定位 cache、一致性和所有权问题”。
- 不是“MQTT QoS 区别”，而是“远程开阀命令如何避免重复执行和误报成功”。
- 不是“HardFault 怎么查”，而是“现场设备重启后只剩复位记录，如何让下一版固件留下足够证据”。

这样训练，题库会从记忆材料变成项目表达和问题解决能力的放大器。
