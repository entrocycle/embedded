# 教程标题

## 学习目标

- 概念目标：能说明本主题解决什么问题，不解决什么问题。
- 机制目标：能解释关键资源、数据流、时序或状态机。
- 工程目标：能设计接口、异常处理、验证方法和复盘输出。
- 洞察目标：能指出一个常见直觉的失效条件，并说明真正的主导约束。
- 迁移目标：能判断换芯片、换 RTOS、换 Linux 板卡或换产品约束后，哪些结论仍成立。

## 深入思考

1. 如果把资源限制缩小一半，方案需要如何调整？
2. 如果该模块在中断、任务和多核环境中同时被访问，边界如何变化？
3. 如果换芯片、换 RTOS 或换 Linux 板卡，哪些设计可以保留，哪些必须重写？
4. 如果错误不能现场复现，哪些诊断数据足以支持远程判断？
5. 如果本章推荐方案失败，最可能是哪个假设错了？
6. 如果该功能涉及安全或财产风险，默认失败状态应该是什么？

## 教程完成后的反向评审

写完一章后，不要只检查“内容是否完整”，还要检查它是否改变了读者的判断方式。可以用下面问题反向审视：

- 读者是否能从现象反推出机制，而不是只记住 API？
- 读者是否知道这个方案在资源紧张、弱网、断电、长稳、量产或安全约束下会怎样变形？
- 章节是否给出了至少一个“看起来合理但其实危险”的反例？
- 最小实验是否有清晰预期，并能区分两个竞争假设？
- 故障注入是否覆盖检测、隔离、恢复和诊断，而不是只制造失败？
- 练习是否要求读者输出图、表、代码、测试或复盘，而不是只回答概念题？

如果这些问题答不上来，章节可能已经讲清了知识，却还没有完成工程训练。优秀教程的价值不在于替读者做完判断，而在于展示判断如何被构造、证伪、修正和迁移。
