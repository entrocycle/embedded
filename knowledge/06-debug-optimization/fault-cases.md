# 典型故障案例库

故障案例的价值在于训练定位思路。下面按现象、证据、根因、处理方式组织。

## 本章任务与边界

完成本章后，你应能在看答案前为案例建立分层假设、预测和最小观测顺序，区分触发条件、故障机制、薄弱设计与可见症状，并把案例中的方法迁移到不同平台。

短案例中的“常见原因”不是根因概率表，也不能替代目标板资料与测量。一个现象可能由多个故障叠加，同一修改也可能只改变触发概率。

完成证据：选择三个短案例写调查计划，再完整复盘一个陌生故障；必须记录未证明事项、修复边界和新增回归，而不是只记录最终根因。

前面的短案例适合现场建立检查顺序，不足以训练完整推理。使用时先遮住“常见根因”，为每个候选原因写出可区分的预测，再选择下一项观测。不要把清单从上到下机械执行：优先选择成本低、区分能力强、不会改变现场状态的证据。

## 上电无日志

现象：

- 串口无任何输出。
- 调试器偶尔无法连接。

定位顺序：

1. 测 VDD/VDDA 是否正常。
2. 检查 BOOT 引脚。
3. 检查 NRST 是否被拉低。
4. 确认串口 TX/RX 和波特率。
5. 用调试器 halt，看 PC 是否在预期地址。
6. 确认向量表和链接地址。

常见根因：

- Boot 模式进了系统 BootROM。
- 外部晶振未起振，时钟初始化卡死。
- 串口引脚复用错误。
- 固件链接到 `0x08008000`，但未使用 Bootloader 跳转。

## 一进中断就 HardFault

现象：

- 主循环正常。
- 使能某外设中断后崩溃。

定位：

- 查看向量表中对应 IRQ handler 是否存在。
- 查看中断函数名是否和启动文件一致。
- 查看是否清中断标志。
- 查看 ISR 是否访问空指针或未初始化对象。
- 读取 CFSR/HFSR。

常见根因：

- 中断函数名写错，进 Default_Handler。
- 外设中断标志未清，反复进入 ISR。
- ISR 调用了阻塞函数。
- RTOS 中 ISR 调用了非 FromISR API。

## I2C 设备无 ACK

现象：

- 逻辑分析仪看到地址后 NACK。

定位：

1. 确认设备供电。
2. 确认上拉电阻。
3. 确认 7 位地址还是 8 位地址。
4. 确认 SCL/SDA 是否接反。
5. 确认设备是否需要延迟启动。
6. 扫描 I2C 地址。

常见根因：

- 把 7 位地址左移后又交给 HAL，导致地址错误。
- 传感器地址脚电平与代码不一致。
- I2C 总线无上拉。
- 设备还未完成上电复位。

## SPI 读到全 0xFF 或 0x00

现象：

- 写命令有波形，读数据恒定。

定位：

- CS 是否有效。
- MISO 是否连接。
- CPOL/CPHA 是否正确。
- 读命令后是否需要 dummy byte。
- 芯片是否处于 power-down。

常见根因：

- CPHA 配错。
- CS 提前拉高。
- SPI 模式和器件手册不一致。
- MISO 被其他设备占用。

## UART 丢包

现象：

- 小数据正常，大量数据丢失。
- 日志输出影响接收。

定位：

- 看是否使用阻塞接收。
- 检查 RX buffer 大小。
- 统计 overrun 错误。
- 检查中断优先级。
- 检查处理任务是否来不及消费。

处理：

- 中断或 DMA 接收。
- 环形缓冲。
- 高水位统计。
- 协议层加入帧头、长度、CRC。

## RTOS 随机死机

现象：

- 运行几小时后无响应。
- 复现不稳定。

定位：

- 开启栈溢出检测。
- 记录每个任务 high water mark。
- 记录 heap 最小值。
- 统计队列满。
- trace 任务状态。
- 检查死锁和优先级反转。

常见根因：

- 某任务栈过小。
- 日志队列满后阻塞高优先级任务。
- 低优先级任务持锁时间过长。
- 动态内存碎片。

## OTA 后无法启动

现象：

- 下载完成重启后 Bootloader 不跳转。
- 或跳转后 HardFault。

定位：

- 检查固件头 magic/size/hash。
- 检查 App 链接地址。
- 检查向量表第 0 项是否在 RAM。
- 检查 Reset_Handler 是否在 App Flash 范围。
- 检查 VTOR/MSP 设置。

常见根因：

- App 仍链接到 `0x08000000`。
- 下载区和运行区混淆。
- 固件写入偏移错误。
- Bootloader 未关闭中断。

## 低功耗电流偏高

现象：

- Stop 模式电流远高于预期。

定位：

- 测整机和各模块分支电流。
- 逐个关闭外设和 GPIO。
- 断开外部模块验证。
- 检查调试器是否连接。
- 检查是否进入了真正的低功耗模式。

常见根因：

- GPIO 悬空。
- 外设时钟未关闭。
- 传感器/无线模组未 sleep。
- IO 反向供电。
- RTOS tick 周期唤醒。

## 完整调查案例：无线发送时偶发复位

本案例采用渐进揭示。每一轮先停下来回答三个问题：当前至少有哪些竞争假设？哪条证据最能区分它们？观测本身会不会改变故障？

### 初始报告

一批电池供电节点升级固件后，实验室运行正常，现场每天随机离线数次。设备会自动恢复。云端只看到：

```text
10:24:02 telemetry ok
10:24:03 offline
10:24:11 online boot_id=184
```

已知：设备每分钟上报，告警时立即上报；新版本增加了 TLS 调试日志。此时不能直接得出“网络不稳定”。至少列出四类假设：

| 假设 | 若为真可能看到什么 | 优先证据 |
|---|---|---|
| 云端或链路断开 | 设备没有重启，单调时间连续 | boot ID、复位原因 |
| 看门狗复位 | 关键任务长时间不推进 | watchdog 标志、任务心跳快照 |
| brown-out | 复位集中在无线峰值，电源跌破阈值 | 复位标志、电源触发波形 |
| 内存破坏 | PC/LR 或栈哨兵异常，可能与输入大小相关 | fault record、栈/heap 高水位 |

**选择题不是“你猜哪个根因”，而是“先取哪条证据”。** 先读设备已持久化的复位原因成本低、区分能力强，且不会像连接调试器那样改变功耗。

### 第一轮证据

现场诊断返回：

```text
reset_flags=0x00000004  brown_out=1 watchdog=0 software=0
last_state=TLS_WRITE
uptime_before_reset_ms=3864012
```

这提高了电源假设的可信度，但还不能证明“电池不好”。Brown-out 标志可能是上一次复位残留，`last_state` 也只表示当时软件状态，不证明 TLS 代码导致电压下降。

下一步应先确认：启动代码是否在记录后清复位标志；同一设备的故障是否总发生在无线发送；电源轨是否真的越过 brown-out 阈值。

### 第二轮实验

在可复现设备上，用电源轨下降沿触发示波器，同时观察无线模组使能和 MCU 复位：

```text
CH1 3V3: idle 3.28 V, TX peak drops to 2.61 V for 1.8 ms
CH2 RADIO_TX: rises 120 us before voltage drop
CH3 NRST: falls after 3V3 crosses 2.70 V
current: idle 18 mA, peak 412 mA
```

这条时间关系支持“发送峰值触发 brown-out”。但仍有多个机制：电池内阻、DC-DC 瞬态响应、去耦不足、连接器压降，或新固件让 MCU 与无线峰值重叠。

设计对照实验：

1. 用低阻实验电源直接给板供电，保持固件不变。
2. 保持原电池，恢复旧固件。
3. 保持新固件，关闭详细日志但不改变网络 payload。
4. 在新旧固件上对齐采样、Flash 写和无线发送的 GPIO trace。

不要同时更换电池、固件和电容，否则即使故障消失也无法区分原因。

### 第三轮结果

| 条件 | 1000 次发送复位数 | 最低 3V3 |
|---|---:|---:|
| 原电池 + 旧固件 | 0 | 2.91 V |
| 原电池 + 新固件详细日志 | 17 | 2.58 V |
| 原电池 + 新固件关闭详细日志 | 1 | 2.84 V |
| 低阻电源 + 新固件详细日志 | 0 | 3.15 V |

trace 进一步显示，新日志在 TLS 发送前同步写入 SPI Flash，Flash program 与无线 TX 峰值重叠。现在可以构造因果链：

```text
薄弱设计：电源预算没有覆盖器件老化和并发峰值
软件变化：同步 Flash 日志改变负载相位
现场条件：低温和电池内阻上升
故障机制：Flash + RF 峰值导致 3V3 低于 BOR 阈值
症状：MCU 复位，云端观察到离线
```

“增加日志”是触发变化，不是唯一根因；“更换新电池”只能减少触发概率。若只关闭日志，其他未来功能仍可能制造相同峰值。

### 修复与验证

修复组合可能包括：

- 调度层避免 Flash program 与 RF 峰值重叠，并把等待设为有界状态机。
- 日志进入 RAM ring，按电源状态批量落盘；队列满时按等级降级。
- 硬件重新核算最差温度、最低电压、老化电池和负载阶跃，必要时调整 DC-DC、储能和布局。
- 保留 brown-out 计数、负载状态和启动会话 ID，防止问题再次变成“随机离线”。

验证不应只看常温下不再复位：

| 验证 | 条件 | 通过标准 |
|---|---|---|
| 负载阶跃 | 最低工作电压、低温等效内阻、最坏负载组合 | 3V3 保持在设计裕量以上 |
| 相位扫描 | 在 RF 周期不同相位触发 Flash/传感器/LED | 无复位，最大压降有记录 |
| 软件回归 | 10000 次发送，强制日志队列满和 Flash busy | 不阻塞关键路径，降级计数正确 |
| 现场诊断 | 人工触发 brown-out | 复位原因和故障前状态可恢复读取 |

### 本案没有证明什么

- 10000 次通过不等于整个寿命期零故障。
- 实验电源通过不等于所有电池和线束组合通过。
- brown-out 是这批离线的主因，不代表网络状态机无需独立修复。
- 软件错峰降低峰值，不替代硬件电源裕量评审。

### 读者任务与评价标准

1. 如果没有示波器，只能远程诊断，你会在下一版本中增加哪些最小指标？
2. 若产品不能改硬件，提出两个软件方案，并比较数据丢失、Flash 寿命、峰值功耗和实现复杂度。
3. 构造一个与现有证据一致但根因不同的解释，再设计一条证据排除它。

合格答案应暴露假设，区分相关与因果，说明证据的区分能力，并明确修复后的残余风险；只回答“加电容”或“关闭日志”仍停留在症状层。

## 案例库的使用方式

故障案例库不应只是“问题和答案”的集合，而应是训练诊断思维的素材。阅读每个案例时，先遮住根因，自己完成假设排序和证据设计，再对照案例中的实际路径。这样才能训练从现象到机制的推理能力。

每个案例建议额外补充：

- 竞争假设：当时还有哪些合理解释，为什么被排除。
- 最小证据：哪一条日志、波形、寄存器或测试最关键。
- 误导线索：哪些现象看似相关，最后证明只是触发或噪声。
- 修复边界：修复覆盖了哪些条件，哪些硬件/版本/负载仍未验证。
- 预防资产：新增了什么测试、断言、诊断命令、评审清单或设计约束。

好的故障复盘会把一次痛苦经历转化为组织能力。差的复盘只留下“以后注意”，下一次仍会以不同形式重复。
