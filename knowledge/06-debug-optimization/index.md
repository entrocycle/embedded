# 调试、测试与可观测性

调试和验证不是开发结束后的检查，而是从第一个程序开始贯穿所有模块的方法。调试用于区分竞争假设，测试用于把结论变成可重复判定，可观测性用于让同样的证据在实验室之外仍能获得。

本模块既可以顺序阅读，也应在每个项目阶段反复回访。遇到问题时不要先搜索相似现象，先判断当前证据位于电源、时钟、CPU、驱动、调度、协议、系统还是业务哪一层。

## 模块契约

### 进入条件

- 已完成至少一个可运行实验，经历过一次主动失败与回归。
- 理解目标、假设、预测、观测和证据边界。
- 有当前项目可供实际诊断；脱离真实问题只读方法很难形成能力。

### 完成后应能

- 将模糊症状改写为可复现条件和可判定现象。
- 区分症状、触发条件、故障机制、根因和薄弱设计。
- 提出竞争假设，并选择成本与区分度合适的观测。
- 根据层次使用日志、GDB、寄存器、波形、trace、抓包和功耗测量。
- 设计等价类、边界值、状态序列、属性和故障注入测试。
- 解释测试替身、覆盖率、长稳和单次实验的证据边界。
- 将修复转化为回归测试、诊断指标、错误码和设计约束。
- 为现场设备保留版本、复位原因、故障上下文和健康趋势。

### 本模块不解决

- 不把任何单一工具当作万能证据。
- 不用“长稳没出错”证明所有可靠性主张。
- 不以自动复位替代故障隔离、根因保留和恢复验证。

## 通用诊断闭环

```text
冻结现象和环境
  -> 分层定位
  -> 提出竞争假设
  -> 为每个假设写可观察预测
  -> 选择最有区分力的观测
  -> 更新假设置信度
  -> 找到机制与薄弱设计
  -> 修复并运行原触发条件
  -> 加回归、诊断和设计约束
```

“改了某行后问题消失”只证明现象与修改相关，不自动证明根因。日志改变时序、调试器暂停外设、示波器探头改变负载，都可能让观测本身影响系统。

## 证据选择矩阵

| 问题层 | 首选证据 | 常见误判 |
|---|---|---|
| 电源/复位 | 示波器、电源触发、复位寄存器 | 空载电压正常就排除供电 |
| 时钟/引脚 | MCO/GPIO 探针、寄存器、波形 | 配置值正确就认为实际频率正确 |
| CPU/内存 | fault 寄存器、PC/LR/SP、dump、map | 最后一条日志就是故障位置 |
| 总线/外设 | 状态位、错误位、逻辑分析仪 | 驱动 timeout 等于器件没有响应 |
| RTOS/并发 | trace、阻塞链、栈水位、队列指标 | 高优先级必然及时运行 |
| Linux | 启动日志、runtime DT、sysfs、ftrace | 源 DTS 正确等于运行时正确 |
| 网络 | 抓包、状态机、request ID、服务日志 | `send`/publish 成功等于业务完成 |
| 功耗 | 分状态电流和时间、触发波形 | 一个平均电流解释所有状态 |

## 贯穿产物：问题调查包

每次重要问题至少留下：

```text
设备/板卡/软件/配置/工具版本：
可复现步骤和发生概率：
期望与实际：
竞争假设及预测：
原始证据及时间基准：
排除过程：
故障机制与薄弱设计：
临时缓解与永久修复：
原触发条件下的回归：
新增测试、指标或约束：
仍未覆盖的风险：
```

这份包必须允许没有参与原始调试的人复核结论。

## 必做训练

- 对“上电无日志”写出供电、复位、启动、时钟、引脚和串口六层假设树。
- 制造一个 C 边界错误，让 sanitizer 或断言稳定失败并回归。
- 对一条 I2C/SPI 操作同时保存软件日志与总线波形，校准时间关系。
- 在 RTOS 或 Linux 系统中制造资源阻塞，记录等待者、拥有者和持续时间。
- 设计一个 24 小时长稳，不只记录是否崩溃，还记录资源趋势和负载覆盖。
- 把一个已修复问题转化为自动测试或硬件在环步骤。

## 阶段门

- [ ] 能在修改代码前写出至少两个竞争假设和预测。
- [ ] 每个关键结论能指向原始证据，而不是只引用总结日志。
- [ ] 能说明观测工具如何影响被测系统。
- [ ] 修复后使用原触发条件和至少一个相邻边界回归。
- [ ] 长稳测试有负载、指标、采样、阈值和失败保存策略。
- [ ] 现场设备能报告版本、复位原因、关键错误计数和健康水位。
- [ ] 一次问题至少沉淀为测试、诊断、契约或设计规则之一。

## 最重要的迁移能力

调试能力不来自记住故障列表，而来自缩小问题空间的效率。换芯片、换 RTOS 或换 Linux 版本后，命令和寄存器会变化，但以下方法不变：

- 明确可判定目标。
- 区分事实、假设和偏好。
- 用模型产生预测。
- 选择能区分竞争解释的证据。
- 承认证据边界。
- 让修复改变系统资产，而不只改变当前代码。

当你能把陌生故障变成一组逐步收敛的问题，才真正具备可迁移的调试与验证能力。
