# 现场诊断与可观测性

实验室调试依赖 J-Link、示波器和工程师坐在板子旁边；现场问题通常没有这些条件。产品级设备必须内置诊断能力，让工程师能在远程或售后环境中判断设备发生了什么。

## 本章任务与边界

完成本章后，你应能设计版本身份、复位原因、fault 快照、健康指标、事件日志和受控诊断命令，在存储、带宽、功耗与隐私约束下保留最有区分力的现场证据。

可观测性不是无限日志。高频日志可能改变实时性、耗尽 Flash、泄露敏感信息并放大网络成本；远程诊断接口还必须认证、授权、限速和审计。

完成证据：一个版本化诊断 schema、一次崩溃后冷启动恢复、指标容量/保留预算，以及现场分流演练。自动重启恢复服务不等于现场问题已被解释。

## 诊断目标

现场诊断要回答：

- 设备是谁，硬件和软件版本是什么？
- 设备运行了多久，最近重启原因是什么？
- 网络、传感器、执行器、存储、OTA 状态如何？
- 最近发生过哪些错误？
- 问题是个例、批次问题还是版本问题？
- 是否需要重启、回滚、恢复出厂或返修？

## 设备身份信息

最少记录：

| 字段 | 说明 |
|---|---|
| SN | 设备序列号 |
| HW revision | 硬件版本 |
| FW version | 固件版本 |
| build id | 构建 commit 或 hash |
| bootloader version | Bootloader 版本 |
| config version | 配置版本 |
| manufacturing date | 生产日期 |

这些信息要能通过串口、日志、云端和本地命令查询。

## 重启原因

启动时第一时间读取并保存复位原因：

- 上电复位。
- 软件复位。
- 看门狗复位。
- 低功耗唤醒。
- Brown-out。
- 外部复位脚。
- HardFault 后主动复位。

建议维护重启记录：

```text
timestamp
reset_reason
firmware_version
last_error
fault_pc
uptime_before_reset
```

## Fault 记录

HardFault 或异常时，尽量保存：

- PC、LR、SP。
- CFSR、HFSR、BFAR、MMFAR。
- 当前任务名。
- 栈剩余量。
- 最近错误码。
- 最近日志序号。

保存位置可以是备份寄存器、保留 RAM、Flash 小分区或外部存储。写 Flash 前要评估异常上下文是否安全。

## 健康指标

建议每个关键模块暴露 health：

| 模块 | 指标 |
|---|---|
| 系统 | uptime、reset count、CPU、heap、stack |
| RTOS | 任务心跳、队列深度、锁等待 |
| 传感器 | 最近读数、错误计数、断连次数 |
| 网络 | 状态、RSSI、重连次数、最后错误 |
| 存储 | 可用空间、写失败、坏块/擦写次数 |
| OTA | 当前版本、升级状态、失败原因 |
| 电源 | 电池电压、充电状态、低电压次数 |

## 诊断命令

串口 CLI 或本地 shell 建议支持：

```text
version
status
reset_reason
tasks
heap
net
sensor
log tail
ota status
factory_reset
```

命令输出要简洁稳定，方便人工和脚本解析。

## 日志分层

| 类型 | 保存策略 |
|---|---|
| 实时调试日志 | 串口/RTT，开发阶段为主 |
| 运行日志 | 环形缓存，限制大小 |
| 错误事件 | 持久化，带错误码和上下文 |
| 审计事件 | 激活、升级、配置变更 |
| 崩溃转储 | 异常寄存器和少量栈 |

产品日志要避免泄露密钥、证书、用户数据和内部服务地址。

## 远程诊断

远程诊断应支持：

- 拉取设备状态。
- 拉取最近日志。
- 调整日志级别。
- 触发自检。
- 触发安全重启。
- 查询 OTA 和配置状态。

远程命令要有权限控制、审计记录和速率限制。不要留下无认证的调试后门。

## 自检设计

上电自检：

- Flash/配置校验。
- 传感器 ID。
- 网络模组响应。
- 电池/电源状态。
- 关键输出默认安全态。

运行中自检：

- 任务心跳。
- 队列积压。
- 传感器合理范围。
- 存储可写性。
- 网络心跳。

产测自检：

- SN 写入。
- 校准。
- 外设全路径测试。
- 通信测试。
- 指示灯/按键/执行器测试。

## 现场问题分流

| 证据 | 倾向 |
|---|---|
| 同一硬件批次集中出现 | 硬件、器件、工艺或电源问题 |
| 同一固件版本集中出现 | 软件回归 |
| 看门狗复位且任务心跳缺失 | 任务卡死或死锁 |
| Brown-out 增多 | 电源或负载峰值 |
| 网络重连次数激增 | 弱网、证书、云端或运营商问题 |
| OTA 新版本启动后回滚 | 新固件健康检查失败 |

## 练习

1. 为一个 RTOS 项目设计 `status` 命令输出字段。
2. 写一个 HardFault 记录结构体，包含 PC/LR/SP/CFSR 和任务名。
3. 设计一个远程诊断流程：用户报修后你需要拉取哪些信息。

## 工程洞察

现场诊断的价值在于缩短“问题发生”和“获得有效证据”之间的距离。开发阶段可以接调试器、示波器和串口；现场设备往往只能提供有限日志、状态快照、错误码和用户描述。如果设计时没有预留诊断入口，现场问题会退化成猜测。

诊断系统应覆盖三层信息：

- 当前状态：版本、配置、连接、任务、队列、存储、功耗模式、传感器和执行器状态。
- 历史证据：复位原因、最后错误、故障前日志、关键计数器、OTA 历史和配置变更记录。
- 操作入口：安全的自检、日志拉取、配置导出、网络测试、外设测试和受控重启。

优秀诊断命令要稳定、低风险、可脚本化，并且不依赖系统完全健康。例如网络故障时仍能本地串口导出状态，文件系统异常时仍能看到复位原因和只读关键计数。现场诊断不是附加功能，而是产品可维护性的核心组成。
