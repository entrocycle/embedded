# 调试、优化与故障定位

好的嵌入式工程师不是不遇到问题，而是有稳定的定位路径。

## 本章学习方式

调试不是“多试几次”，而是持续缩小不确定性。每次定位都要把猜测变成证据：

```text
现象
  ↓
假设
  ↓
最小验证
  ↓
证据
  ↓
结论或新假设
```

如果一次修改同时改变了多个因素，即使问题消失，也很难证明根因。

## 通用定位框架

先分层：

```text
硬件层：供电 / 时钟 / 复位 / 焊接 / 连接
启动层：Boot / 向量表 / 链接地址 / 栈
驱动层：时钟使能 / 引脚复用 / 寄存器 / 中断 / DMA
系统层：任务 / 栈 / 锁 / 内存 / 文件系统
协议层：时序 / 包格式 / 超时 / 重试
应用层：状态机 / 参数 / 边界条件
```

再缩小：

1. 能否复现。
2. 复现条件是什么。
3. 最近改动是什么。
4. 最小复现代码是什么。
5. 哪个层次的证据最直接。

### 证据优先级

| 证据 | 适合证明 | 局限 |
|---|---|---|
| 复现脚本 | 问题存在且可重复 | 可能覆盖范围窄 |
| 日志 | 状态机、错误码、时间顺序 | 可能被缓冲、丢失或影响时序 |
| 寄存器 dump | 硬件状态、fault 原因 | 需要知道采样时机 |
| 波形 | 电气和总线时序 | 不能直接解释软件状态 |
| trace | 调度和调用路径 | 成本较高，可能改变时序 |
| 构建产物 | 链接地址、符号、大小 | 不能证明运行时行为 |

不要让单一证据承担过宽结论。例如“日志没报错”不能证明总线没问题。

## 串口日志

日志要服务定位，而不是越多越好。

建议包含：

- 时间戳。
- 模块名。
- 级别。
- 错误码。
- 关键状态。

示例：

```text
[0012.340][net][INFO] mqtt connected
[0015.120][ota][ERROR] chunk crc failed, offset=4096, err=-5
```

裸机早期日志可以使用轮询 UART；系统运行期推荐异步日志队列。

日志设计要避免两个问题：

- 关键路径打印太多，改变实时性。
- 错误日志只有一句“failed”，没有上下文。

一条可用于定位的错误日志至少应包含模块、操作、错误码、关键参数和状态。例如 OTA 校验失败时，`offset`、`expected_crc`、`actual_crc`、镜像版本都比单纯 `ota failed` 更有价值。

## GDB 常用命令

```gdb
target remote :3333
monitor reset halt
load
break main
continue
next
step
finish
info registers
x/16wx 0x20000000
bt
p variable
```

定位 HardFault 时重点看：

- PC：出错指令地址。
- LR：返回地址。
- SP：栈位置。
- CFSR/BFAR/MMFAR：Fault 原因和地址。

HardFault 复盘时建议固定输出：

```text
pc:
lr:
sp:
xpsr:
cfsr:
hfsr:
bfar:
mmfar:
active_irq:
firmware_version:
reset_reason:
```

只保存 PC 不够，因为很多 fault 的根因在栈、总线访问或中断上下文。

## 逻辑分析仪

适合验证数字协议。

UART：

- 波特率是否正确。
- 是否有帧错误。
- TX/RX 是否接反。

I2C：

- 是否有 START/STOP。
- 地址是否 ACK。
- 数据阶段是否 NACK。
- SCL/SDA 是否被拉死。

SPI：

- CPOL/CPHA。
- CS 时序。
- SCK 频率。
- MOSI/MISO 数据。

CAN：

- 波特率。
- ACK。
- 错误帧。
- 终端电阻。

## 示波器

适合验证模拟和高速信号。

常见用途：

- 电源上电波形和跌落。
- 晶振是否起振。
- PWM 频率和占空比。
- ADC 输入噪声。
- 复位脚异常抖动。

电源问题常表现为：

- 随机复位。
- 无法下载。
- 外设偶发异常。
- 无线发射时掉线。

## RTOS 调试

重点指标：

- 每个任务栈剩余量。
- CPU 使用率。
- 中断频率。
- 队列积压。
- 锁等待时间。
- 高优先级任务是否被低优先级路径阻塞。

常见问题：

| 问题 | 现象 | 定位 |
|---|---|---|
| 栈溢出 | 随机 HardFault | stack overflow hook、哨兵值 |
| 死锁 | 任务不再推进 | 查看任务状态和锁持有者 |
| 优先级反转 | 高优先级任务延迟异常 | trace 任务切换 |
| 队列满 | 数据丢失或阻塞 | 记录队列 high watermark |
| ISR 过长 | 实时性变差 | 统计 ISR 执行时间 |

## 性能优化

优化顺序：

1. 明确指标：延迟、吞吐、CPU、RAM、Flash、功耗。
2. 测量基线。
3. 找热点。
4. 一次只改一个因素。
5. 复测并记录。

常见优化：

- 用 DMA 降低 CPU 搬运。
- 减少中断频率，批量处理。
- 优化数据结构，避免拷贝。
- 使用定点替代浮点。
- 开启编译优化。
- 裁剪未用模块。
- 把常量表放 Flash。

### 优化前先定义指标

“变快一点”不是指标。更可执行的指标包括：

- 控制任务周期 10ms，99.9% 抖动小于 200us。
- MQTT 上报峰值 CPU 占用低于 40%。
- OTA 下载时日志不丢失，队列最大占用低于 80%。
- 固件 Flash 占用低于 384KB，RAM 静态占用低于 80KB。

没有指标的优化容易变成局部炫技，甚至牺牲可维护性。

## 功耗优化

功耗由硬件、外设、软件调度共同决定。

策略：

- 不用的 GPIO 设置为低漏电状态。
- 不用的外设关闭时钟。
- 降低主频或动态调频。
- 批量采集和发送，减少唤醒次数。
- 使用 Sleep/Stop/Standby。
- 无线通信减少连接时间和发射次数。

功耗测量要分状态：

| 状态 | 关注 |
|---|---|
| 上电启动 | 峰值电流、启动时间 |
| 空闲 | sleep 是否进入 |
| 采样 | 传感器和 ADC 电流 |
| 通信 | Wi-Fi/BLE/蜂窝峰值 |
| OTA | 下载和写 Flash 功耗 |

功耗问题必须分状态测量。如果只看平均电流，很难知道问题来自 sleep 未进入、传感器未关断、GPIO 漏电还是通信峰值过长。建议画出状态机，并在每个状态标注理论电流、实测电流和停留时间。

## 故障复盘模板

每个复杂问题都建议记录：

```text
问题标题：
影响范围：
复现步骤：
期望结果：
实际结果：
硬件版本：
软件版本：
日志/波形/寄存器：
根因：
修复：
验证：
预防措施：
```

### 根因与触发条件要分开

复盘时要区分：

- 根因：代码、硬件或设计中真正错误的地方。
- 触发条件：让问题暴露出来的场景。
- 放大因素：让问题更频繁或更严重的因素。
- 逃逸原因：为什么测试阶段没有发现。

例如“弱网下 OTA 失败”可能只是触发条件，真正根因可能是状态机没有处理重复 chunk、Flash 写入错误没有回滚，或下载任务阻塞导致看门狗复位。

## 调试反思问题

1. 我现在的结论由什么证据支持？有没有证据只支持了现象而没有支持根因？
2. 如果这个问题明天在现场复现，我还缺少哪些日志或 dump？
3. 当前修复是消除根因，还是只绕过触发条件？
4. 如何把这次问题转化为自动测试、故障注入或发布检查项？

## 故障注入建议

| 方向 | 注入方式 | 观察 |
|---|---|---|
| 启动 | 改错向量表或栈大小 | fault 记录是否完整 |
| 外设 | 拔传感器、短暂断电 | 驱动错误码和恢复 |
| RTOS | 队列长度减小、任务卡死 | 看门狗和任务状态 |
| 网络 | broker 不可达、频繁断网 | 重连退避和缓存 |
| 存储 | 写入中断电、CRC 错 | 回滚和默认配置 |
| 功耗 | 禁止 sleep、保持 GPIO 浮空 | 电流状态分解 |

## 练习

1. 故意把 UART 波特率配错，用逻辑分析仪判断实际波特率。
2. 写一个 HardFault handler，保存 PC/LR/SP/CFSR。
3. 对一个周期任务测量执行时间和抖动。
4. 选择一个真实或模拟故障，按“现象-假设-证据-结论-预防”写完整复盘。
