# 嵌入式测试与验证方法

调试回答“这次为什么坏了”，测试回答“哪些行为已经被系统性检查”，验证回答“证据是否足以支持需求已经满足”。跑通 demo 只证明某一组条件下主路径出现过，无法证明边界、异常、时序和长期演化。

## 本章任务与边界

完成本章后，你应能从需求和风险设计测试预言，选择主机、仿真、目标板、HIL、长稳与现场证据，使用边界值、状态序列、属性和故障注入，并解释替身、覆盖率和通过结果的外推边界。

测试不能穷尽硬件、时序、环境和攻击状态。高覆盖率不证明需求正确，长稳通过不证明未覆盖负载可靠，mock 通过不证明真实驱动行为；验证结论必须带版本、条件与未覆盖风险。

完成证据：一个需求到风险/测试/证据追踪表、仓库三项主机实验结果、一项目标板故障注入，以及失败后保留原始证据的自动化流程。

## 从需求建立证据链

每条重要需求都应能追到具体证据：

```text
需求
  -> 风险与失效模式
  -> 可判定验收条件
  -> 测试或分析方法
  -> 原始证据
  -> 结论与残余风险
```

示例需求“断电不能损坏配置”仍然含糊。更可测的版本是：

> 在配置提交过程的每个可注入断电点复位后，系统必须加载新配置或最后一个已确认配置；不得加载 CRC 错误或字段越界的配置；10000 次随机断电注入中不得进入不可恢复状态。

这里仍要声明 Flash 型号、磨损状态、电压、断电斜率和“不可恢复”的定义。验证深度应与风险相匹配，不能把有限测试次数解释为数学证明。

## 测试层次

| 层次 | 运行位置 | 擅长发现 | 不足 |
|---|---|---|---|
| 静态检查 | 构建机 | 类型、未定义行为线索、规则违规、依赖问题 | 不观察真实时序和硬件 |
| 主机单元测试 | PC | 解析、状态机、算法、边界值 | ABI、并发和硬件行为可能不同 |
| 仿真/模拟 | 模拟器 | CPU 异常、部分外设交互、可重复故障 | 模型不一定覆盖电气和真实外设 |
| 目标板组件测试 | 实板 | 驱动、时钟、中断、DMA、资源占用 | 搭建和定位成本较高 |
| HIL 系统测试 | 实板 + 仪器/夹具 | 端到端行为、异常输入、时序、电源 | 自动化成本高，oracle 设计困难 |
| 长稳与现场回放 | 产品环境 | 泄漏、磨损、尾部延迟、环境耦合 | 反馈慢，覆盖难量化 |

层次不是替代关系。状态机可在 PC 上穷举大量事件序列，再用少量目标板测试验证时间、驱动和集成假设。

## 先设计可测试边界

把纯策略与硬件副作用分开：

```c
typedef enum {
    LINK_DOWN,
    LINK_CONNECTING,
    LINK_ONLINE,
    LINK_BACKOFF,
} link_state_t;

typedef enum {
    EVT_START,
    EVT_CONNECTED,
    EVT_FAILED,
    EVT_TIMEOUT,
} link_event_t;

typedef struct {
    link_state_t next;
    bool start_connect;
    bool arm_timer;
    uint32_t delay_ms;
} link_action_t;

link_action_t link_step(link_state_t state, link_event_t event,
                        uint32_t failure_count, uint32_t random_value);
```

`link_step` 不直接访问 socket、tick 或随机数，因而可在主机上测试状态转移。平台层负责执行 action。这里的抽象不是为了“代码漂亮”，而是让决策逻辑具备可控输入和可观察输出。

## 测试设计：不要只列样例

### 等价类与边界值

对长度为 `0..MAX_PAYLOAD` 的帧，至少覆盖：

- 0、1、`MAX_PAYLOAD - 1`、`MAX_PAYLOAD`、`MAX_PAYLOAD + 1`。
- 声明长度小于、等于和大于实际输入。
- `offset + length` 发生整数溢出。
- CRC 正确、错误、字段截断和未知版本。

边界来自实现和协议约束，而不是随便选择几个数字。

### 状态转换与序列

状态机 bug 往往不在单次事件，而在序列：

```text
connected -> timeout -> late_connected -> stop -> timer_fired
```

为每个状态定义允许事件、忽略事件和非法事件。测试重复事件、乱序事件、迟到事件，以及复位后从持久化状态恢复。

### 属性与不变量

样例断言某个输入得到某个输出；属性描述一类输入始终满足的条件：

- 解析失败绝不读取输入范围之外的字节。
- 任意事件序列下，退避时间不超过上限且不会整数溢出。
- 配置提交前后，至少一个副本可被校验并加载。
- 队列计数始终在 `0..capacity` 内。

对纯逻辑可使用随机序列、property-based testing 或 fuzzing；对硬件状态机可由脚本生成受约束事件。

## 测试替身的边界

| 替身 | 用途 | 主要风险 |
|---|---|---|
| Fake | 内存中实现简化存储或总线 | 行为比真实设备理想 |
| Stub | 返回预设值或错误 | 容易只覆盖预想路径 |
| Mock | 校验调用顺序和参数 | 与实现细节耦合，重构易碎 |
| Simulator | 模拟寄存器或协议设备 | 模型错误会产生虚假信心 |

不要 mock 掉你真正要验证的机制。测试重试状态机可以 stub 网络结果；验证 I2C 上升时间必须使用实物和测量仪器。

## 故障注入矩阵

按故障域设计，而不是只拔一次网线：

| 故障域 | 注入例子 | 关键观测 | 通过条件 |
|---|---|---|---|
| 电源 | 启动、Flash 写、OTA 切换时断电 | 复位原因、metadata、可启动镜像 | 不违反恢复不变量 |
| 时间 | 超时前后、tick 回绕、RTC 跳变 | 状态、deadline、重试计数 | 无永久卡死或重试风暴 |
| 内存 | 分配失败、栈接近上限、buffer 截断 | 错误码、高水位、越界检测 | 有界降级，无静默破坏 |
| 外设 | NACK、总线忙、DMA 丢中断 | 寄存器、波形、恢复次数 | 超时并恢复或进入明确故障态 |
| 并发 | 迟到事件、队列满、锁竞争 | trace、最大延迟、丢弃计数 | 不变量和截止时间成立 |
| 网络 | DNS、TLS、半开连接、重复命令 | 状态机、退避、幂等记录 | 有界资源，无重复副作用 |
| 存储 | 写满、坏块、CRC 错、旧版本 | 选择记录、磨损计数 | 加载有效版本或安全默认值 |

每个故障都要定义注入点、触发概率或次数、预期行为、证据和恢复后检查。只看到“设备重启了”不能证明恢复正确。

## 时间和实时性验证

平均值会隐藏最危险的尾部。至少记录：

- 最小、最大、分位数和样本数。
- 测量区间与时钟来源。
- 同时运行的负载和中断频率。
- 首次运行、稳态、错误路径的区别。
- 测量探针对系统的扰动。

GPIO 翻转配合逻辑分析仪适合测 ISR 延迟和执行时间；RTOS trace 适合分析阻塞链；CPU cycle counter 适合短路径。串口日志通常不适合证明微秒级截止时间。

对周期任务，端到端延迟可以拆为：

```text
response = interrupt_latency
         + queueing_delay
         + blocking_time
         + execution_time
         + downstream_delay
```

优化前先找主导项。只缩短函数执行时间，无法修复由锁导致的长尾。

## 长稳测试不是“放着跑”

长稳测试需要加速应力、可观测指标和停止条件：

- 提高连接切换、配置提交和 OTA 次数，覆盖状态转换。
- 记录 heap/栈最低余量、队列高水位、错误计数、复位原因和温度。
- 对计数器变化率设阈值，避免测试结束后才发现日志早已刷屏。
- 保存固件、硬件、配置、随机种子和外部服务版本。
- 测试结束后检查持久化数据一致性，而不只看设备还在线。

168 h 无故障只证明这组条件下未观察到故障。若需求涉及年级寿命，需要结合加速试验、器件数据、磨损模型和现场统计，不能简单外推。

## 覆盖率应该怎样解释

代码覆盖率可以发现从未执行的代码，但不能单独证明测试有效：

- 语句覆盖不代表分支和条件组合覆盖。
- 分支覆盖不代表并发时序和硬件故障覆盖。
- 故障注入数量不代表风险覆盖。
- 需求通过数量不代表需求本身完整。

更有价值的是联合看：需求追踪、关键不变量、代码覆盖、故障模式覆盖、目标板证据和未覆盖风险。对高风险模块，应说明为什么选择这些测试，而不是只报告百分比。

## 可重复的测试记录

```text
测试 ID 与需求 ID：
硬件版本 / SN：
固件 commit / 构建 hash：
工具链与配置：
仪器型号、探头与校准状态：
前置状态：
输入、随机种子与时间基准：
步骤与自动化命令：
预期和容差：
原始证据位置：
实际结果：
偏差、结论与残余风险：
```

截图应保留时间刻度、通道量纲和触发条件；日志应能关联设备、启动会话和单调时间。只有总结、没有原始证据的报告很难复核。

## CI 与发布门禁

建议逐层建立门禁：

1. 格式、编译告警、静态检查和依赖许可。
2. 主机单测、解析器 fuzz seed 和状态机属性测试。
3. 交叉编译、固件大小预算、链接段和禁止符号检查。
4. 每日目标板冒烟测试和接口 loopback。
5. 每周 HIL 故障注入、功耗和长稳测试。
6. 发布候选版本执行固定硬件矩阵、OTA 升降级和产测回归。

不稳定测试不能长期靠重跑掩盖。应记录失败率、隔离根因，并区分产品不确定性、测试夹具问题和测试本身的竞态。

## 综合实验：可恢复配置存储

设计一个双副本配置模块，要求带 magic、schema version、sequence、length 和 CRC。

仓库提供了可直接运行的 [主机实验：双槽配置日志与断电一致性](../00-learning-path/host-config-journal.md)。建议先不看实现，根据下面四个阶段写出自己的不变量和断电点预测，再运行测试比较。

### 第一阶段：模型

1. 写出提交步骤和启动选择规则。
2. 定义不变量：任意提交中断后，新旧副本至少一个有效。
3. 列出 sequence 回绕、两个副本都有效、schema 不兼容等歧义。

### 第二阶段：主机验证

把 Flash 抽象成可注入失败的 byte array，在每次 erase/program 操作后模拟断电并重新初始化。枚举小规模操作点，再用随机种子扩展序列。

### 第三阶段：目标板验证

使用可控电源或夹具在写入窗口断电，检查真实 Flash 粒度、电压下降和 brown-out 行为是否符合模型。记录每次恢复选择和 CRC。

### 第四阶段：反思

- 主机 fake 中哪些假设没有被实板支持？
- CRC 能检测随机损坏，但能防恶意回滚吗？
- 擦写寿命用什么工作负载模型估算？
- 若两个副本都损坏，安全默认值是什么，设备是否仍可远程恢复？

## 深入思考

1. 一个测试从未失败，可能说明产品可靠，也可能说明测试没有观察力。如何区分？
2. 为了让代码易测而增加抽象，会增加 Flash、RAM 和调用开销。什么证据能支持这笔成本？
3. 随机故障注入与枚举故障点各有什么盲区？如何保存随机种子同时避免只回归历史种子？
4. 需求写“99.99% 可用”时，单台设备测试多久才有统计意义？实验室测试还缺少哪些现场分布？
5. 测试通过后修改了编译器、链接脚本或硬件 BOM，哪些证据仍有效，哪些必须重做？

## 练习

1. 为一个帧解析器列出等价类、边界值、属性和至少三个 fuzz seed。
2. 给 RTOS 队列满设计可重复测试，明确生产速率、消费暂停、通过条件和观测指标。
3. 把项目的一条模糊需求改成可判定验收标准，并建立需求到证据的追踪表。
4. 设计一次“错误恢复也失败”的二阶故障，例如日志分区写满后又发生网络断开，说明预期降级路径。

## 工程洞察

测试与验证的目标不是证明“代码没有 bug”，而是在明确条件下建立足够可信的证据。嵌入式系统尤其需要分清主机测试、仿真测试、目标板测试、硬件在环、长稳和现场数据各自能证明什么。

一个高质量验证计划应包含：

- 需求追踪：每条需求对应至少一种可判定证据。
- 分层测试：纯逻辑主机测试、驱动替身、目标板外设验证、系统压力和现场回归。
- 故障注入：超时、NACK、队列满、断电、弱网、内存不足、校验失败和配置损坏。
- 观测指标：栈高水位、队列水位、错误计数、复位原因、延迟分布、功耗和存储磨损。
- 停止条件：运行多久、负载多大、样本多少、允许哪些已知风险。

测试通过后仍要写证据边界：它覆盖了哪些硬件版本、固件版本、环境温度、网络条件和数据规模；哪些条件没有覆盖，不能被外推。成熟验证不是追求“全都测过”，而是让未覆盖风险显性化并可决策。
