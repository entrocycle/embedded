# 主机实验：双槽配置日志与断电一致性

这个实验不依赖开发板。它用一个 64 字节的内存数组模拟 Flash，在 macOS 或 Linux
主机上验证一个问题：**配置更新过程中任意时刻断电，重启后能否只读到旧配置或完整的
新配置，而绝不会把半写记录当成有效配置？**

重点不是记住“双槽”这个名词，而是把设计主张变成可枚举、可失败、可重复的证据。

## 学习目标

完成实验后，你应该能：

1. 解释 Flash 的 `1 -> 0` 编程约束，以及为什么覆盖写入前必须擦除。
2. 从字节布局说明 CRC、序列号、提交标记分别证明什么，不能证明什么。
3. 推导“先写数据，最后提交；先保留旧槽，再擦目标槽”的断电安全性。
4. 枚举每一个 Flash 原子步骤后的断电点，用测试验证恢复不变量。
5. 区分“写入函数没有返回成功”和“新记录没有持久化”这两个不同命题。

建议用时：60-90 分钟。

## 前置知识

- 能读懂 C 结构体、数组、指针和整数类型。
- 知道 CRC 用于检测意外损坏，不用于抵抗恶意篡改。
- 会在终端运行 `make` 和阅读测试失败位置。

只需要一个支持 C11 的编译器和 Make。macOS 可使用 Apple Clang，Linux 可使用 GCC 或
Clang。检查命令：

```bash
cc --version
make --version
```

## 运行

在本目录执行：

```bash
make test
```

默认编译命令包含：

```text
cc -std=c11 -Wall -Wextra -Werror -pedantic -O2 -g ...
```

可选地运行 AddressSanitizer 和 UndefinedBehaviorSanitizer：

```bash
make sanitize
```

若编译器或宿主环境只支持其中一种，可以单独选择：

```bash
make sanitize SANITIZERS=undefined
make sanitize SANITIZERS=address
```

如果 sanitizer 程序在测试输出前就退出，先用一个空 `main` 检查编译器运行时；这与测试
断言失败不同。容器限制、旧版运行库或 macOS 的 shadow memory 映射问题都可能让 ASan
无法启动。不要因为 ASan 环境不可用而跳过普通严格构建和 UBSan，也不要把“运行时没启动”
报告成“代码通过 ASan”。

也可以切换编译器：

```bash
make clean
make test CC=clang
make clean
make test CC=gcc
```

原子步骤数由当前记录格式决定。未经修改时，关键输出应为：

```text

[PASS] round trip and slot rotation
[PASS] invalid input preserves flash
[PASS] CRC falls back from corrupted newest slot

[PASS] power cut after every first-store primitive
```

## Flash 模型

模拟器刻意保留三个硬件约束：

1. 擦除以整个槽为单位，擦除后每一位都是 `1`，即字节值 `0xFF`。
2. 编程只能把位从 `1` 清成 `0`。若需要 `0 -> 1`，必须先擦除。
3. 一次槽擦除或一次字节编程是一个可观测原子步骤；步骤完成后可能立刻断电。

真实芯片的擦除粒度可能是一页或一个扇区，最小编程粒度也可能是 4、8 或 16 字节。
本实验使用“单字节编程”是为了让状态空间小而完整，不代表所有 MCU 都按字节写 Flash。
迁移到具体芯片时，应把这里的原子步骤替换成数据手册保证的最小编程单元。

## 记录格式

每个槽占 32 字节，当前使用前 28 字节：

| 偏移 | 大小 | 字段 | 作用 |
|---:|---:|---|---|
| 0 | 4 | magic | 区分本日志与随机内容 |
| 4 | 4 | sequence | 在两个有效槽中选择较新的记录 |
| 8 | 12 | config | 上报周期、告警阈值、标志位 |
| 20 | 4 | CRC32 | 检查 magic、sequence 和 config 是否完整 |
| 24 | 4 | commit | 最后写入；只有完整匹配才表示事务提交 |
| 28 | 4 | reserved | 保持 `0xFF`，用于将来扩展 |

字段按小端字节序显式编码。代码没有把 C 结构体直接写入 Flash，因为结构体可能包含
padding，宿主和目标还可能有不同的字节序、对齐与 ABI。

### 一次更新的 29 个步骤

1. 擦除非当前槽：1 步。
2. 写入 magic、sequence、config 和 CRC：24 个字节编程步骤。
3. 最后写入 commit：4 个字节编程步骤。

合计 `1 + 24 + 4 = 29`。测试先运行一次无故障更新得到步骤数，再从 `cut=1` 到
`cut=29` 逐一重建相同初始 Flash，并在该步骤完成后模拟断电。它不是抽样测试。

## 写入与恢复状态机

```text
找到最新有效槽
       |
       v
选择另一个槽 -> 擦除目标槽 -> 写数据与 CRC -> 最后写 commit
       ^                                              |
       |                                              v
旧槽始终保留                                  新槽成为最新有效槽
```

启动时不信任“上次写到了哪一步”，只扫描两个槽：

```text
magic 正确?
  -> commit 完整?
    -> CRC 正确?
      -> 配置语义有效?
        -> 比较 sequence，选择较新者
```

注意最后一个编程步骤后的断电：新记录已经完整提交，但调用者没有收到 `OK`。重启后读到
新值是正确行为。这说明 API 返回失败只代表“调用没有完成握手”，不代表事务一定没发生。
上层若要重试，需要让操作幂等，或重新读取当前配置后再决定。

## 必须成立的不变量

测试围绕以下不变量，而不是围绕某几行实现编写：

1. **旧值保留**：新槽提交前，旧槽不得被擦除或修改。
2. **提交在最后**：commit 有效时，记录数据和 CRC 必须已经写完。
3. **半写不可见**：启动恢复只能返回旧配置、新配置或“尚无配置”。
4. **损坏可检测**：已提交记录任一受保护字节变化后，CRC 必须使其失效。
5. **单调选择**：两个槽都有效时选择逻辑上的新序列号，包括 `UINT32_MAX -> 0` 回绕。
6. **非法输入无副作用**：参数校验必须发生在擦除之前。

这里没有承诺“任意两次连续掉电后永远有配置”。例如首次写入尚未提交就断电，合理结果
是 `NOT_FOUND`。产品可以在此之上提供 ROM 默认配置，但不能把默认值伪装成已持久化记录。

## 测试如何形成证据

`test_every_power_cut_during_first_store` 证明首次落盘时，所有中间状态要么不可见，要么在
最后提交后成为完整新值。

`test_every_power_cut_during_update` 分别从最新记录位于槽 0 和槽 1 的快照开始，逐个注入
断电，因此两个轮换方向都被覆盖。每次都重新调用 `journal_load` 模拟冷启动，而不是继续
使用写入函数的 RAM 状态。判定集合只有：

```text
{ OLD_CONFIG, NEW_CONFIG }
```

任何第三个配置、CRC 通过的半写配置或 `NOT_FOUND` 都立即失败。这条属性比“更新一次后
能读回来”强得多。

CRC 损坏测试进一步主动清除最新槽中的一个数据位，证明恢复会忽略损坏的新槽并退回旧
槽。CRC 只能说明“内容与写入时一致”，不能证明来源可信；安全配置还需要签名/MAC、密钥
保护和防回滚机制。

## 建议练习

按顺序完成，每步先预测测试结果，再改代码。

### 练习 1：制造错误的提交顺序

把 `journal_store` 改为先写 commit，再写数据，运行 `make test`。记录第一个失败的断电点，
并解释为什么 magic/commit 检查可能通过、CRC 是否仍能阻止所有错误。

### 练习 2：删除 CRC 检查

临时跳过 `journal_inspect_slot` 中的 CRC 比较。哪个测试先失败？如果掉电后的未写字节仍为
`0xFF`，语义校验能否偶然挡住问题？这能否替代 CRC？

### 练习 3：破坏旧槽保留原则

把目标槽错误地改成当前最新槽。列出擦除完成后断电的恢复结果，并说明为什么“写得更快”
不等于“事务安全”。

### 练习 4：改变编程粒度

把一次编程原子步骤从 1 字节改成 4 字节，保证地址和长度对齐。更新预期断电点数量，检查
同一组不变量是否仍成立。不要只修改计数器，要修改真正的 Flash primitive。

### 练习 5：增加 schema version

在记录中加入格式版本。设计从 v1 到 v2 的迁移规则，并回答：旧固件遇到 v2 应该回退、
忽略还是拒绝启动？如果 OTA 回滚到旧固件，谁负责配置向后兼容？

## 分层提示

遇到失败时按层查看，不要直接读参考结论：

1. **提示一**：打印 `cut`、两个槽的 magic/sequence/CRC/commit，先找哪条有效性条件变化。
2. **提示二**：区分“物理内容写完”“commit 可见”“调用者收到成功”三个时刻。
3. **提示三**：证明断电安全时，以一个断电点为界，分别列出该点之前已经持久化的字节和
   仍为 `0xFF` 的字节。
4. **提示四**：如果恢复出第三个值，检查 commit 是否真的最后写，以及 CRC 覆盖范围是否
   包含所有参与恢复决策的字段。
5. **提示五**：序列号比较不能直接使用 `candidate > reference`，因为无符号计数会回绕。

## 评价量规

满分 20 分。能通过测试只是起点，不自动等于满分。

| 维度 | 0 分 | 1 分 | 2 分 | 3 分 | 4 分 |
|---|---|---|---|---|---|
| 机制解释 | 只能复述代码 | 知道双槽 | 能解释 CRC/commit | 能推导写入顺序 | 能联系真实 Flash 粒度与数据手册 |
| 不变量 | 没有明确主张 | 只说“不丢数据” | 写出旧/新二选一 | 覆盖首次写、更新、损坏 | 能指出模型未保证的边界 |
| 故障实验 | 只跑正常路径 | 手工测一个断电点 | 测若干点 | 自动枚举全部 primitive | 修改模型后仍能自动发现全部点 |
| 证据表达 | 只说“测试过了” | 给出通过截图 | 给出命令和输出 | 解释每项测试排除什么假设 | 能说明测试覆盖不到什么 |
| 迁移思考 | 直接照搬 | 知道芯片不同 | 查到擦写粒度 | 调整布局/对齐/驱动 | 同时考虑磨损、并发、OTA 回滚与安全 |

建议通过标准：至少 14 分，且“不变量”和“故障实验”均不低于 3 分。

## 复盘与延伸思考

1. CRC、commit 和 sequence 各自回答的是“完整性、原子性、新旧关系”中的哪一个问题？
2. 如果 commit 所在的最小编程单元同时包含 CRC，现有顺序还能成立吗？应怎样重新布局？
3. 两槽方案在持续更新时如何磨损？页擦除寿命为 10 万次、每天更新 100 次时，粗略寿命
   是多少？该估算忽略了哪些现场因素？
4. 中断和另一个任务同时调用 `journal_store` 会发生什么？互斥应该放在日志层还是 Flash
   驱动层？中断上下文是否应该允许写 Flash？
5. 掉电不是瞬时数字事件。若电压缓慢跌落、CPU 仍运行但 Flash 已低于可靠编程电压，当前
   模型缺了什么？Brown-out 阈值、储能电容和写入时间如何进入证明？
6. 两个有效槽的序列号恰好相差 `2^31` 时，模比较没有唯一答案。为什么正常双槽轮换不会
   到达该状态？位翻转或人为复制是否可能制造它？产品应如何处理歧义？
7. 若配置写成功但应答丢失，上层重试会多消耗一次擦写寿命。如何设计 request ID、内容
   比较或幂等更新来避免无意义写入？
8. 若攻击者能修改整个记录并重新计算 CRC，这个方案还能保证什么？加入 MAC 或签名后，
   密钥放在哪里，防回滚计数由谁保护？

最终输出不应只是一份通过的测试。请保留你的初始假设、一次被测试推翻的判断、故障点证据
以及迁移到具体芯片时需要查阅的数据手册条目。
