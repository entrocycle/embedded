# 主机实验：环形缓冲的边界与不变量

本实验实现一个保留槽环形缓冲，并用参考队列模型检查 100000 次确定性随机操作。目标不是背下 `head` 和 `tail` 的写法，而是学会先定义状态表示与不变量，再验证边界、wrap 和失败语义。

## 设计主张

当 `head` 指向下一写入位置、`tail` 指向下一读取位置，并永久保留一个空槽时：

- `head == tail` 唯一表示空。
- `next(head) == tail` 唯一表示满。
- 可用容量为底层数组长度减一。
- push/pop 成功序列必须与普通 FIFO 队列模型一致。
- 对满缓冲 push 或空缓冲 pop 不得修改已有数据语义。

## 运行

```bash
make test
make sanitize
```

普通测试使用 C11、严格警告和 `-Werror`。`sanitize` 额外运行 AddressSanitizer 与 UndefinedBehaviorSanitizer；若宿主环境不支持 sanitizer，普通测试仍必须通过，并记录环境限制。

## 先预测

运行前回答：

1. 底层数组为 4 字节时为什么只能保存 3 字节？
2. `head=1, tail=3, storage_size=5` 时当前长度是多少？
3. 如果允许覆盖最旧数据，`push` 的错误语义和项目数据语义会怎样变化？
4. 该实现能否直接用于 ISR 生产、任务消费？哪些架构和内存可见性假设尚未验证？

## 测试证据

测试分为四层：参数契约、空满语义、索引 wrap、与独立队列模型对照。模型测试固定随机种子，因此失败可复现；它能发现顺序、计数和边界错误，但不能证明任意并发执行正确。

## 故意破坏

依次尝试：

1. 删除 `ring_buffer_push` 的满检查，观察哪个测试先失败。
2. 把 `ring_buffer_size` 的 wrap 分支改成简单 `head - tail`，解释无符号回绕为什么不等同于正确容量。
3. 让满时覆盖最旧数据，同时更新 API 名称、测试和数据语义；不要只让旧测试“重新通过”。
4. 给结构增加 `overflow_count`，思考它由哪个上下文更新、是否会回绕、现场如何读取一致快照。

## 迁移到目标板

这个主机实验只验证顺序逻辑。迁移到 UART ISR + 任务时，还要证明：

- `head` 和 `tail` 的访问宽度在目标 CPU 上是否原子。
- 单生产者/单消费者约束是否始终成立。
- 编译器与 CPU 的可见性和顺序要求如何满足。
- ISR 生产速率长期高于任务消费速率时采用背压、丢弃还是覆盖。
- overrun、硬件错误和软件队列满是否使用不同计数。

主机通过是 E1 证据，目标板中断压力、波形和计数才会把结论推进到 E2/E3。
