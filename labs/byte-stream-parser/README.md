# 主机实验：有界字节流解析器

本实验解析一种简单帧格式：2 字节大端 payload 长度，后跟 1-64 字节 payload。它模拟 TCP、UART DMA 或文件读取的共同事实：输入是字节流，读取边界不等于消息边界。

## 设计主张

- 同一合法字节序列无论怎样分片，都应产生相同消息。
- 多帧合并在一次输入中，仍应逐帧交付。
- 零长度和超长帧必须在分配或越界前被拒绝。
- 输入结束时的半个头或半个 payload 必须被报告为截断。
- 解析器遇到协议错误后进入失败状态，调用方必须重建会话或显式重置，不能悄悄把剩余字节当成新帧。

## 运行

```bash
make test
make sanitize
```

预期六组测试通过，其中 split 测试会枚举一帧的所有双块切分位置。

## 先预测

1. 一次 `recv()` 返回半个长度字段时，解析器应保存什么状态？
2. 两帧被一次 `recv()` 返回，为什么不能只调用一次业务回调？
3. 长度为 65535 而最大 payload 为 64 时，检查应该发生在分配之前还是之后？
4. 若长度头损坏，本协议为什么无法可靠地在同一 TCP 连接中重新同步？

## 证据边界

测试证明纯解析逻辑对已覆盖分片、合并、超长和截断场景符合契约。它没有证明：

- socket 超时、短写、`EINTR`、EOF 和重连策略正确。
- payload 业务语义、安全授权和幂等正确。
- 对恶意流量具备足够 CPU、内存和连接级限流。
- 长度头损坏后可以流内重同步。

因此真实网络协议还需要 deadline、连接状态机、身份认证、request ID、资源配额和错误指标。

## 故意破坏

1. 假设每次输入都至少包含完整头，观察 one-byte 测试如何失败。
2. 回调一帧后直接返回，观察 coalesced 测试丢失第二帧。
3. 删除最大长度检查，并说明即使固定数组暂时阻止越界，这种接口为何仍然危险。
4. 将协议改成“magic + version + length + payload + CRC”，设计头部损坏后的重新同步规则和扫描上限。

## 迁移问题

把解析器放进设备协议时，继续回答：

- payload 是状态、事件、命令还是文件分片？
- 回调返回失败后，连接关闭、丢帧还是重试分别会产生什么语义？
- 命令已执行但 ACK 丢失时，request ID 和持久化去重如何处理？
- UART 丢字节后如何重同步，TCP 为什么通常不需要处理传输层丢字节却仍需处理应用帧损坏？
- 加密和认证放在哪一层，长度字段是否属于受保护内容？

一个正确的 parser 只解决消息边界问题，不解决端到端业务正确性。
